# F2.17 — Admin Backend Foundations: Contract Lock

Status: contract / scope freeze. Locks the exact surface of F2.17 before any
implementation begins. No code, models, migrations, or frontend changes are
introduced by this document.

This document is the output of subphase **F2.17.0 — Contract Lock / Scope
Freeze**. Implementation subphases (F2.17.1 onward) must match this contract
exactly; any deviation requires updating this document first.

---

## 1. Purpose

F2.17 prepares the backend so that F2.18 can begin turning the Admin Platform
Console from shell to functional app. F2.17 is **backend-only**. No Admin UI
is built in F2.17.

The diagnostic in the preceding session confirmed:

- F2.15 `/auth/users` already serves Admin Users globally — no rebuild needed.
- F2.16 `list_store_audit` aggregator is reusable for a global admin feed,
  with `AuditEventRead.store_id` already in the envelope.
- Store backend is minimal (single `GET` / `PATCH`); list, create, deactivate,
  and reactivate are missing.
- The Store model already carries `is_active`; lifecycle endpoints are viable
  without any model change or migration.

F2.17 closes the backend gaps required for Admin Stores and Admin Audit so
that F2.18 can build real Admin UI against real contracts.

## 3. Namespace decision

F2.17 keeps the existing `/stores` namespace for store CRUD and lifecycle.
A separate `/admin/stores` namespace is **not** introduced.

Rationale: F2.15 set the precedent by extending `/auth/users` rather than
creating `/admin/users`. The `require_store_member` and role guards already
applied to `/stores/{store_id}` continue to work; admin-only operations are
gated by `require_admin` at the route layer.

Final endpoint list for F2.17:

```txt
GET    /stores
POST   /stores
GET    /stores/{store_id}
PATCH  /stores/{store_id}
POST   /stores/{store_id}/deactivate
POST   /stores/{store_id}/reactivate
GET    /admin/audit
```

`GET /admin/audit` is the only `/admin/*` endpoint in F2.17 and exists
because there is no natural store-scoped path for a global feed.

## 5. Schemas

### 5.1 Schemas to add

**`StoreCreate`** — request body for `POST /stores`.

```txt
name:     str          required, trimmed, non-empty
code:     str          required, trimmed, non-empty
timezone: str          optional, default "America/New_York", trimmed, non-empty
```

Rules:

- `extra="forbid"`.
- No `id`, `is_active`, `created_at`, `updated_at` in payload.
- `name` and `code` validated as trimmed non-empty.
- Duplicate `code` surfaces as 422 via `IntegrityError` translation.

**`StoreListResponse`** — paginated list envelope for `GET /stores`.

```txt
items:  list[StoreRead]
total:  int            ge=0     pre-pagination count
limit:  int            ge=1
offset: int            ge=0
```

Shape mirrors `UserListResponse` and `AuditEventListResponse` for
consistency.

### 5.2 Schemas to keep unchanged

- **`StoreRead`** — response shape for all GET / PATCH / lifecycle endpoints.
  Fields: `id`, `name`, `code`, `is_active`, `timezone`, `created_at`,
  `updated_at`.
- **`StoreUpdate`** — PATCH payload. Stays limited to `name?` and `timezone?`.
  `code` remains immutable post-create. `is_active` lifecycle is **not**
  driven through PATCH.

## 6. Services

### 6.1 Services to add

**`list_stores(db, *, actor, is_active=None, q=None, limit=25, offset=0)`**

- Admin-only. Non-admin → 403.
- Filters:
  - `is_active`: optional bool.
  - `q`: optional ILIKE over `name` and `code` (case-insensitive,
    whitespace-stripped, empty `q` collapsed to `None`).
- Pagination: `1 <= limit <= 100`, `offset >= 0`.
- Sort: `created_at DESC`, then `id ASC` as stable tie-breaker.
- Returns `StoreListResponse` with `total` computed pre-pagination.

**`create_store(db, payload, *, actor)`**

- Admin-only. Non-admin → 403.
- Creates store with `is_active=True` (server default).
- Validates `code` uniqueness; duplicate or any `IntegrityError` → 422.
- No owner assignment in F2.17.
- No audit event in F2.17 (store source deferred).
- Returns `StoreRead`.

**`deactivate_store(db, store_id, *, actor)`**

- Admin-only. Non-admin → 403.
- Unknown store for admin → 404.
- Already inactive → **422** (not idempotent — explicit conflict).
- Sets `is_active=False`, commits, refreshes, returns `StoreRead`.

**`reactivate_store(db, store_id, *, actor)`**

- Admin-only. Non-admin → 403.
- Unknown store for admin → 404.
- Already active → **422** (not idempotent — explicit conflict).
- Sets `is_active=True`, commits, refreshes, returns `StoreRead`.

### 6.2 Services to keep unchanged

- `get_store(db, store_id)` — unchanged.
- `update_store(db, store_id, payload)` — unchanged.

### 6.3 Lifecycle decision (locked)

```txt
deactivate_store: already inactive → 422
reactivate_store: already active   → 422
```

Idempotency is **rejected** for F2.17. Conflict semantics are easier to test,
keep error matrices clear, and avoid silent no-ops. Future phases may
introduce idempotency keys if operationally needed.

## 7. Admin Audit service

### 7.1 Service to add

**`list_admin_audit(db, *, actor, store_id=None, source=None, entity_type=None, action=None, actor_id=None, date_from=None, date_to=None, limit=50, offset=0)`**

- Admin-only. Non-admin → 403.
- `store_id` provided:
  - Validate store exists.
  - Unknown store → 404.
  - Inactive store: returned events still valid (deactivated stores retain
    audit history). No 400 in this path.
  - Feed scoped to that store.
- `store_id` omitted:
  - Global feed across all stores.
- Reuses F2.16 normalization (`_query_inventory_events`,
  `_query_order_events`, `_query_compliance_events`).
- Reuses `AuditEventRead` and `AuditEventListResponse` envelopes.
- Pagination: `1 <= limit <= 200`, `offset >= 0`.
- Sort: stable two-pass — `(source, id)` ascending, then `created_at`
  descending. Identical to F2.16.

### 7.2 Audit schema changes

```txt
None.
```

`AuditEventRead.store_id` already exists from F2.16 and carries the
per-event store identifier across all three sources. No schema work is
required in F2.17.

### 7.3 Global audit fan-out behavior

| Source | Per-event `store_id` |
| --- | --- |
| inventory | `InventoryLog.store_id` (direct) |
| order | `OrderAuditLog.store_id` (direct) |
| compliance | `InventoryItem.store_id` via join through `ProductVariant` |

Compliance events may fan out once per affected store. De-duplication is
keyed on `(audit_log.id, store_id)` to keep one row per store carrying the
product. This matches F2.16 per-store semantics and is intentional for the
global feed.

### 7.4 Safety stance for global feed

```txt
Do not require date_from in F2.17.
Keep limit capped at 200.
Document the in-memory merge risk but do not gate implementation on it.
```

Operational caveat: a global query loads all matching events across all
stores into memory before sorting and slicing. Acceptable for current
deployment scale (low double-digit stores). Re-evaluate in a later phase if
store count grows or admins query unbounded ranges.

## 8. RBAC matrix

| Endpoint | admin | owner | manager | staff | driver |
| --- | --- | --- | --- | --- | --- |
| `GET /stores` | 200 | 403 | 403 | 403 | 403 |
| `POST /stores` | 201 | 403 | 403 | 403 | 403 |
| `GET /stores/{id}` | 200 any | 200 own / 403 cross | 200 own / 403 cross | 200 own / 403 cross | 200 own / 403 cross |
| `PATCH /stores/{id}` | 200 any | 200 own | 403 | 403 | 403 |
| `POST /stores/{id}/deactivate` | 200 | 403 | 403 | 403 | 403 |
| `POST /stores/{id}/reactivate` | 200 | 403 | 403 | 403 | 403 |
| `GET /admin/audit` | 200 | 403 | 403 | 403 | 403 |

Guards in use (no new guards introduced in F2.17):

```txt
require_admin
require_owner_or_admin
require_store_member
```

Anti-probe behavior:

- Unknown store for admin on any lifecycle endpoint → 404.
- Non-admin on any admin-only endpoint → 403 before existence is checked.
- Existing `require_store_member` anti-probe semantics on `GET` / `PATCH`
  `/stores/{id}` are preserved.


- **F2.17.1** — schemas only. No services, no routes.
- **F2.17.2** — services only. No routes. Schemas already in place.
- **F2.17.3** — routes wired to services from F2.17.2. RBAC matrix
  exhaustively tested at HTTP boundary.
- **F2.17.4** — `list_admin_audit` only. No route. Service tests cover
  global feed, store-scoped feed, filters, pagination, RBAC.
- **F2.17.5** — `GET /admin/audit` wired to F2.17.4 service. API tests
  cover query-param validation, RBAC, multi-store isolation.
- **F2.17.6** — full backend regression. No new endpoints. No frontend.
