# F2.27 Contract Lock

## Purpose

This document locks the scope, boundaries, subphase order, and validation
gates for **F2.27 — Staging / Production Readiness + Operational Backlog**.

F2.26 (Web App MVP closure) is shipped. F2.27 builds the readiness
infrastructure (RLS CI, staging, deploy, smoke/e2e) and the next set of
operational Web App features (regulatory aggregate/KPIs, store regulatory
read-only surface, writable admin settings, Excel inventory import,
automatic regulatory ingestion, QuickBooks connection), while keeping Stripe
as planning-only and leaving the Customer App and Driver App as future
surfaces.

This lock is authoritative. Any change that contradicts the Excluded Scope,
Hard No-Go Boundaries, or Stripe Readiness Boundary requires an explicit
scope change to this document first.

## Included Scope

F2.27 includes the following subphases (the Web App backlog only):

- **F2.27.0 — Contract Lock + Placeholder Cleanup**
- **F2.27.1 — RLS Active CI**
- **F2.27.2 — Staging Environment + Seed**
- **F2.27.4 — Deployment Readiness**
- **F2.27.3 — Manual Smoke + E2E Harness**
- **F2.27.5 — Regulatory Aggregate + KPIs**
- **F2.27.10 — Admin Settings Writable**
- **F2.27.6 — Store Regulatory Read-Only Surface**
- **F2.27.8 — Excel Inventory Import**
- **F2.27.7 — Automatic Regulatory Source Ingestion / FDA**
- **F2.27.9 — QuickBooks Inventory Connection**
- **F2.27.X — Stripe Readiness Roadmap** (planning doc only — no payment code)
- **F2.27 Final — Full Regression + Closure**

## Excluded Scope

The following are explicitly **out of scope** for F2.27 and must not be
implemented in any subphase:

- Customer App
- Driver App
- checkout
- PaymentIntent
- Stripe webhook
- payment capture
- delivery execution
- proof of delivery
- final age verification
- real customer payment flow
- real driver delivery flow

## Confirmed Subphase Order

The agreed implementation order (readiness/infra before features; external
dependencies last):

1. **F2.27.0** — Contract Lock + Placeholder Cleanup
2. **F2.27.1** — RLS Active CI (foundational gate)
3. **F2.27.2** — Staging Environment + Seed (incl. auth backfill)
4. **F2.27.4** — Deployment Readiness (backend host decision + runbook)
5. **F2.27.3** — Manual Smoke + E2E Harness
6. **F2.27.5** — Regulatory Aggregate + KPIs
7. **F2.27.10** — Admin Settings Writable
8. **F2.27.6** — Store Regulatory Read-Only Surface
9. **F2.27.8** — Excel Inventory Import
10. **F2.27.7** — Automatic Regulatory Source Ingestion / FDA
11. **F2.27.9** — QuickBooks Inventory Connection
12. **F2.27.X** — Stripe Readiness Roadmap (anytime; no code risk)
13. **F2.27 Final** — Full Regression + Closure

Rationale: readiness/infra (RLS → staging → deploy → smoke) must land before
feature work, because the only way to validate features beyond unit tests —
and to retire the F2.26 un-run manual smoke — is a real seeded staging stack
with enforced RLS. QuickBooks and automatic FDA ingestion are sequenced last
because they depend on external credentials/sources.

## Hard No-Go Boundaries

The following must never be touched or introduced under F2.27 unless this
lock is explicitly amended:

- No Customer App or Driver App surfaces.
- No checkout, no delivery execution, no proof of delivery, no final age
  verification.
- No real customer payment flow and no real driver delivery flow.
- No automatic compliance enforcement (see Compliance Boundary).
- No Stripe payment code (see Stripe Readiness Boundary).

## Stripe Readiness Boundary

Stripe is **planning-only** in F2.27:

- No PaymentIntent.
- No webhook.
- No checkout.
- No payment capture.
- No Stripe payment code of any kind.

F2.27.X delivers a Stripe readiness roadmap document only. Live payment
processing remains gated on Stripe account approval and the future Customer
App, both outside F2.27.

## Compliance Boundary

Regulatory intelligence remains human-gated:

- Automatic regulatory ingestion may create notices / signals / alerts only.
- No automatic hold/ban/block/compliance enforcement.
- Human admin decision remains required for all lifecycle actions
  (acknowledge / dismiss / resolve no_action / hold / ban).

## Validation Gates

Gates apply per subphase where relevant:

- Backend focused tests + full backend suite for backend-touching subphases.
- Frontend lint + typecheck + Vitest for frontend-touching subphases.
- RLS active CI for F2.27.1 (bypass-role provisioning → alembic upgrade →
  ordered Supabase migrations → SQL tests → active `test_rls_bypass.py` →
  full backend suite).
- Staging smoke / e2e after staging + deployment readiness land.
- RBAC / tenancy / security gates for risky features (store regulatory
  read-only surface, writable admin settings, Excel import).
- No-go grep / scope-integrity checks on every subphase diff (only allowed
  files changed; excluded areas untouched).
- Alembic check clean for any subphase introducing migrations.

## F2.27.0 Scope

F2.27.0 is limited to:

- Create this contract lock (`docs/f2.27-contract-lock.md`).
- Remove dead, unrouted frontend placeholder components:
  - `web/src/features/admin/pages/AdminPlaceholderPage.tsx`
  - `web/src/features/settings/pages/SettingsPlaceholderPage.tsx`
  - `web/src/features/settings/pages/__tests__/SettingsPlaceholderPage.test.tsx`
  - `web/src/layouts/components/FeaturePlaceholder.tsx`
- Remove the 11 dead `Admin*Placeholder` exports and the `AdminPlaceholderPage`
  import from `web/src/app/route-components.tsx`, preserving all real shells
  and redirects (`AdminShell`, `StoreShell`, `PublicShell`,
  `AppIndexRedirect`, `StoreRedirect`, `LegacyInventoryRedirect`,
  `LegacyOrderRedirect`, `LegacyProductRedirect`).
- Remove the `FeaturePlaceholder` import and `describe` block from
  `web/src/layouts/layout-shell.test.tsx`.
- Add a stable anti-reintroduction guard in `web/src/app/router.test.tsx`.

F2.27.0 does **not** touch backend, Supabase, CI workflows, deploy config,
navigation, regulatory features, inventory import, QuickBooks, Stripe code,
or any real route/page.

## Definition of Done

F2.27.0 is done when:

- `docs/f2.27-contract-lock.md` exists and reflects the locked scope.
- All four dead placeholder files are deleted.
- `route-components.tsx` exports only real shells/redirects; no
  `Admin*Placeholder` and no `AdminPlaceholderPage` reference remain.
- `layout-shell.test.tsx` no longer imports or tests `FeaturePlaceholder`.
- The router guard fails if any removed placeholder symbol is reintroduced.
- Frontend lint, typecheck, and the full Vitest suite pass.
- The diff contains only the allowed files; no excluded area is modified.

F2.27 as a whole is done when every included subphase meets its validation
gates and the F2.27 Final full regression + closure pass.
