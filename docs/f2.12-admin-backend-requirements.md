# F2.12 — Admin Backend Requirements

## 1. Purpose

This document defines the backend requirements needed to turn the
current NubeRush Admin Shell into a real Admin App.

The frontend already has these Admin surface foundations:

```txt
/app/admin/*
AdminLayout
ADMIN_NAV_ITEMS
Admin placeholders
No fake admin data
```

The backend still needs global or platform-level capabilities for:

- Stores
- Users
- Products
- Inventory
- Orders
- Audit
- Compliance
- Operations
- Settings
- Billing / commissions
- KPIs

Until these contracts exist, the frontend must keep honest placeholders
instead of simulating global platform data.

## 2. Non-goals

This document does not implement:

- Backend endpoints
- Frontend API clients
- Frontend hooks
- Fake admin data
- Frontend-side cross-store aggregation
- Customer checkout
- Cart
- Marketplace
- Payment checkout
- Driver app
- Mobile customer app

Frontend must not become the source of truth for roles, permissions,
tenant boundaries, stock, totals, compliance, or valid transitions.

## 4. Shared Backend Contract Standards

Admin endpoints should follow the standards in this section unless an
existing backend convention is already stricter or more specific.

### 4.1 Auth / authorization

- Admin endpoints require an authenticated user.
- Admin endpoints require platform-level admin authorization.
- Store users must not access global admin endpoints.
- Backend remains the source of truth for authorization.
- Frontend navigation is UX only.

### 4.2 Tenant boundaries

- Cross-store endpoints must be explicitly admin-scoped.
- Store-scoped endpoints must not leak other store data.
- Admin detail endpoints may include store context only when authorized.
- Store IDs in URLs are route context, not permission grants.

### 4.3 Pagination

List endpoints should support pagination.

Recommended response shape:

```txt
items
limit
offset
total
```

Recommended request parameters:

```txt
limit
offset
```

Safe max page sizes should be defined per endpoint.

### 4.4 Filtering

Global endpoints should support filters as applicable:

```txt
store_id
status
role
date_from
date_to
query
search
compliance_status
inventory_status
order_status
```

Each endpoint should document which filters are supported.

### 4.5 Sorting

Recommended request parameters:

```txt
sort_by
sort_direction
```

Each endpoint should document allowed `sort_by` values.
`sort_direction` should support `asc` and `desc`.

### 4.6 Error shape

Admin endpoints should follow the current backend/frontend error shape.
If a canonical backend error contract exists, reuse it.

The contract must consistently represent:

- Unauthorized
- Forbidden
- Not found
- Validation error
- Conflict
- Backend unavailable

Errors should include enough detail for user-facing retry, empty, and
permission states without leaking sensitive platform data.

### 4.7 Auditability

Future admin mutations must create audit logs.

Examples:

- Store creation
- Store deactivation
- User role changes
- Compliance review actions
- Settings changes
- Billing or commission changes

Audit entries should include actor, timestamp, resource, action, and
enough metadata to investigate platform changes.

### 4.8 Idempotency

Create store may require an idempotency key.

Admin actions that can be retried should be idempotent where practical.
This is especially important for destructive actions, billing settings,
and compliance review transitions.

### 4.9 Rate limiting / safety

Admin endpoints should be protected against accidental bulk misuse.

Safety requirements:

- Pagination limits
- Safe max page sizes
- Audit logging
- Confirmation requirements for destructive actions
- Rate limits for expensive global queries
- Clear conflict errors for concurrent updates

## 5. Stores Requirements

Stores are the core platform resource for Admin App management.

### 5.1 Store List

Endpoint:

```txt
GET /stores
```

Required filters:

```txt
status
query
created_from
created_to
```

Response fields:

```txt
id
name
status
created_at
updated_at
owner_count
primary_owner
active_users_count
products_count
orders_count
inventory_health
compliance_summary
```

Count and summary fields may be optional when expensive, but the
contract should state whether they are included.

### 5.2 Store Create

Endpoint:

```txt
POST /stores
```

Request fields:

```txt
name
legal_name
contact_email
contact_phone
address
status
```

Only `name` is expected to be required unless product policy requires
additional fields.

### 5.3 Store Detail

Endpoint:

```txt
GET /stores/:storeId
```

Response should include:

- Store profile
- Status
- Created timestamp
- Updated timestamp
- Operational summary
- Compliance summary
- Inventory summary
- Orders summary
- Users summary

### 5.4 Store Update

Endpoint:

```txt
PATCH /stores/:storeId
```

Updates should be audited. Validation errors should identify invalid
fields without exposing unrelated store data.

### 5.5 Store Deactivation

Recommended endpoints:

```txt
POST /stores/:storeId/deactivate
POST /stores/:storeId/reactivate
```

If the backend has a different action style, use the existing style.
The contract must define allowed transitions and conflict behavior.

### 5.6 Store Health Summary

Endpoint:

```txt
GET /stores/:storeId/health
```

This may also be included in `GET /stores/:storeId`.

Expected health categories:

- Store readiness
- Active user coverage
- Inventory risk
- Order risk
- Compliance risk
- Recent operational incidents

## 16. Suggested Backend Namespace

Recommended namespace for global admin endpoints:

```txt
/admin/*
```

Existing conventions may keep some resources outside `/admin/*`.
Examples:

```txt
/stores/*
/auth/users
```

Naming should match backend style, but global admin endpoints must be
clearly distinguishable from store-scoped endpoints.

## 17. Frontend Integration Plan

Frontend should consume these future endpoints only after backend
contracts exist.

Integration rules:

- Create API clients only after backend contract exists.
- Create query keys per admin feature.
- Create hooks per admin capability.
- Keep backend as source of truth.
- Preserve placeholders until endpoints are implemented.
- Replace placeholders incrementally by route.
- Do not fake global data while waiting for backend.

Suggested future frontend phases:

```txt
F2.13 — Admin Stores Backend + Frontend Integration
F2.14 — Admin Users Integration
F2.15 — Admin Global Orders/Inventory/Products Read-only Oversight
F2.16 — Admin Audit/Compliance/Operations
```

These phase names are planning suggestions only.

## 18. Open Questions

Backend and product must answer:

- Should global admin endpoints live under `/admin/*`?
- Should store CRUD live under `/stores/*` or `/admin/stores/*`?
- What roles can access Admin App?
- Is `admin` the only platform-level role?
- Do we need support, super-admin, or operator role separation?
- What is the canonical pagination shape?
- What is the canonical error shape?
- How should billing and commission data be modeled?
- Should compliance review actions be sync or async?
- What audit events are mandatory?
- Do admin endpoints need export support?
- Do global searches need full-text search?
- What is the max page size?

## 19. Definition of Ready for Admin Feature Implementation

A specific Admin frontend feature is ready only when:

```txt
Backend endpoint exists
Auth/RBAC behavior is defined
Request/response contract is documented
Pagination/filtering behavior is documented
Error states are documented
Audit expectations are defined
Tests can be written against real contract
No fake frontend aggregation is needed
```

## 20. Definition of Done for F2.12.8

F2.12.8 is done when:

```txt
Backend requirements document exists
All admin routes are mapped to backend needs
Shared backend standards are documented
No runtime code changed
No fake data introduced
No endpoints implemented
Admin next-phase planning is clear
```
