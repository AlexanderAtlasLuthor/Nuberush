# F2.27 Deploy Runbook

> Unified deploy readiness runbook for NubeRush backend (Render Web Service)
> and frontend (Cloudflare Pages). This document is **declarative readiness
> only** — it describes how a deploy is configured and gated. It does **not**
> execute a deploy.

## Purpose

F2.27.4 prepares NubeRush for a confident, repeatable backend/frontend deploy
by making the deploy configuration **declarative and reviewable**:

- A unified runbook (this file) describing the end-to-end deploy procedure.
- A root-level `render.yaml` blueprint describing the backend Render Web
  Service.

This phase **does not perform a real deploy**, does not change application
behavior, and does not auto-run database migrations. It only documents and
declares how the deploy should be performed when an operator chooses to run it.

## Deployment Boundaries

The following boundaries apply to F2.27.4 and to the deploy procedure it
describes:

- **No real deploy in this phase.** Nothing here triggers Render or Cloudflare
  to build or release. Operators run the steps manually when ready.
- **No automatic migrations from app startup.** The backend start command must
  never run Alembic or Supabase migrations. Migrations are a separate, gated,
  manual step (see Database Migration Gate / Supabase Migration Gate).
- **No secrets committed.** No real credentials appear in this repo — not in
  docs, not in `render.yaml`, not in frontend env, not in logs.
- **No service-role key in frontend.** The Supabase service-role key is
  server-only. The frontend uses only the public anon key.
- **No Stripe / payment code touched.**
- **No QuickBooks touched.**
- **No Excel import touched.**
- **No Customer App touched.**
- **No Driver App touched.**

## Backend Render Setup

The backend is deployed as a single Render **Web Service**. Configuration is
declared in the root `render.yaml` blueprint and mirrored here for reference:

| Setting            | Value                                                       |
| ------------------ | ----------------------------------------------------------- |
| Service type       | Web Service                                                 |
| Runtime            | Python                                                       |
| Root directory     | `backend`                                                   |
| Build command      | `pip install -r requirements.txt`                           |
| Start command      | `uvicorn app.main:app --host 0.0.0.0 --port $PORT`          |
| Health check path  | `/health`                                                   |
| Python version     | `3.12`                                                       |

Notes:

- The application entrypoint is `app.main:app` and already exists.
- The start command **must not** contain a migration step. Migrations are gated
  manually (see Database Migration Gate).
- `$PORT` is provided by Render at runtime; do not hardcode a port.

## Frontend Cloudflare Pages Setup

The frontend is deployed as a Cloudflare Pages project built from the `web`
directory:

| Setting            | Value          |
| ------------------ | -------------- |
| Root directory     | `web`          |
| Build command      | `npm run build`|
| Output directory   | `dist`         |
| Node version       | `20`           |
| Production branch   | `main`         |

Notes:

- The frontend already ships Cloudflare readiness assets (`_headers`,
  `_redirects`) and produces a `dist` output via `npm run build`.
- `_redirects` provides SPA deep-route fallback so refreshing a deep route does
  not 404.

## Environment Variables

Use placeholders only when documenting or configuring. **No real credentials.**

### Backend variables

| Variable                       | Notes                                            |
| ------------------------------ | ------------------------------------------------ |
| `DATABASE_URL`                 | Server-only. Secret. Set in Render dashboard.    |
| `SUPABASE_URL`                 | Supabase project URL.                            |
| `SUPABASE_SERVICE_ROLE_KEY`    | Server-only. Secret. Never in frontend.          |
| `BACKEND_CORS_ORIGINS`         | Comma/JSON list of allowed origins.              |
| `APP_ENV`                      | `staging` or `production`.                        |
| `APP_DEBUG`                    | `false` outside development.                       |
| `APP_PUBLIC_BASE_URL`          | Public base URL for auth callback origin.         |
| `EMAIL_ENABLED`                | `false` unless email is configured.               |
| `EMAIL_PROVIDER`               | e.g. `resend`.                                    |
| `EMAIL_FROM_ADDRESS`           | Sender address.                                  |
| `EMAIL_FROM_NAME`              | Sender display name.                             |
| `RESEND_API_KEY`               | Server-only. Secret. Only if email enabled.       |
| `SUPABASE_JWKS_URL`            | Optional. Overrides derived JWKS URL.            |
| `SUPABASE_JWT_ISSUER`          | Optional.                                         |
| `SUPABASE_JWT_AUDIENCE`        | Optional.                                         |

Placeholder examples (never commit real values):

```
DATABASE_URL=<server-only-database-url>
SUPABASE_URL=<supabase-project-url>
SUPABASE_SERVICE_ROLE_KEY=<server-only-service-role-key>
BACKEND_CORS_ORIGINS=<allowed-origins>
APP_ENV=staging
APP_DEBUG=false
APP_PUBLIC_BASE_URL=<public-base-url>
EMAIL_ENABLED=false
RESEND_API_KEY=<server-only-resend-api-key>
```

### Frontend variables

| ------------------------- | ---------------------------------- |
| `VITE_API_BASE_URL`       | Backend base URL (Render service). |
| `VITE_SUPABASE_URL`       | Supabase project URL.              |
| `VITE_SUPABASE_ANON_KEY`  | Public anon key only.              |

Placeholder examples:

```
VITE_API_BASE_URL=<backend-base-url>
VITE_SUPABASE_URL=<supabase-project-url>
VITE_SUPABASE_ANON_KEY=<public-anon-key>
```

### Frontend forbidden variables

The following must **never** be exposed to the frontend build or bundle:

- `DATABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `RESEND_API_KEY`
- `NUBERUSH_BACKFILL_PASSWORD`
- Any Postgres credential
- Any server-only secret

The frontend only ever receives the public Supabase anon key
(`VITE_SUPABASE_ANON_KEY`) and non-secret configuration.

## CORS Setup

`BACKEND_CORS_ORIGINS` must explicitly list the allowed frontend origins. The
backend rejects an empty list and rejects `*` outside development.

Staging example:

```
APP_ENV=staging
BACKEND_CORS_ORIGINS=https://<staging-project>.pages.dev
```

Production example:

```
APP_ENV=production
BACKEND_CORS_ORIGINS=https://<production-domain>
```

Rules:

- Do **not** use `BACKEND_CORS_ORIGINS=*` outside development.
- Do **not** leave `BACKEND_CORS_ORIGINS` empty outside development.

## Database Migration Gate

Alembic migrations are applied **manually**, before the backend deploy, never
from the Render start command.

```
cd backend
python -m alembic upgrade head
```

Explicit rules:

- **Do not** run Alembic from the Render start command.
- **Do not** migrate using a runtime role that is not the schema owner.
- The migration gate **must happen before** the backend deploy that depends on
  the new schema.

## Supabase Migration Gate

Supabase SQL migrations (RLS policies, storage rules, etc.) are **not** managed
by Alembic and are applied separately.

- Apply Supabase SQL migrations separately, in timestamp order.
- Supabase RLS / storage migrations are not Alembic migrations and have their
  own apply step.
- F2.27.1 `rls-active` CI validates the RLS baseline in CI, but the staging /
  production database still needs its Supabase migrations applied manually
  before deploy.

## Health Check

The backend exposes a liveness endpoint used by Render's health check:

```
GET /health
```

Expected:

- HTTP `200`
- Body contains `status: ok`

## Post-Deploy Verification

After a deploy, verify:

- [ ] `GET /health` returns `200`.
- [ ] Frontend loads from Cloudflare.
- [ ] Deep route refresh works (no 404 on refresh of a nested route).
- [ ] Network tab does **not** call `localhost`.
- [ ] No CORS errors in the browser console.
- [ ] No `SUPABASE_SERVICE_ROLE_KEY` or `DATABASE_URL` appears in the frontend
      env or bundle.
- [ ] Auth callback origin matches `APP_PUBLIC_BASE_URL`.
- [ ] Backend logs show no secret values.

## Rollback

High-level safe rollback (do not rollback the database blindly):

- **Backend:** use Render → roll back to the previous successful deploy.
- **Frontend:** use Cloudflare Pages → roll back to the previous deployment.
- **Database:** do **not** rollback the database blindly after migrations have
  been applied.
- If a migration rollback is genuinely required, treat it as a **separate
  manual incident process** with its own review — not part of the standard
  deploy rollback.

## No-Secret Rules

The following must never appear in committed files (docs, `render.yaml`),
frontend env, or logs:

- No real `DATABASE_URL`.
- No real `SUPABASE_SERVICE_ROLE_KEY`.
- No real `RESEND_API_KEY`.
- No private keys.
- No tokens.
- No credentials of any kind in docs, `render.yaml`, frontend env, or logs.

Use placeholders such as `<server-only-service-role-key>` when documenting.
