# F2.25 Contract вҖ” Real Email Delivery, Owner Activation & Notification Foundation

Status: contract / scope freeze. This document is the output of subphase
**F2.25.0 вҖ” Contract / Scope Freeze**. It locks the scope, boundaries,
security rules, non-goals, and per-subphase acceptance criteria for F2.25.

No runtime code, backend logic, frontend logic, tests, provider packages,
env secrets, or `.env` files are introduced or modified by this document.
Implementation subphases (F2.25.1 onward) must match this contract exactly;
any deviation requires updating this document first.

This contract builds on the F2.24.C8 email seam (mock/log-only) and the
F2.22 hybrid Supabase + FastAPI architecture. It does not supersede them; it
extends the email seam toward real delivery and adds owner activation and a
minimal notification foundation.

---

## 1. Title

**F2.25 вҖ” Real Email Delivery, Owner Activation & Notification Foundation.**

F2.25 turns the existing transport-free business email seam into a real
delivery path, wires owner account activation and password flows to Supabase
Auth, persists an `email_triggered` audit, and lays a minimal, email-only
notification foundation вҖ” without rewriting the business core or breaking
the protected sender/template boundaries.

---

## 2. Purpose

The problems F2.25 solves:

- **Business emails do not actually send.** `app.services.email_sender`
  ships only `_LoggingEmailSender`: no transport, no provider, no network
  call. The store-application lifecycle (submitted / approved / rejected)
  logs that an email *would* fire but nothing reaches the recipient.
- **Owners cannot activate their account.** An approved application points
  the owner at "the official account-activation flow," but that flow does
  not yet exist end to end. There is no set-password / first-login path.
- **There is no durable record that an email fired.** The audit log model
  reserves an `email_triggered` event type but no subphase writes it, so
  delivery attempts are not auditable.
- **There is no notification foundation.** Beyond the three store-application
  events there is no shared email-event / template / dispatch shape to grow
  future business notifications from.

What F2.25 deliberately does **not** try to solve: any notification center,
SMS, push, scheduling, queueing, retries, or dead-letter system. See В§14.

---

## 3. Current Context

Grounding facts as of this freeze (verify before each subphase):

- `backend/app/services/email_sender.py` defines the protected seam:
  `BusinessEmailEvent` (a closed `Literal` of the three store-application
  events), `BusinessEmailMessage` (frozen dataclass: `event_type`,
  `to_email`, `subject`, `body`), the `EmailSender` `Protocol`, the
  `EmailSenderError` type, the `_LoggingEmailSender` implementation, the
  private module-level `_default_sender`, and the single entry point
  `send_business_email(message)`.
- `backend/app/services/email_templates.py` holds pure, I/O-free builders
  that turn a `StoreApplication`-like object into a `BusinessEmailMessage`.
  Content rules are enforced here and by the C8 tests: branded, no plaintext
  passwords, no auth tokens, no Supabase/internal detail.
- `backend/app/services/store_applications.py` calls `send_business_email`
  only from post-commit seams; every seam swallows `EmailSenderError` and any
  other exception so a committed application still returns its success
  response.
- `backend/app/db/models.py` defines `store_application_audit_logs` and notes
  that `email_triggered` audit rows "are produced in later subphases."
- Auth is Supabase (F2.22 hybrid): Supabase owns sessions and account/auth
  emails; FastAPI owns business logic. RLS is defense-in-depth only.
- `backend/tests/test_store_application_emails.py` pins the current seam
  behavior (post-commit, swallowed failures).

Key decisions (all binding):

- **FastAPI owns business email events.** Track A is FastAPI's authority.
- **Supabase Auth owns account/auth emails.** Activation, invite, reset,
  recovery, and set-password emails are Track B and belong to Supabase Auth.
- **The existing `EmailSender` seam must remain.** The `Protocol`, the
  `BusinessEmailMessage` shape, `send_business_email`, and the call sites do
  not change their contract.
- **The real provider lives in a new backend module.** It implements the
  `EmailSender` `Protocol`; it must not break the protected sender/template
  boundaries or move logic into the templates module.
- **Provider config is server-only.** No provider key, secret, or endpoint
  appears in frontend code or frontend env files.
- **Business emails fire only after commit.** No email send or audit write
  occurs inside the business transaction.
- **Provider and audit failures are swallowed/logged.** They must never roll
  back committed business state.

## 5. Track A вҖ” Business Emails

Scope: the closed set of business-event emails FastAPI sends. F2.25 starts
from the three existing store-application events and may grow the closed
`Literal` set only through the notification foundation (В§11, F2.25.7).

Rules:

- Track A emails are always built by `email_templates` (pure functions) and
  sent through `send_business_email` (the seam). No call site builds raw
  provider payloads.
- Track A emails carry **no auth tokens** and **no plaintext passwords**.
  When a business email must direct an owner to activate or set a password,
  it links to a frontend route that initiates the Supabase Auth flow; the
  token lives in the Supabase-generated link, not in FastAPI-authored copy.
- Adding a new business event requires extending the `BusinessEmailEvent`
  `Literal`, adding a pure template builder, and a post-commit call site вҖ”
  in that order вҖ” and updating this contract first.

---

## 6. Track B вҖ” Account / Auth Emails

Scope: invite, owner activation, set-password, password reset, and account
recovery emails.

Rules:

- These emails are **sent by Supabase Auth**, not by FastAPI and not through
  the `EmailSender` seam.
- FastAPI may **trigger** a Supabase Auth action (e.g. invite / generate
  link) via the server-side Supabase admin path, but never composes or sends
  the auth email itself.
- **No custom password reset / activation token system is built in FastAPI.**
  All auth tokens are Supabase-issued.
- The frontend gains shared auth-callback and set-password routes (later
  subphases) that consume the Supabase-issued link and complete the flow.

---

## 7. Business Email Flow

1. A business request mutates state inside a DB transaction (e.g. application
   submit / approve / reject).
2. The transaction **commits**.
3. A **post-commit seam** runs, outside the transaction:
   a. `email_templates` builds the `BusinessEmailMessage` from the persisted
      row (pure, no I/O).
   b. `send_business_email(message)` dispatches through the seam to the real
      provider module.
   c. An `email_triggered` audit row is written in an isolated, non-blocking
      path (В§9).
4. Any failure in steps 3aвҖ“3c is **caught, logged, and swallowed**. The
   committed business state stands and the request returns its success
   response regardless.

No step 3 work may execute before the commit, and no step 3 failure may
trigger a rollback or change the response status.

---

## 8. Account/Auth Email Flow

1. A business action determines an owner needs account access (e.g. an
   application is approved).
2. After the business transaction commits, FastAPI triggers the appropriate
   **Supabase Auth** action (invite / generate activation link) through the
   server-side admin path.
3. **Supabase Auth sends** the activation / set-password / reset / recovery
   email.
4. The owner clicks the Supabase-issued link and lands on a shared frontend
   **auth-callback / set-password route**, which completes the flow against
   Supabase Auth.
5. Triggering the Supabase Auth action is subject to the same post-commit,
   swallow-and-log discipline as Track A: a Supabase trigger failure must not
   roll back the committed business state.

---

## 9. Email Audit Contract

- The `email_triggered` audit is persisted using the existing
  `store_application_audit_logs` model (and equivalent audit shape for any
  later notification events), with `event_type = "email_triggered"`.
- Audit rows are written **post-commit**, in an **isolated, non-blocking**
  path вҖ” never inside the business transaction and never on the request's
  critical path.
- The audit row records that an email was triggered: event type, recipient,
  and non-PII context. It records **no** plaintext passwords, **no** auth
  tokens, **no** full body dumps, and **no** provider secrets.
- **Audit write failures are swallowed and logged.** A failed audit write
  never rolls back the business commit and never blocks the response.
- The audit records that delivery was *attempted/triggered*; it is not a
  proof-of-delivery / bounce-tracking system (out of scope, В§14).

## 10. Store Onboarding Scope

In scope for F2.25:

- A simple onboarding **landing route** in the frontend.
- A minimal onboarding **checklist** presented on that route.
- An **email link** that directs an approved owner to onboarding /
  activation (the link initiates the Supabase Auth set-password flow; it
  carries no FastAPI-issued token).

Out of scope: any multi-step onboarding wizard, document upload, KYC,
contract signing, or provisioning automation beyond the existing flow.

Store onboarding remains application-gated: there is no self-signup; owners
apply, the team reviews, and the backend provisions. Onboarding CTAs route
to the existing apply / activation paths, not to a self-registration form.

---

## 11. Notification Foundation Scope

In scope for F2.25:

- A shared **email-event / template / dispatch** shape that the existing
  business emails and any new F2.25 business emails conform to.
- Extending the closed `BusinessEmailEvent` `Literal` and adding pure
  template builders for any new email events introduced this phase.

That is the entire notification foundation: **email events, templates, and
dispatch only**. See В§14 for everything excluded.

---

## 12. Provider Configuration Rules

- The real email provider is configured **server-side only**.
- Provider configuration (API key, endpoint, sender identity) is read from
  **backend** server configuration at runtime. F2.25.0 introduces **no**
  secrets and does **not** touch real `.env` files; later subphases document
  required config keys without committing secret values.
- **No provider package is added in F2.25.0.** The provider-library decision
  is made in F2.25.1 and remains a backend-only dependency.
- The provider integration lives in a **new backend module** implementing the
  `EmailSender` `Protocol`. It replaces `_LoggingEmailSender` as the wired
  `_default_sender` without changing the seam's public contract or the
  template module.
- A safe default remains: if the provider is not configured, the seam falls
  back to log-only behavior rather than failing the request.

---

## 13. Security Constraints

Binding for every F2.25 subphase:

- **No provider key may ever appear in frontend code or frontend env files.**
  Provider secrets are server-only.
- **No plaintext passwords** anywhere вҖ” not in emails, logs, audit rows, or
  responses.
- **No auth tokens inside business (Track A) emails** unless explicitly
  approved in a later phase. Auth tokens live only in Supabase-issued links.
- **No custom password reset / activation token system in FastAPI.** Owner
  activation and set-password use **Supabase Auth**.
- Business emails and audit writes run only **post-commit**; provider and
  audit failures are **swallowed and logged** and never roll back committed
  state.
- The protected sender/template boundary is preserved: templates stay pure
  and secret-free; the provider stays out of the template module.
- PII posture follows the C8 rule: recipient and branded subject may be
  logged; phone numbers, postal addresses, tokens, internal IDs, and full
  body dumps are not.

---

## 14. Explicit Non-Goals

F2.25 does **not** build any of the following:

- A notification center or in-app notification inbox.
- SMS or push notifications.
- A scheduler, cron, or time-based send.
- A queue, worker, or async job system for email.
- Retries, backoff, or a dead-letter system.
- Delivery / bounce / open / click tracking or webhooks.
- A custom FastAPI password reset or activation token system.
- Any change to the orders, inventory, RBAC, tenancy, compliance, earnings,
  or dashboard business logic.
- Self-signup for store owners.
- Replacing or bypassing Supabase Auth for account/auth emails.

## 17. Risk Register

- **Provider key leaks into frontend code/env** вҖ” *Critical, secret
  exposure.* Mitigation: server-only config (В§12); review every subphase diff
  for frontend secret references.
- **Email send/audit inside the business txn** вҖ” *High, committed state could
  roll back on a notification failure.* Mitigation: post-commit seams only
  (В§7, В§9); swallow-and-log discipline.
- **Provider failure rolls back business state** вҖ” *High, lost legitimate
  writes.* Mitigation: `EmailSenderError` swallowed by existing post-commit
  seams; commit precedes any send.
- **Template module gains provider/secret logic** вҖ” *Medium, breaks protected
  boundary and risks a PII/secret sink.* Mitigation: provider lives in a new
  module; templates stay pure (В§4, В§13).
- **Custom FastAPI auth-token system creeps in** вҖ” *High, duplicate and weaker
  auth surface.* Mitigation: Track B is Supabase-only; no FastAPI token
  system (В§6, В§13).
- **Auth token embedded in a business email** вҖ” *High, token exposure.*
  Mitigation: no tokens in Track A emails; tokens live in Supabase-issued
  links (В§5).
- **Audit write failure blocks/rolls back request** вҖ” *Medium, request fails
  for a logging problem.* Mitigation: isolated, non-blocking audit; failures
  swallowed/logged (В§9).
- **Scope creep into notification center / queue / SMS** вҖ” *Medium, phase
  overrun.* Mitigation: explicit non-goals (В§14); deviations require a
  contract update first.
- **PII over-logging** вҖ” *Medium, privacy exposure.* Mitigation: maintain the
  C8 PII posture вҖ” recipient and branded subject only (В§13).
