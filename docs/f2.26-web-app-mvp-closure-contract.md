# F2.26 — Web App MVP Closure Contract

Status: contract / scope freeze. This document is the output of subphase
**F2.26.0 — Web App MVP Closure Contract Lock / Scope Freeze**. It freezes
the exact scope, boundaries, security rules, non-goals, and per-subphase
acceptance criteria for F2.26 **before any implementation begins**.

No runtime code, backend logic, frontend logic, tests, migrations, provider
packages, env secrets, or `.env` files are introduced or modified by this
document. Implementation subphases (F2.26.1 onward) must match this contract
exactly; any deviation requires updating this document first.

This contract builds on the existing operational core: the transactional
Inventory Engine, the Order Engine, the Compliance gate, the unified audit
feed (`app/services/audit.py`), the F2.22 hybrid Supabase + FastAPI auth, the
F2.24 store-application provisioning, and the F2.25 email/notification
foundation. It does not supersede any of them; it closes the Web App as the
**Operational Core MVP** and lays declared, bounded foundations for
operational traceability, product imagery, and regulatory intelligence.

---

## 1. Purpose

F2.26 exists to declare the NubeRush **Web App** feature-complete as an
internal Operational Core MVP and to draw a hard line between it and the two
future mobile surfaces (Customer App, Driver App) and real payments (Stripe).

The problems F2.26 closes:

- **The operational scope is implicitly "still growing."** There is no single
  document that says the Web App is *done* as an internal panel and enumerates
  what "done" means. F2.26 freezes that definition.
- **Operational traceability has a gap.** The unified audit feed today
  aggregates only inventory, order, and product-compliance events. User
  management (create / update / role change / store assignment / activation)
  and store lifecycle (create / update / activation) are **not** auditable.
  For a compliance-posture business this is material.
- **Product imagery is partial.** A `ProductImage` data model, storage seam,
  RLS, and an admin upload panel already exist, but the surface is not closed
  end to end (store-side surfacing, placeholders, change audit, validation
  contract).
- **Panel language is uneven.** Demo placeholders, technical wording, and the
  risk of implying live payments/delivery remain in the Admin and Store
  panels.
- **There is no regulatory intelligence base.** NubeRush has no structured way
  to ingest a regulatory notice, match it against the catalog, and route it to
  a human reviewer — while never auto-banning a product without review.
- **Payment status is ambiguous.** Stripe approval is pending; the contract
  must guarantee the Web App neither processes nor pretends to process real
  payments, and that this does not block MVP closure.

What F2.26 deliberately does **not** solve: see §4 (Explicit No-goals).

## 2. Product Boundary

NubeRush is a multi-surface platform for vape retailers. F2.26 concerns
**only** the Web App. The boundary between surfaces is frozen as follows.

- **Web App (this contract):** the internal operational core. Used by `admin`,
  `owner`, `manager`, and `staff`. Controls products, inventory, orders,
  compliance, users, audit, settings, and earnings reporting. It is an
  *operations console*, not a point of sale, not a checkout, not a payment
  processor, not a delivery dispatcher.
- **Customer App (future, out of scope):** customer catalog, cart, checkout,
  payments, customer account, order tracking, customer-facing notifications,
  and any customer-side age/compliance pre-check.
- **Driver App (future, out of scope):** delivery assignments, pickup,
  navigation, **final** age/ID verification, proof of delivery, delivery
  completion, and the failed-delivery / return flow.

Cross-surface invariants that F2.26 must respect and document:

- The Web App **does not** execute a real checkout.
- The Web App **does not** process real payments.
- The Web App **does not** perform delivery execution.
- The Web App **does not** replace, preview, or stub the Customer App or the
  Driver App.
- `reserved` stock lives as `quantity_reserved` on the inventory item, never as
  an item status.
- `returned` and `damaged` are inventory **movements**, never item statuses.
- Final age/ID verification and proof of delivery are **Driver App**
  responsibilities, not Web App ones.

## 4. Explicit No-goals

F2.26 **does not** include, and any implementation subphase that touches the
following is out of contract:

- Customer App (any screen, route, or endpoint).
- Driver App (any screen, route, or endpoint).
- Stripe `PaymentIntent` creation or handling.
- Stripe webhooks or any payment-provider callback.
- Real checkout of any kind.
- Real payment capture, authorization, or settlement.
- Refunds or chargebacks.
- Delivery execution or dispatch.
- Driver navigation or routing.
- Proof of delivery capture.
- Push notifications (mobile).
- SMS notifications.
- **Auto-ban / irreversible compliance action without human review.**
- Public customer catalog.
- Customer order tracking.
- Mobile app store release (iOS / Android).

The regulatory intelligence foundation (§9) is explicitly a *human-gated*
alerting base. It must never, in F2.26, block or ban a product automatically.

Required Admin surfaces:

- Store list and Store detail (lifecycle, status).

Required Store surfaces:

- Products and Product Detail (store-scoped visibility + proposals).
- Inventory (store-scoped, with movement actions).
- Orders (store-scoped, with create + lifecycle actions).
- Users (store team members).

- Audit (store-scoped unified feed).
- Settings (store).
- Earnings (store) — already present; stays read-only and backend-computed.

Closure invariants (must be explicitly true and, where testable, asserted in
the QA phase):

- The Web App **controls internal operation** of products, inventory, orders,
  compliance, users, and audit.
- The Web App **does not execute real checkout**.
- The Web App **does not process real payments**.

- The Web App **does not replace** the Customer App or the Driver App.

  values only** until Stripe is live (see §12).

## 6. Operational Audit Requirements

F2.26 closes the operational traceability gap. The current unified feed
(`app/services/audit.py`) aggregates three append-only sources — inventory
logs, order audit logs, product compliance audit logs — into the normalized
`AuditEventRead` shape and exposes them through `list_store_audit`
(store-scoped) and `list_admin_audit` (global). F2.26 must add operational
events for user management and store lifecycle **using the same architecture**.

### 6.1 Events that must be captured

User events:

- User created.
- User updated (profile fields).
- User role changed.
- User assigned to a store.
- User removed from a store.
- User activated.
- User deactivated.

Store events:

- Store created.
- Store updated.
- Store activated.
- Store deactivated.

Compliance / regulatory events:

- Regulatory / compliance decision recorded (review, mark not applicable,
  request manual check, apply compliance hold, ban/block). These must reconcile
  with, and not duplicate, the existing product-compliance audit source.

Settings events:

- Critical settings changes, **only where a change is security- or operations-
  relevant** (e.g. a toggle that affects RBAC, tenancy, or compliance posture).
  Cosmetic or per-user preference changes are out of scope.

### 6.2 Event shape (contract, not a migration)

Every operational audit event must record at least:

- `actor_user_id` — who performed the action (nullable only for system actions;
  if nullable is used it must be justified per event type).
- `target_type` — the kind of entity acted on (e.g. user, store, setting).
- `target_id` — the id of the acted-on entity.
- `action` — a closed, documented verb per event type.
- `before` / `after` — the changed values **when safe to store**. Secrets,
  password material, auth tokens, and full PII payloads must never be written.
  When `before`/`after` cannot be stored safely, the event records the fact of
  the change without the sensitive values.
- `store_id` — **nullable**. Set for store-scoped events; `NULL` for global
  events (e.g. admin-global actions, cross-store changes).
- `metadata` — structured, JSON-safe, event-specific context bag.
- `created_at` — append-only timestamp.

Integrity rules:

- Events are **append-only**. No update, no delete path may be exposed.
- The event store must normalize into the existing `AuditEventRead` shape so it
  flows through the unified feed without a parallel read path. The mapping is:
  `actor_user_id → actor_id`, `target_type → entity_type` (or an equivalent
  documented binding), `target_id → entity_id`, plus `action`, `store_id`,
  `metadata`, `created_at`, and a human `summary`.
- A new `AuditSource` value (e.g. `operational`) is introduced for these
  events; existing source values are not repurposed.

### 6.3 Visibility rules

- **Admin (global)** sees every operational event across all stores, including
  global events with `store_id = NULL`.
- **Store users** (`owner` / `manager` / `staff`) see operational events for
  **their own store only**.
- **Global events** (`store_id = NULL`) are visible to **admin only** and never
  surface in a store-scoped feed.
- `driver` never sees the audit surface (consistent with the current feed).
- Cross-store and unknown-store access for non-admins collapses to the same
  response, preserving the existing existence-probe protection.

## 7. Product Image Requirements

A `ProductImage` model (one primary image per product, enforced by a unique
constraint), a Supabase Storage seam, storage RLS, and an admin product-image
panel already exist. F2.26 **closes** this surface; it does not rebuild it.

The closed product-image contract must cover:

- **Primary image per product** — exactly one, consistent with the existing
  one-image-per-product constraint.
- **Placeholder** — every product surface (admin + store, list + detail)
  renders a neutral placeholder when no image exists. No broken image, no
  layout shift.
- **Upload or secure association** — image bytes live in storage; the business
  metadata row is owned by the backend. Association is authorized server-side.
- **File-type validation** — only an allowed set of image MIME types is
  accepted (the exact allow-list is fixed in the implementation subphase and
  documented there; it must reject non-image and active-content types).
- **File-size validation** — a maximum size is enforced server-side; the limit
  is fixed and documented in the implementation subphase.
- **Frontend preview** — the operator sees a preview before and after
  association.
- **Safe change / removal** — replacing or removing an image is authorized,
  atomic, and leaves no orphaned primary reference.
- **Image-change audit** — create / replace / remove of a product image emits
  an operational audit event (§6) tied to the product.
- **No regression** — products without an image continue to function
  everywhere; image presence is never required to read, sell, or manage a
  product.
- **Compliance independence** — the presence, absence, or content of an image
  **does not** affect `compliance_status`, `allowed_for_sale`, sellability, or
  any compliance decision.

## 8. Admin/Store Language Polish Requirements

F2.26 polishes all Admin and Store panel copy to read as a professional
operational platform.

Required outcomes:

- **Remove demo placeholders** — no lorem-ipsum, no "coming soon" stubs left in
  shipped surfaces, no fake data labels.
- **Remove unnecessary technical wording** — internal phase codes, schema
  names, and engineering jargon do not appear in user-facing copy.
- **No false promises** — copy must not imply that payments are live, that
  delivery is executed by the Web App, or that customer/driver flows exist.
- **Professional, role-appropriate copy** for owners, managers, staff, and
  admins.
- **Consistency** between Admin and Store panels (shared terminology,
  capitalization, tone).
- **Clear loading / empty / error states** on every data surface.
- **Business-appropriate labels** rather than literal field names.

Tone guidance to encode:

- The **Admin Panel** reads as a control/oversight platform. Representative
  vocabulary: "Compliance alerts", "Inventory risk", "Store status", "Product
  approval", "Operational audit".
- The **Store Panel** reads as operational. Representative vocabulary:
  "Available stock", "Reserve inventory", "Prepare order", "Update product",
  "Review compliance hold".

This section governs copy only. It must not change business logic, routes,
permissions, or data shapes.

## 9. Regulatory Intelligence Requirements

F2.26 lays a **human-gated** foundation that lets NubeRush react to regulatory
communications about vape products. It must **never** auto-block or auto-ban a
product without human review in F2.26.

The foundation must cover, as contract/boundaries (real scraping is **not**
required in F2.26.0; later subphases may add ingestion within these bounds):

- **Configurable regulatory source(s)** — a source is a configured, named
  origin of notices; sources are data, not hard-coded.
- **Regulatory notices / communications** — an ingested notice is a stored,
  structured record (title, source, reference, issued date, body/summary,
  ingestion timestamp).
- **Matching against internal products** — a notice can be matched (by
  name/brand/category/identifier heuristics, defined in the implementation
  subphase) against catalog products, producing candidate matches.
- **Compliance alerts** — a match produces an alert that enters a review queue.
- **Admin review gate** — every alert requires an explicit human decision; no
  state advances automatically to a blocking action.
- **Decision actions** available to the reviewer:
  - review (acknowledge / open),
  - mark not applicable,
  - request manual check,
  - apply compliance hold,
  - ban / block product.
  The hold/ban/block actions route through the **existing** compliance path
  (`set_product_compliance` semantics) so the established compliance audit and
  inventory-quarantine cascade still apply.
- **Audit of every alert and decision** — alert creation and each reviewer
  decision emit operational audit events (§6).
- **Internal notification** — new alerts and required-decision states notify
  the relevant admin / store owner over allowed channels (§10).
- **Preparation for future automation** — the data shapes and decision states
  must be expressible such that a future phase could add automation behind a
  policy flag, but F2.26 ships none.
- **No irreversible auto-ban** — restated as a hard invariant.

Conceptual initial sources (documented as intent; no live integration
required in F2.26.0):

- FDA Center for Tobacco Products.
- FDA authorized ENDS / product lists.
- FDA enforcement / advisory actions.
- Retailer compliance guidance.

## 10. Compliance Notification Requirements

F2.26 surfaces internal notifications for the regulatory/compliance lifecycle.

Required notification triggers:

- New regulatory notice ingested.
- Product match detected.
- Product recommended for hold.
- Admin decision required.
- Product blocked / held **after** admin confirmation.

Allowed channels in F2.26:

- **Dashboard alert** (Admin and/or Store dashboard surfaces).
- **Internal notification center**, if one exists or is created within this
  phase. If created, it is internal-only and read by Web App operators.
- **Email**, **only** if the existing email/notification foundation
  (`app/services/notification_emails.py`, F2.25) already supports the case.
  F2.26 does not build a new transport.

Explicitly excluded channels:

- Mobile push notifications.
- SMS.
- Any customer-facing notification.

Notifications carry no compliance authority of their own: they inform
operators and link to the human-review gate (§9); the actual state change still
flows through the compliance path with its audit and cascade.

## 12. Pre-Stripe Readiness Requirements

NubeRush is awaiting Stripe approval. F2.26 may close **without** Stripe,
provided every one of the following holds:

- No real payments are processed anywhere in the Web App.
- Nothing in the UI or API pretends payment processing is active.
- Order totals, earnings, and commission figures are retained as **internal
  reporting values** only — a base for future billing, not a live ledger.
- Any payments-related copy states clearly that Stripe / payment activation is
  **pending**.
- There is no `PaymentIntent`, no payment webhook, and no real checkout.

Stripe integration is a separate, later phase and is a No-goal here (§4).

## 13. Security / RBAC / Tenancy Requirements

Every F2.26 implementation subphase must preserve the established security
model:

- **Backend authority** — all authorization, tenancy, compliance, and audit
  decisions are made server-side. The frontend never gates on business rules.
- **RBAC** — the existing role hierarchy and matrices
  (`app/core/permissions.py`, `app/api/deps.py`) are the source of truth. New
  operational-audit reads and regulatory-review actions must be gated by the
  appropriate role dependency (admin-only for global; store-member for
  store-scoped). Regulatory decisions that change compliance state require at
  least the privilege the existing compliance path requires.
- **Tenancy** — store users see only their store's data; admins are global.
  Operational audit visibility (§6.3) and notification targeting follow this.
- **Append-only audit** — no exposed update/delete on any audit source.
- **No sensitive data in audit/metadata** — no passwords, tokens, secrets, or
  full PII in `before`/`after`/`metadata`.
- **Admin identity invariant** — the `@nuberush.com` admin-domain rule and the
  last-active-admin guard remain intact; no F2.26 surface weakens them.
- **Storage authorization** — product-image upload/association/removal is
  authorized server-side and respects the existing storage RLS.
- **Supabase auth** — identity remains Supabase JWT → `auth_user_id`; no legacy
  auth is reintroduced.

## 16. Definition of Done

F2.26.0 (this subphase) is **done** when:

- `docs/f2.26-web-app-mvp-closure-contract.md` exists with all 18 mandated
  sections.
- No runtime, migration, test, package, or env file is changed.
- The validation commands in the report (§ report) have been run and reported
  honestly, including an honest statement if a tool is unavailable.

F2.26 (the whole phase) is **done** when:

- Every subphase F2.26.1–F2.26.8 is complete and conforms to this contract.
- The operational audit gap is closed and integrated into the unified feed
  with correct visibility (§6).
- Product images are closed end to end with audit and placeholders (§7).
- Admin/Store copy is polished with no demo placeholders or false
  payment/delivery promises (§8).
- A human-gated regulatory intelligence foundation exists with no automated
  irreversible action (§9) and internal notifications over allowed channels
  (§10).
- PRD alignment and Stripe-pending posture are documented (§11, §12).
- All §14 QA gates pass.
- Every No-goal in §4 remains untouched.
