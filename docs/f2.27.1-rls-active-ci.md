# F2.27.1 — RLS Active CI

## Purpose

F2.27.1 adds the repository's first CI workflow (`.github/workflows/ci.yml`)
and turns RLS from a partially-skippable local check into an **active gate**.
Before this subphase, `backend/tests/test_rls_bypass.py` skipped its
detection and helper checks whenever only Alembic had been applied — which is
the default local state — so nothing automated ever proved that the deny-all
RLS baseline is actually enforced and actually bypassed only by the intended
runtime role.

The `rls-active` job closes that gap: it applies Alembic **and** the
Supabase RLS migration tree, runs the SQL RLS tests, and runs the bypass
regression as a genuine non-superuser BYPASSRLS role.

## Jobs

The workflow has three independent jobs:

- **frontend** — `npm ci`, `npm run lint`, `npx tsc -b`, `npm run test`. No
  database, no feature-code changes.
- **backend-unit** — Postgres service + `alembic upgrade head` + the full
  `pytest` suite, connecting as the `postgres` superuser. RLS migrations are
  **not** applied here, so the RLS-aware tests behave exactly as in the local
  "alembic only" state (detection + helper checks SKIP). This job validates
  the application; it is not the RLS gate.
- **rls-active** — the real RLS gate (see below).

## `backend-unit` vs `rls-active`

| | backend-unit | rls-active |
| --- | --- | --- |
| Connects as | `postgres` (superuser) | `nuberush_app` (non-superuser, BYPASSRLS) for the bypass regression |
| Supabase RLS migrations | not applied | applied (vanilla-compatible subset) |
| SQL RLS tests | not run | run (`rls_baseline`, `rls_helpers`, `realtime_orders_inventory`) |
| `RLS_ACTIVE_CI` | unset → SKIPs allowed | `1` → SKIPs become FAILs |
| Scope of pytest | full suite | `tests/test_rls_bypass.py` only |

Keeping them separate is deliberate: the unit suite only needs schema
(Alembic) and can run as the superuser; the RLS gate needs the full Supabase
migration tree applied and must run as a non-superuser role to prove anything.

## Why `nuberush_app` is BYPASSRLS but NOSUPERUSER

`docs/f2.22.3-rls-bypass-role.md` §1 locks the runtime identity: FastAPI is
the **primary** authorization layer (RBAC, tenancy, service-layer guards) and
RLS is **defense-in-depth**, so the runtime role must bypass the deny-all
baseline (`BYPASSRLS`) — but it must **not** be a superuser. A superuser
bypasses RLS unconditionally, which would make the bypass check pass even on a
deploy where the application role had lost `BYPASSRLS`. Running the gate as a
non-superuser role with `BYPASSRLS` is the only configuration that actually
proves the contract. `test_rls_active_ci_runtime_role_is_nonsuperuser_bypass`
asserts `current_user = 'nuberush_app'`, `rolsuper = false`,
`rolbypassrls = true`.

## Why the `auth.uid()` shim exists only in CI

The F2.22.3.E helper migration (`supabase/migrations/20260526144321_rls_helpers.sql`)
is `LANGUAGE sql` and references `auth.uid()` in its body. On a real Supabase
project `auth.uid()` is a platform function; on the bare Postgres used by
GitHub Actions it does not exist, so the migration would fail to compile.
`supabase/ci/bootstrap_rls.sql` creates a CI-only `auth` schema and a small
`auth.uid()` stub that reads the same `app.fake_uid` session GUC the SQL tests
already use. It lives only in the CI bootstrap — never in the migration tree —
mirroring the stub the SQL test files create inside their own
`BEGIN/ROLLBACK` blocks.

## Why storage is not shimmed in F2.27.1

`supabase/migrations/20260527134127_storage_buckets.sql` does
`INSERT INTO storage.buckets`, which requires the Supabase Storage schema.
Storage is infrastructure that has no bearing on the RLS deny-all invariant,
so F2.27.1 does **not** shim a `storage` schema and the `rls-active` job does
**not** apply that migration. `supabase/tests/storage_buckets.sql` already
NOTICE-skips when the `storage` schema is absent, so storage validation is
simply out of scope here and is left to a Supabase-enabled environment.

## No-go boundaries

F2.27.1 does not touch: frontend feature code, `backend/app/`, existing
`supabase/migrations/`, regulatory features, Excel/inventory import,
QuickBooks, Stripe, the Customer App, the Driver App, or any staging/deploy
config. It does not extend RLS to any new (e.g. regulatory/audit) tables —
the gate validates the existing baseline only. No real secrets are created;
the `nuberush_app` CI password is a throwaway for the disposable CI database.
