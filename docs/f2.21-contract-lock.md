# F2.21 Contract Lock — Public Website / Merchant Acquisition Site

Status: contract / scope freeze. Locks the exact surface of F2.21 before
any frontend or backend implementation begins. No code, components,
routes, content modules, or tests are introduced by this document.

This document is the output of subphase **F2.21.0 — Contract Lock /
Scope Freeze**. Implementation subphases (F2.21.1 onward) must match
this contract exactly; any deviation requires updating this document
first.

---

## 1. Purpose

F2.21 converts NubeRush from a minimal single-page landing into a full
professional public website. The phase delivers the public surfaces a
serious operator needs in order to be credible to:

- Prospective merchant stores evaluating the platform.
- Partners, suppliers, and operational contacts.
- Investors, advisors, and other external stakeholders.
- General visitors arriving at the NubeRush domain.

F2.21 builds:

- A public marketing / education website (home + product education).
- A merchant acquisition surface (`/for-stores`).
- A contact / demo intent surface (`/contact`, `/request-demo`).
- A support / FAQ surface (`/support`).
- Legal / trust draft pages under `/legal/*`.
- Public navigation, header, and footer foundation.
- Basic SEO, accessibility, and responsive hardening.

F2.21 does **not** build:

- A backend lead endpoint (`POST /public/leads` or similar).
- A fake contact form with simulated success states.
- A self-serve merchant signup or onboarding flow.
- Customer checkout, cart, or storefront.
- A driver app or driver surface.
- Payments, billing, commission, or invoicing.
- Delivery workflow, dispatch, or routing.
- Admin platform settings (out of scope; remains a placeholder).
- Product media upload / image hosting.
- Reports, exports, or analytics dashboards.
- CRM integration (HubSpot, Salesforce, etc.).
- Ticketing or helpdesk system (Zendesk, Intercom, etc.).
- Final legal approval or counsel-reviewed legal text.
- Fake testimonials, fake customer quotes, or fake case studies.
- Fake partner / press / integration logos.
- Copied competitor copy, structure, or legal language.

Any work that drifts into a non-goal must update this document first.

## 2. Current state

After F2.20, the public surface of the NubeRush web app is minimal:

- `/` exists and renders a real but minimal `LandingPage` (brand mark,
  short hero copy, three feature rows, single "Sign In" CTA).
- `/login` exists (`AuthScreen`). Public registration is disabled at the
  backend; the sign-up toggle is dead code today.
- **No `PublicLayout` exists.** Both `/` and `/login` render hardcoded
  full-screen `<div>` shells with inline color values.
- **No public header, public footer, or shared public navigation
  component exists.**
- The following routes do **not** exist:
  `/for-stores`, `/how-it-works`, `/features`, `/contact`,
  `/request-demo`, `/support`, `/legal`, `/legal/terms`,
  `/legal/privacy`, `/legal/merchant-agreement`,
  `/legal/acceptable-use`, `/legal/cookies`.
- **No backend contact, lead, or demo endpoint exists.** The backend
  `app/api/routes/` namespace contains only internal-operations routers.
- **No tests cover the public surface.** Router tests validate redirect
  logic only; there is no render or navigation test for `LandingPage`.

F2.21 is greenfield work against this state.

## 4. Layout contract

F2.21 introduces a dedicated public layout hierarchy. It must be fully
isolated from the authenticated app shells.

### New layout primitives

- `PublicLayout` — wraps every public route. Provides header, main
  content area, and footer slots.
- `PublicHeader` — public top navigation: logo, product nav links,
  `Sign in` link to `/login`, primary CTA to `/request-demo`.
- `PublicFooter` — global footer: legal cluster, contact email, app
  entry link to `/login`, copyright.
- `PublicSection` — shared section wrapper for vertical rhythm,
  background variants, and container width.
- `PublicPageHeader` — page-level header block (title + lede) used on
  non-home public pages.
- `PublicCtaBand` — reusable CTA strip (used by home and education
  pages) linking to `/request-demo` and `/contact`.

### Layout rules

- Public routes **must not** wrap in `AdminLayout` or `StoreLayout`.
- Public routes **must not** render `ADMIN_NAV_ITEMS` or
  `STORE_NAV_ITEMS`.
- Public routes **must not** require authentication. They render
  identically for unauthenticated and authenticated visitors.
- `/login` **must remain accessible** from every public page (header +
  footer).
- Public CTAs must link honestly. The only allowed CTA destinations
  from public pages are:
  - `/request-demo`
  - `/contact`
  - `/how-it-works`
  - `/features`
  - `/for-stores`
  - `/support`
  - `/legal/*`
  - `/login`
- A public CTA must never link to a route that does not exist by the
  end of F2.21.

## 5. File architecture

F2.21 introduces a single new feature namespace for all public-website
code. Authenticated-app code is not moved.

```
web/src/features/public/
  components/
    PublicHeader.tsx
    PublicFooter.tsx
    PublicHero.tsx
    PublicSection.tsx
    PublicCtaBand.tsx
    PublicFeatureGrid.tsx
    PublicFaq.tsx
    PublicTrustBand.tsx
    PublicPageHeader.tsx
    PublicLegalNotice.tsx
  pages/
    HomePage.tsx
    ForStoresPage.tsx
    HowItWorksPage.tsx
    FeaturesPage.tsx
    ContactPage.tsx
    RequestDemoPage.tsx
    SupportPage.tsx
  legal/
    LegalDocumentPage.tsx
    LegalHubPage.tsx
    TermsPage.tsx
    PrivacyPage.tsx
    MerchantAgreementPage.tsx
    AcceptableUsePage.tsx
    CookiesPage.tsx
  content/
    publicCopy.ts
    legalDrafts.ts
    faqContent.ts
  tests/
    (per-page render + nav tests; structure mirrors pages/ and legal/)
```

Rules:

- Public layout primitives live under `components/` and are exported
  for use by `pages/` and `legal/`.
- All marketing copy strings live in `content/publicCopy.ts` and are
  imported by page components — no copy is inlined in JSX.
- All legal draft text lives in `content/legalDrafts.ts` and is
  consumed by `LegalDocumentPage.tsx` (or per-policy page components).
- FAQ data lives in `content/faqContent.ts`.
- The existing `web/src/pages/LandingPage.tsx` is superseded by
  `features/public/pages/HomePage.tsx` and may be removed in F2.21.2;
  no other current page is touched.

## 6. Home page contract

The `/` route renders `HomePage` and must include, in order:

1. **Hero** — clear positioning, single primary CTA to
   `/request-demo`, secondary link to `/how-it-works`.
2. **Trust / positioning band** — honest, non-fake credibility band
   (e.g., "Built in South Florida for regulated local commerce"). No
   fake logos.
3. **Problem section** — what regulated local-commerce stores
   struggle with.
4. **Solution section** — how NubeRush addresses those problems at
   the operational level (products, inventory, orders, compliance,
   audit).
5. **Feature grid** — three to six features rendered through
   `PublicFeatureGrid`, sourced from `publicCopy.ts`.
6. **How it works preview** — short 3-step preview linking to
   `/how-it-works`.
7. **Built for regulated local commerce** — compliance-aware framing
   without claiming guaranteed compliance.
8. **Merchant operations preview** — short description of the
   operator surface (no screenshots of admin/store UI in F2.21).
9. **Contact / demo CTA** — `PublicCtaBand` to `/request-demo` and
   `/contact`.
10. **FAQ** — three to five entries, sourced from `faqContent.ts`.

Rules:

- **No fake stats.** No "10,000 stores served", "99.9% uptime", "$X
  GMV processed", or any unverifiable number.
- **No fake testimonials.** No invented customer quotes, names, or
  store names.
- **No fake partner logos.** No fabricated press, integration, or
  certification logos.
- **No copied competitor language.** Copy must originate inside this
  repo and reflect NubeRush positioning.
- **No guaranteed compliance claims.** Phrases like "fully compliant"
  or "guaranteed legal" are prohibited.
- **No customer checkout / driver / payment claims** unless those
  features actually ship in the same release.

## 8. Contact / request-demo contract

F2.21 ships `/contact` and `/request-demo` as honest intent surfaces.
No backend lead endpoint exists, and **F2.21 must not introduce one or
simulate one**.

### Locked strategy

- The primary action on both pages is an **honest email CTA** (mailto
  link or visible email address). No HTML form that POSTs to a
  non-existent endpoint, and no client-only form that displays a fake
  success state.
- The business contact email used on both pages is:

  **`team@fuenmayorindustries.com`**

- `/contact` and `/request-demo` may share copy primitives but render
  distinct page intents (general inquiries vs. merchant demo).
- Both pages must clearly state that responses come from the NubeRush
  team and set expectations honestly (no SLA commitments).
- Neither page may render a "Submit" button that does nothing, and
  neither page may show a "Thank you, we'll be in touch" success
  state without a real submission.

### Out of scope (future optional phase)

The following are explicitly deferred and must not be implemented in
F2.21:

- `POST /public/leads`
- `POST /public/contact`
- Email notification pipeline (SES, SendGrid, etc.).
- An admin lead inbox surface under `/app/admin/*`.
- Persistence of leads/contacts in the application database.

A follow-up phase may introduce these. F2.21 does not.

## 9. Legal / trust contract

F2.21 ships five legal pages plus an index, all under `/legal/*`.

### Routes

- `/legal` — hub / index linking to the five policies below.
- `/legal/terms`
- `/legal/privacy`
- `/legal/merchant-agreement`
- `/legal/acceptable-use`
- `/legal/cookies`

### Draft posture

All legal pages in F2.21 are **operational drafts**. They are not
counsel-reviewed and they are not binding. Every legal page must
render the following visible notice via `PublicLegalNotice` at the
top of the document:

> **Draft notice:** This document is provided as an operational draft
> for review and approval by qualified legal counsel before public
> launch.

Legal pages **must not** claim:

- Final legal approval.
- That the page constitutes legal advice.
- Guaranteed compliance with any specific law or jurisdiction.
- Automatic legality verification of merchant products.

### Minimum coverage per policy

**Terms of Service (`/legal/terms`)**

- Website use.
- Account access.
- Platform availability.
- Prohibited use.
- Intellectual property.
- Disclaimers.
- Limitation of liability.
- Contact.

**Privacy Policy (`/legal/privacy`)**

- Information collected.
- Use of information.
- Contact / demo inquiries.
- Cookies.
- Service providers.
- Security.
- User choices.
- Contact.

**Merchant Agreement (`/legal/merchant-agreement`)**

- Merchant responsibilities.
- Product information accuracy.
- Age-restricted product responsibility.
- Compliance responsibility.
- Platform role.
- Operations / order tools.
- Fees governed by a separate agreement.
- Suspension and termination.
- Audit and logs.

**Acceptable Use Policy (`/legal/acceptable-use`)**

- No illegal products.
- No evasion of platform or regulatory controls.
- No fraud.
- No platform abuse.
- No scraping.
- Enforcement.

**Cookie Policy (`/legal/cookies`)**

- What cookies are.
- Essential cookies.
- Analytics cookies (if added later).
- Managing cookies.
- Contact.

### Content sourcing

All legal text lives in `content/legalDrafts.ts`. No copying or
paraphrasing of third-party legal text (competitor sites, template
sites, etc.) is permitted; drafts must originate inside this repo.

## 10. SEO contract

F2.21 ships **basic SPA-level SEO only**. No SSR, no prerendering, no
static-site generation.

Required per public route:

- `document.title` set on mount (unique per route).
- `<meta name="description">` set on mount (unique per route).
- Optional but recommended: `og:title` and `og:description` per route.

Out of scope for F2.21:

- Server-side rendering.
- Prerendering / static export.
- Sitemap generation, `robots.txt` strategy beyond defaults.
- Structured data (JSON-LD).
- Per-route image OG cards.

These may be revisited in a future SEO phase. F2.21 does not commit
to them.

## 11. Accessibility contract

Every public route in F2.21 must satisfy:

- Exactly one `<h1>` per page.
- Semantic landmarks: `<header>`, `<main>`, `<footer>`, `<nav>`.
- `<nav>` elements carry a meaningful `aria-label`.
- Link text is descriptive on its own (no "click here" / "read more"
  without context).
- Any input or interactive control is labeled.
- No clickable `<div>` or `<span>` standing in for a button or link.
- Visible focus states are preserved on all interactive elements.
- Legal pages are readable: comfortable line length, adequate
  contrast, hierarchical headings.
- Mobile navigation is usable with touch and keyboard.

Automated checks beyond render/nav tests (axe, Lighthouse audits)
are not required to pass in F2.21 but should not regress.

---

## 12. Responsive / native contract

- Every public route renders correctly at mobile, tablet, and desktop
  widths.
- F2.21 must not break the Capacitor / native shell build. Existing
  native behavior at `/` and `/login` continues to work.
- Public components must avoid browser-only APIs without guards
  (`window`, `document`, `localStorage`, `navigator`) or wrap them
  defensively.
- F2.21 does not perform a native routing overhaul. Whether public
  routes are exposed inside the native shell is a UX decision; the
  default for F2.21 is that they remain reachable and harmless.

## 13. Testing contract

Every public route added by F2.21 must have a render + navigation
test in `web/src/features/public/tests/`. The suite must cover:

- Each public route renders without throwing.
- `PublicHeader` links resolve to the expected routes
  (`/for-stores`, `/how-it-works`, `/features`, `/support`,
  `/login`, `/request-demo`).
- `PublicFooter` legal cluster links resolve to all five legal pages
  plus `/legal`.
- The header / footer "Sign in" link points to `/login`.
- The primary CTA on home + education pages points to
  `/request-demo`.
- No public page renders admin or store navigation items.
- Each legal page renders its title and the draft notice from
  `PublicLegalNotice`.
- `/contact` and `/request-demo` render an honest email CTA
  (`team@fuenmayorindustries.com`) and **do not** render a form
  submit button or a fake success state.
- The router preserves `/login`, `/app/admin/*`, and `/app/store/*`
  behavior unchanged.

Existing 1625-test baseline must remain green at the end of every
F2.21 subphase. Typecheck must remain clean.
