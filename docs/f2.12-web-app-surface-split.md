**Subphase:** F2.12.0 — Surface Decision & Migration Map.
**Status:** Decisions adopted. Documentation only — no runtime code changes.

**Supersedes:** the F2.12.0 ADR draft of 2026-05-04 morning at the same path

**Related:** [PRD.md](PRD.md), [WEB_APP_VISION.md](WEB_APP_VISION.md),

## 1. Purpose

F2.12 exists to **separate the NubeRush Web App into two distinct,
URL-distinct surfaces**:

```txt
/app/admin/*
/app/store/*
```

Today the Web App is a single shell under `/app/*` shared by every
authenticated user — platform admin and store roles alike. Continuing
to build features inside one shell would compound architectural chaos:
each new admin feature would need conditional rendering, each new store
feature would need to ignore admin context, and the shell itself would
slowly accumulate role-conditional logic that the thin-client contract
forbids.

F2.12 stops that drift before it scales. It establishes Admin App and
Store App as independent surfaces that share primitives but never share
identity.

---

## 2. Product Decision

> **NubeRush Web App is not customer checkout in F2.12.**
> **NubeRush Web App is an operational platform for Admin and Store users.**

The Web App is one of three planned NubeRush surfaces. The other two
(Customer App, Driver App) are explicitly out of scope. F2.12 makes the
operational identity of the Web App permanent.

### 2.1 Admin App

Admin App is for **NubeRush / platform operators** — the founder, the
internal operations team, future support and finance staff.

Responsibilities:

- Global visibility across the platform.
- Store management (onboard, edit, deactivate stores).
- Global users oversight (cross-store user management).
- Global products oversight (cross-store catalog and compliance).
- Global inventory oversight (cross-store stock view).
- Global orders oversight (cross-store order investigation).
- Global audit (cross-resource, cross-store audit trail).
- Global compliance (platform-wide compliance state).
- Operations monitoring (alerts, anomalies, platform health).
- Future billing / commissions (commission per order, payouts,
  invoices).

### 2.2 Store App

Store App is for **stores using NubeRush to operate** — owners,
managers, staff bound to a single store via `user.store_id`.

Responsibilities:

- Store dashboard (single-store operational pulse).
- Product management (catalog and variants for this store).
- Inventory management (receive, adjust, damage, reserve, sell, return).
- Order management (create, transition, cancel, return, audit).
- Store audit (inventory logs and per-resource audit panels).
- Store users (manage staff scoped to this store).
- Store settings (store profile, preferences).
- Daily store operations (the rhythm that ties the above together).

### 2.3 Out of scope in F2.12

The following are explicitly **not** part of F2.12 and **must not** be
introduced as side effects of this work:

- Customer checkout.
- Cart.
- Public marketplace.
- Payment checkout.
- Customer ordering flow.
- Driver App.
- Mobile customer app.
- Fake admin data (mocked KPIs, stub stores list, fabricated audit rows).
- Cross-store aggregation in the frontend (no client-side merging of
  store-scoped responses into a "global view").

## 3. Current State Summary

- Current app uses `/app/*` (see [src/app/router.tsx](nuberushfrontendd/src/app/router.tsx)).
- Current layout is single-surface ([src/layouts/DashboardLayout.tsx](nuberushfrontendd/src/layouts/DashboardLayout.tsx)).
- Current navigation is single-surface ([src/layouts/navigation.ts](nuberushfrontendd/src/layouts/navigation.ts)).
- Admin and store users share the same shell, the same sidebar, and
  the same landing.
- Products, Inventory, and Orders are mature enough to become Store
  App as-is.
- Users (create-only), Audit (hub MVP), and Settings (placeholder)
  are partial or placeholder.
- Admin App has no real surface yet — there is no `/app/admin/*` and
  no admin-specific layout, navigation, or page.
- Admin data is **blocked by missing backend endpoints** (see §9). F2.12
  may build the Admin shell but must not fabricate admin data.

## 4. Target Route Architecture

### 4.1 Target route tree

```txt
/app
  /admin
    /dashboard
    /stores
    /stores/:storeId
    /users
    /products
    /inventory
    /orders
    /audit
    /compliance
    /operations
    /settings

  /store
    /dashboard
    /products
    /products/:productId
    /inventory
    /inventory/:variantId
    /orders
    /orders/new
    /orders/:orderId
    /users
    /audit
    /settings
```

### 4.2 Legacy redirects

Every legacy `/app/<feature>` path must permanently redirect to its
Store-scoped equivalent:

```txt
/app/products              -> /app/store/products
/app/products/:productId   -> /app/store/products/:productId
/app/inventory             -> /app/store/inventory
/app/inventory/:variantId  -> /app/store/inventory/:variantId
/app/orders                -> /app/store/orders
/app/orders/new            -> /app/store/orders/new
/app/orders/:orderId       -> /app/store/orders/:orderId
/app/users                 -> /app/store/users
/app/audit                 -> /app/store/audit
/app/settings              -> /app/store/settings
```

### 4.3 Role-based dynamic redirect

The `/app` index resolves dynamically based on `user.role`:

```txt
/app
  admin                  -> /app/admin
  owner/manager/staff    -> /app/store
```

Driver users are not routed into the Web App. The post-login flow signs
them out with a one-line message ("NubeRush for drivers lives on the
Driver App. The web console is not your surface.") and returns them to
`/login`. No `/app/*` route is mounted for drivers.

## 6. Layout Decision

> **AdminLayout and StoreLayout will be separate surfaces.**
> **They may share shell primitives, but they must not be the same user**
> **experience with different labels.**

A label-only diff (one app, one layout, one nav, one badge that flips
between "admin" and "store") would re-introduce the very coupling F2.12
exists to remove. Each surface owns its own shell composition,
navigation, post-login landing, and topbar copy.

### 6.1 Shared shell primitives (reused, not duplicated)

These primitives stay surface-agnostic and are consumed by both layouts:

- `AppShell` (chrome composition primitive — current
  [`DashboardLayout`](nuberushfrontendd/src/layouts/DashboardLayout.tsx)
  renamed and stripped to slots).
- `AppSidebar` (refactored to receive `items` via props).
- `AppTopbar` (refactored to receive surface-specific scope copy).
- `UserMenu`.
- `EmptyState`, `ErrorState`, `LoadingState`.
- `FeaturePlaceholder`.

### 6.2 Surface-specific configs (separate, never shared)

- `adminNavItems` (in `src/layouts/nav-items-admin.ts`).
- `storeNavItems` (in `src/layouts/nav-items-store.ts`).
- Surface-specific topbar scope copy (Admin: "NubeRush admin · platform";
  Store: "Store: \<store name when available, UUID until then\>").
- Surface-specific landing page (`AdminDashboardPage`, `StoreDashboardPage`).

Implementation is the job of F2.12.1+, not F2.12.0.

## 7. Navigation Decision

> **There will be separate navigation configs:**
> **— `ADMIN_NAV_ITEMS`**
> **— `STORE_NAV_ITEMS`**

A single array with a per-item `scope: "admin" | "store" | "both"` flag
is rejected: it is too easy to misconfigure (one bad flag exposes an
admin link to a staff user), and it encourages cross-surface coupling
inside the nav file itself.

### 7.1 Admin nav (target)

In order:

1. Dashboard
2. Stores
3. Users
4. Products
5. Inventory
6. Orders
7. Audit
8. Compliance
9. Operations
10. Settings

### 7.2 Store nav (target)

In order:

1. Dashboard
2. Products
3. Inventory
4. Orders
5. Users
6. Audit
7. Settings

### 7.3 Authority disclaimer

> **Navigation is UX only.**
> **Frontend navigation must not become the source of truth for authorization.**

The backend enforces every endpoint via `require_*` dependencies and
`resolve_store_scope`. A nav item the user shouldn't see is at worst a
cosmetic regression — never a security one. Nav visibility is not a
permission grant and must not be used as one.

---

## 8. Thin Client Boundary

The frontend is a thin client. F2.12 does not relax this boundary; it
extends it across two surfaces.

### 8.1 The frontend MUST NOT decide

- Roles.
- Permissions.
- Tenant boundaries.
- Stock truth.
- Order totals.
- Compliance truth.
- Valid order transitions.

### 8.2 The frontend SHOULD

- Consume API.
- Render backend state.
- Execute backend-backed actions.
- Show loading / error / success states.
- Invalidate queries correctly after mutations.
- Navigate cleanly between surfaces and pages.
- Show UX gates (nav visibility, role-aware affordances) **without
  treating them as security**.

If a backend endpoint does not exist, the corresponding UI must either
not be built or must explicitly state the limitation. **Never invent
endpoints. Never fabricate data. Never relabel backend errors.**

---

## 9. Admin Backend Blockers

Admin App cannot reach real data without the following backend work.
F2.12 may ship Admin shell + honest placeholders; it must not fake any
of the data behind these endpoints.

- `GET /stores`
- `GET /stores/:storeId`
- `POST /stores`
- `PATCH /stores/:storeId`
- `GET /auth/users`
- `PATCH /auth/users/:userId`
- User deactivate endpoint (e.g. `PATCH /auth/users/:userId/deactivate`)
- Global products query (cross-store, admin-scoped)
- Global inventory query (cross-store, admin-scoped)
- Global orders query (cross-store, admin-scoped)
- Global audit feed (cross-resource, cross-store, with filters)
- Global compliance feed (cross-store compliance state)
- Admin dashboard KPI endpoints (platform-wide aggregates)
- Operations alerts endpoint (anomalies, thresholds, incidents)
- Billing / commissions endpoints (future — entirely new domain)

> **F2.12 may create the Admin Shell, but it must not fake Admin Data.**

Each placeholder Admin page must state the limitation explicitly
("Stores listing requires backend `GET /stores`. Pending."). This is the
project's no-invented-endpoints rule, restated for the Admin surface.

## 10. Migration Strategy

The migration is staged so existing Store users never lose a working
route mid-flight.

1. **Create documentation** (this file). F2.12.0.
2. **Add `/app/admin` and `/app/store` route structure** as new
   greenfield routes, leaving legacy `/app/*` paths intact. F2.12.1.
3. **Add legacy redirects** from each `/app/<feature>` to its
   `/app/store/<feature>` equivalent, plus the role-based `/app`
   redirect. F2.12.1.
4. **Split layouts** — refactor `DashboardLayout` into `AppShell` +
   compose `AdminShell` and `StoreShell` on top. F2.12.1.
5. **Split navigation** — relocate the current `navigation.ts` to
   `nav-items-store.ts`, add `nav-items-admin.ts`, refactor
   `AppSidebar` to receive items via props. F2.12.1.
6. **Harden Store route migration** — once Store relocation lands,
   audit every internal link, every test selector, and every
   doc reference; verify redirects in CI. Likely a dedicated PR after
   F2.12.1.
7. **Add Admin placeholders** — empty pages under `/app/admin/*` that
   each state which backend endpoint they're waiting on. F2.12.4.
8. **Add tests for router/layout/nav split** — see §11. F2.12.1
   (router/redirects) and F2.12.4 (admin placeholders).
9. **Create backend requirements document for Admin App real data** —
   formal requirements for §9's endpoints, written and reviewed before
   any admin feature beyond a placeholder is built. F2.12.5.

## 11. Testing Expectations

These tests must land alongside the implementation subphases. F2.12.0
does not write them; it specifies them.

- `/app` redirects an `admin` user to `/app/admin`.
- `/app` redirects an `owner` / `manager` / `staff` user to `/app/store`.
- Driver login does not mount any `/app/*` route; user is signed out
  and returned to `/login` with the explanation message.
- Each legacy route redirects to its `/app/store/*` equivalent
  (10 redirect tests, one per row in §4.2).
- `/app/admin/*` routes use `AdminShell` (sidebar shows admin items,
  topbar shows platform scope copy).
- `/app/store/*` routes use `StoreShell` (sidebar shows store items,
  topbar shows store scope copy).
- Admin nav renders the 10 items in §7.1 in the specified order.
- Store nav renders the 7 items in §7.2 in the specified order.
- Existing Products / Inventory / Orders pages still load and behave
  correctly under `/app/store/*`.
- Existing component / hook / mutation / invalidation tests continue
  to pass without rewriting business assertions (only route paths and
  test selectors are updated).
- **No tests should expect real admin global data yet.** Admin
  placeholders test only that the placeholder copy is rendered and
  that the page does not call backend endpoints that do not exist.

## 12. Definition of Done for F2.12

F2.12 (the umbrella phase) is complete when **all** of the following
are true:

- `/app/admin/*` exists as the Admin Surface.
- `/app/store/*` exists as the Store Surface.
- Legacy redirects work and are covered by tests.
- Admin uses `AdminShell` (composed via `AppShell`).
- Store uses `StoreShell` (composed via `AppShell`).
- Admin uses `ADMIN_NAV_ITEMS`.
- Store uses `STORE_NAV_ITEMS`.
- Existing Products / Inventory / Orders / Users / Audit pages work
  under Store App without business-logic regressions.
- Admin pages are honest placeholders that state which backend endpoint
  they require.
- No customer checkout, cart, marketplace, or payment UX is introduced.
- No fake admin data is introduced (no mocked KPIs, no stub stores
  list, no fabricated audit rows, no client-side cross-store merge).
- Tests cover routing, layout, and navigation split per §11.
- Admin backend requirements are documented (§10 step 9).

F2.12.0 (this subphase) is complete when this document satisfies the
self-check at the bottom of the report below.
