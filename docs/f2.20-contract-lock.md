# F2.20 Contract Lock — Admin Products + Compliance Oversight

Status: contract / scope freeze. Locks the exact surface of F2.20 before
any backend or frontend implementation begins. No code, models,
migrations, schemas, services, routes, tests, or UI components are
introduced by this document.

This document is the output of subphase **F2.20.0 — Contract Lock /
Scope Freeze**. Implementation subphases (F2.20.1 onward) must match
this contract exactly; any deviation requires updating this document
first.

---

## 1. Purpose

F2.20 converts the remaining admin placeholders for product and
compliance oversight into real, backend-computed admin surfaces:

- `/app/admin/products` → real Admin Products oversight.
- `/app/admin/products/:productId` → real Admin Product detail /
  compliance drill-down.
- `/app/admin/compliance` → real Admin Compliance oversight + queue.

All three surfaces are backed by **admin-only backend endpoints**.
Compliance review/update continues to flow through the existing
canonical endpoint, not a new one. The phase is **read-only at the new
endpoints** and reuses the **existing audited mutation** for any state
change.

F2.20 finalizes the admin oversight surfaces alongside F2.19. After
F2.20, the only remaining placeholder under `/app/admin` is
`/app/admin/settings`, which is **not** in scope here.

---

## 2. Current admin surface context

After F2.19, the following admin surfaces are real (not placeholders):

- `/app/admin` — Admin Dashboard, backed by `GET /admin/dashboard`.
- `/app/admin/operations` — Admin Operations Alerts feed, backed by
  `GET /admin/operations/alerts`.

F2.20 targets the next three admin surfaces:

- `/app/admin/products`
- `/app/admin/products/:productId`
- `/app/admin/compliance`

`/app/admin/settings` remains a placeholder and is **out of scope** for
F2.20.

## 3. Backend endpoints locked

### New endpoints introduced by F2.20

- `GET /admin/products`
- `GET /admin/compliance`
- `GET /admin/compliance/products`

### Existing endpoint reused (not modified, not duplicated)

- `PATCH /products/{product_id}/compliance`

This existing endpoint already owns:

- Audit row creation (`ProductComplianceAuditLog`).
- Quarantine cascade behavior on `InventoryItem` rows when compliance
  state changes block sale.

Therefore:

- **Do not create** `PATCH /admin/compliance/products/{product_id}/review`.
- **Do not create** any duplicate compliance review/update route under
  `/admin/compliance/...` or elsewhere.
- All compliance state changes from the admin Compliance UI must call
  `PATCH /products/{product_id}/compliance` so that audit and
  quarantine semantics remain centralized.

---

## 4. Product data model rule

Locked rules for F2.20 data model usage:

- Product is **global**. There is one canonical row per product across
  the platform.
- Product **does not have `store_id`**.
- Store-specific product availability belongs to **inventory** /
  `InventoryItem`, not to Product.
- `GET /admin/products` **must not accept** a `store_id` query
  parameter, and **must not implement** any `store_id`-based filter in
  F2.20. Admin Products is global product oversight.

Any future "products available in store X" view is an inventory
question, not a Product question, and is out of scope for F2.20.

## 5. `GET /admin/products` contract

### Auth

- `require_admin`.

### Query parameters

- `limit`
- `offset`
- `q`
- `compliance_status`
- `allowed_for_sale`
- `is_active`
- `category`

### Response shape

```text
AdminProductsListResponse = {
  items: ProductRead[]
  total: number
  limit: number
  offset: number
}
```

### Rules

- Admin-only.
- Read-only.
- Paginated (`limit` / `offset`).
- Filterable by the parameters above.
- `total` reflects matches **before** pagination.
- Deterministic ordering (stable across calls for the same filters).
- No fake data, no fixtures, no client-fabricated rows.
- No mutation of any kind.
- No `store_id` filter (see §4).

### Rules (compliance overview)

- Read-only.

- No fake data, no fixtures, no client-side aggregation as source of
  truth.
- No persisted workflow state (no review queue table, no assignment
  table, no SLA table).
- No incident or task entities.

## 7. `GET /admin/compliance/products` contract

### Auth (compliance products)

- `require_admin`.

### Query parameters (compliance products)

- `limit`
- `offset`
- `q`
- `compliance_status`
- `allowed_for_sale`
- `is_active`

### Default queue rule

By default (when no overriding filters narrow the result further),
return products matching the shared compliance blocker predicate:

```text
Product.allowed_for_sale == false
OR Product.compliance_status IN restricted,banned
```

Explicit query filters (`compliance_status`, `allowed_for_sale`,
`is_active`) refine the result; the default queue rule applies when
the caller does not specify those filters.

### Response shape (compliance products)

```text
AdminComplianceProductsListResponse = {
  items: ProductRead[]
  total: number
  limit: number
  offset: number
}
```

### Rules (compliance products)

- Admin-only.
- Read-only.
- Queue is **computed** at request time from Product fields.
- Paginated (`limit` / `offset`).
- Filterable by the parameters above.
- `total` reflects matches **before** pagination.
- No persisted queue rows.
- No assignment, no claim, no ownership semantics.
- No SLA timers.
- No incident workflow.
- No fake data.

---

## 8. Shared compliance blocker predicate

There is exactly **one** definition of "this product is a compliance
blocker" in F2.20:

```text
Product.allowed_for_sale == false
OR Product.compliance_status IN restricted,banned
```

This predicate must be:

- Reused by `GET /admin/compliance` queue counts.
- Reused by `GET /admin/compliance/products` default queue.
- Kept in sync with the equivalent compliance-blocker definitions used
  by the admin dashboard (F2.19 `GET /admin/dashboard`) and admin
  operations alerts (F2.19 `GET /admin/operations/alerts`).

Drift between these definitions is a contract violation. If the
predicate must change, it must change in one place and be reflected in
all admin surfaces that depend on it.

## 11. RBAC matrix

For every new F2.20 endpoint (`GET /admin/products`,
`GET /admin/compliance`, `GET /admin/compliance/products`):

| Caller                           | Expected status |
| -------------------------------- | --------------- |
| Anonymous                        | 401             |
| Invalid token                    | 401             |
| Admin                            | 200             |
| Owner / Manager / Staff / Driver | 403             |

Additional rules:

- No store tenancy guard for these admin endpoints — they are
  platform-global admin views. There is no `store_id` scoping on
  Product (see §4), so there is no store-tenancy assertion to make
  here.
- The **backend remains the security authority**. The frontend must
  not rely on client-side role gating to enforce admin-only access.
- Frontend role-aware navigation/UI is presentation only; every protected
  data fetch must still be authorized by the backend.

## 12. Non-goals / forbidden scope

The following are **explicitly out of scope** for F2.20. Any of these
appearing in an implementation PR for F2.20 is a contract violation.

- No migrations.
- No model changes.
- No product media (no upload, no images, no asset handling).
- No reporting.
- No exports (no CSV, no PDF, no scheduled report).
- No admin settings work (`/app/admin/settings` remains a placeholder).
- No persisted incidents.
- No persisted tasks.
- No new compliance workflow state (no review queue table, no
  assignment table, no SLA, no status machine beyond existing Product
  fields).
- No duplicate compliance review endpoint
  (e.g. **not** `PATCH /admin/compliance/products/{product_id}/review`).
- No `store_id` filter on the Product list.
- No customer app changes.
- No driver app changes.
- No payments work.
- No delivery workflow work.
- No fake data, fixtures, or client-fabricated rows on the new
  surfaces.
- No client-side-generated compliance truth — compliance status, queue
  membership, and audit history are always backend-sourced.

## 14. Success condition

F2.20 is complete **only** when all of the following are true:

- `/app/admin/products` is real (not a placeholder).
- `/app/admin/products/:productId` is real (not a placeholder).
- `/app/admin/compliance` is real (not a placeholder).
- `GET /admin/products` exists and is admin-only.
- `GET /admin/compliance` exists and is admin-only.
- `GET /admin/compliance/products` exists and is admin-only.
- `PATCH /products/{product_id}/compliance` is reused as the canonical
  audited compliance action; no duplicate compliance review/update
  endpoint was introduced.
- No forbidden scope from §12 was added.
- The shared compliance blocker predicate from §8 is the single source
  of truth across F2.20 compliance code, with no drift from the
  equivalent F2.19 admin dashboard / admin operations definitions.

Anything short of all of the above means F2.20 is not closed.
