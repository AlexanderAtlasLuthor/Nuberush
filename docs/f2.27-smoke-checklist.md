# F2.27 Smoke Checklist

> Manual smoke checklist and Playwright e2e harness for validating a deployed
> NubeRush **staging** stack. This is **readiness validation only** — it never
> runs against production and never commits secrets. The e2e harness is
> **env-gated**: with no staging env configured, specs report an explicit
> **BLOCKED skip**, never a fake pass.

## Purpose

F2.27.3 introduces the first formal layer of smoke testing for NubeRush:

- A **manual smoke checklist** (this document) an operator runs against a
  deployed staging stack.
- A **Playwright e2e harness** (`web/playwright.config.ts`, `web/e2e/*.spec.ts`)
  that automates the same critical-path reachability checks.

It validates that staging/deploy readiness (F2.27.2 seed + F2.27.4 deploy
config) actually produces a working app. It does **not** execute against
production, does **not** commit credentials, and does **not** add a CI e2e job
yet — e2e remains operator-run and env-gated until staging is stable.

## Prerequisites

Before any smoke flow can produce a real PASS, all of the following must exist
(see [f2.27-staging-runbook.md](f2.27-staging-runbook.md) and
[f2.27-deploy-runbook.md](f2.27-deploy-runbook.md)):

- [ ] Deployed **backend URL** (Render) with `/health` returning 200.
- [ ] Deployed **frontend URL** (Cloudflare Pages) built against staging.
- [ ] **Supabase staging project** (public URL + anon key in the frontend;
      service-role key server-only).
- [ ] **Alembic migrations** applied (`alembic upgrade head`).
- [ ] **Supabase migrations** applied (RLS baseline, helpers, storage).
- [ ] **F2.27.2 seed** applied (`--target staging`, idempotent).
- [ ] **Supabase Auth backfill** applied (`--apply`) so smoke users can sign in.
- [ ] **Smoke credentials** available to the operator (out of repo).

If any prerequisite is missing, mark the affected rows **BLOCKED** — do not
infer a PASS.

## Environment Variables

The e2e harness reads its target and credentials from the environment. **Use
placeholders only; never commit real values.** These are e2e-runner vars, not
Vite build vars — they are never inlined into the frontend bundle.

Required:

```
E2E_BASE_URL=https://<staging-frontend-origin>
E2E_USER_EMAIL=<store-smoke-user-email>
E2E_USER_PASSWORD=<operator-provided-password>
```

Optional / recommended (admin flows; fall back to `E2E_USER_*` only if those
credentials are valid for an admin account):

```
E2E_ADMIN_EMAIL=<admin-smoke-user-email>
E2E_ADMIN_PASSWORD=<operator-provided-password>
```

Gating rules:

- Missing `E2E_BASE_URL` → all e2e specs **BLOCKED skip**.
- Missing store credentials → store specs **BLOCKED skip**.
- Missing admin credentials (and no usable `E2E_USER_*` fallback) → admin /
  regulatory / product-image specs **BLOCKED skip**.

## Manual Smoke Matrix

Status legend:

- **PASS** — verified working against staging.
- **FAIL** — reachable but assertion failed (real defect).
- **BLOCKED** — cannot run: missing staging backend/frontend, missing Auth
  backfill, or missing smoke credentials.
- **SKIPPED** — explicitly deferred or not applicable in this phase.

| # | Flow | Spec | Status |
| - | ---- | ---- | ------ |
| 1 | Public `/login` reachable (email/password/sign-in controls) | `public-smoke.spec.ts` | ☐ |
| 2 | Admin login → `/app/admin` dashboard + KPI grid | `admin-smoke.spec.ts` | ☐ |
| 3 | Admin regulatory page + decision trail fixture | `regulatory-smoke.spec.ts` | ☐ |
| 4 | Store login → inventory + audit (store-scoped) | `store-smoke.spec.ts` | ☐ |
| 5 | Admin products → product detail → image panel | `product-images-smoke.spec.ts` | ☐ |
| 6 | Store application intake → admin approval | (manual) | ☐ SKIPPED/deferred |
| 7 | Real product image upload (Supabase Storage) | (manual) | ☐ SKIPPED/deferred |

Status rules:

- BLOCKED if staging backend unavailable.
- BLOCKED if staging frontend unavailable.
- BLOCKED if Supabase Auth backfill missing.
- BLOCKED if smoke credentials missing.
- SKIPPED only when explicitly deferred or not applicable.

## Admin Smoke Flows

Requires `E2E_BASE_URL` + admin credentials.

1. Sign in through the UI as `admin@nuberush.dev` (via `E2E_ADMIN_*`).
2. Admin dashboard loads — `data-testid="admin-dashboard-page"`.
3. Admin KPI grid loads — `data-testid="admin-dashboard-kpi-grid"`.
4. Admin regulatory page loads — `data-testid="admin-regulatory-page"`.
5. Regulatory decision trail visible — `data-testid="regulatory-decision-list"`.
6. Admin products page loads — `data-testid="admin-products-page"`.
7. Product detail → image panel visible — `data-testid="admin-product-image-panel"`.

## Store Smoke Flows

Requires `E2E_BASE_URL` + store credentials (`E2E_USER_*`, recommended
`owner@demo-a.nuberush.dev`).

1. Sign in through the UI as the store owner/manager.
2. Store inventory page loads — `data-testid="inventory-total"` (heading
   "Inventory").
3. Store audit page loads — `data-testid="audit-page"` / `audit-feed`.
4. Store-scoped data appears (rows belong to the signed-in store).

## Regulatory Smoke Flows

Requires `E2E_BASE_URL` + admin credentials.

1. Regulatory fixture visible on `/app/admin/regulatory`
   (`data-testid="admin-regulatory-page"`).
2. Source / notice / product-match / compliance-alert / decision audit chain
   visible (`data-testid="regulatory-decision-list"` and its rows).
3. **Advisory-only**: the regulatory surface is informational. This smoke
   asserts visibility of the seeded audit chain — it does **not** exercise or
   change any hold/ban/block behavior.

With env present against seeded staging, the decision trail is a **real
assertion**. If the regulatory fixture is absent while env is present, the spec
**FAILS** — it must not fake a pass.

## Product Image Smoke

Requires `E2E_BASE_URL` + admin credentials.

1. From `/app/admin/products`, open a product detail via the row drilldown
   (`data-testid="admin-products-row-drilldown"`).
2. Product image **metadata panel** visible —
   `data-testid="admin-product-image-panel"`.
3. **No real upload** is performed in F2.27.3.
4. Real Supabase **Storage** upload is **outside** this smoke harness (manual,
   deferred — see below).

## Known Blocked / Deferred Items

- **Store application intake → admin approval**: manual / deferred unless a
  pending-application fixture exists in the seed. Tracked as a manual matrix row.
- **Product image upload (Supabase Storage)**: manual / deferred. The harness
  asserts panel visibility only; it never uploads or calls Storage.
- **CI e2e job**: future / env-gated. **Not** part of F2.27.3. e2e remains
  operator-run until staging is stable; integrating an env-gated CI job is a
  separate follow-up that would touch `.github/workflows/ci.yml`.

## Pass / Fail Criteria

- **PASS**: with env configured against seeded staging, the spec's real browser
  assertions succeed.
- **FAIL**: env configured but a reachability/assertion check fails — a real
  defect to investigate; never overridden to PASS.
- **BLOCKED**: required env missing → explicit skip with a BLOCKED reason; this
  is **not** a PASS and **not** a FAIL.
- **SKIPPED**: explicitly deferred (rows 6–7) or not applicable.
- **No fake pass**: a spec must never claim success merely because env is
  absent. Env-absent ⇒ skip; env-present ⇒ real assertions.

## No-Secret Rules

Prohibited anywhere in docs, specs, or config:

- Hardcoded passwords.
- Supabase **service-role key** in the frontend or in any e2e artifact.
- `DATABASE_URL` in the frontend.
- Real tokens / JWTs in docs or specs.
- Production URLs unless explicitly authorized (this phase targets staging
  only).

Credentials are supplied **only** via `E2E_*` environment variables at runtime
and are never committed.

## Definition of Done

- [ ] `docs/f2.27-smoke-checklist.md` exists with the full manual matrix.
- [ ] `web/playwright.config.ts` exists (chromium, `baseURL` from
      `E2E_BASE_URL`, no mandatory local server).
- [ ] `web/e2e/*.spec.ts` cover login, admin dashboard, regulatory, store
      inventory/audit, product image panel.
- [ ] `npm run e2e` runs Playwright separately from Vitest unit tests.
- [ ] Missing env produces explicit BLOCKED skips (no fake pass).
- [ ] No hardcoded credentials / secrets committed.
- [ ] Frontend lint / typecheck / unit / build still pass.
- [ ] No no-go files touched (CI, backend, render.yaml, supabase, features).
