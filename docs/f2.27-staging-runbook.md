# F2.27 Staging Runbook

Operator guide for standing up a **staging** NubeRush environment: a fresh
Supabase project, migrated schema, validated RLS posture, idempotent demo
seed, and login-capable smoke users — with no real secrets ever committed.

This runbook covers staging only. It does **not** touch production, deploy
configuration (`render.yaml`, Cloudflare), CI, or any application behavior.

---

## Purpose

Give an operator a repeatable, copy-paste sequence to:

1. Provision a Supabase staging project.
2. Configure backend/frontend environment variables (placeholders only).
3. Apply Alembic + Supabase migrations in the correct order.
4. Validate the RLS / storage posture.
5. Seed safe, idempotent demo data.
6. Backfill Supabase Auth identities so the seeded users can sign in.
7. Run a manual smoke checklist.

The goal is a staging environment whose dashboards, inventory, orders and
Admin Regulatory surfaces render real data, reachable by a known set of
smoke users.

---

## Safety Boundaries

- **Never run any step here against production.** The seed tool hard-rejects
  `--target production|prod|live|main-prod`; there is no production target in
  F2.27.2.
- The seed is **idempotent** — a second run (or a run against a partially
  seeded DB) never duplicates data.
- The seed runs in a **single transaction**: `--dry-run` rolls everything
  back; a real run commits once at the end; any error rolls back.
- **No secrets in the repo.** Everything below uses placeholders. Real values
  live only in your shell / the Supabase dashboard / your secret manager.
- The **service-role key is server-only.** It is never given to the frontend
  and never printed by any tool here.
- Product image fixtures are **metadata-only** — no object is uploaded to
  Supabase Storage by the seed (see [Storage Buckets](#storage-buckets)).

---

## Supabase Staging Project Setup

1. Create a **new** Supabase project dedicated to staging (do not reuse a
   production project). Note its project ref.
2. From **Project Settings → Database**, capture the pooled/direct connection
   string. You will use it as `DATABASE_URL`.
3. From **Project Settings → API**, capture:
   - the project URL (`https://<project-ref>.supabase.co`),
   - the **anon** (public) key — frontend-safe,
   - the **service-role** key — server-only, treat as a secret.
4. Keep these in your shell session or secret manager — **never** in the repo.

---

## Database Migration Sequence

Apply migrations in this order:

1. **Alembic** (FastAPI-owned `public` schema): from `backend/`

   ```bash
   DATABASE_URL="postgresql+psycopg://<user>:<password>@<host>:5432/<db>?sslmode=require" \
       alembic upgrade head
   ```

2. **Supabase migrations** (RLS baseline, helpers, storage policies): apply
   everything under `supabase/migrations/` to the staging project, in
   filename (timestamp) order, via the Supabase SQL editor or the Supabase
   CLI. These install row-level-security and storage posture on top of the
   Alembic-created tables.

> Order matters: Alembic creates the tables; the Supabase migrations layer
> RLS/storage policies on top of them.

---

## RLS Validation Sequence

Confirm the RLS baseline is actually active before seeding/exposing the
environment:

1. Check that the expected `public.*` tables report `relrowsecurity = true`:

   ```sql
   select relname, relrowsecurity
   from pg_class
   where relkind = 'r'
     and relnamespace = 'public'::regnamespace
   order by relname;
   ```

2. Confirm the RLS helper functions installed by
   `supabase/migrations/*_rls_helpers.sql` exist.
3. The repo's `tests/test_rls_bypass.py` self-skips unless the RLS baseline is
   active; running it against a connection where RLS is installed exercises
   the bypass checks. (CI runs the `rls-active` job for this — do not modify
   CI here.)

Do not proceed to seeding until RLS is confirmed active on staging.

---

## Storage Buckets

- Create the `product-images` bucket in the staging project if product image
  upload will be smoke-tested manually.
- The **seed does not upload objects** and does not call Supabase Storage. It
  inserts **metadata-only** `product_images` rows with deterministic
  placeholder object keys (e.g. `products/demo/lumen-vape-350mah.jpg`).
- Uploading a real object for a seeded product is a **manual staging
  validation step**, performed through the app's normal product-image upload
  flow — not by this seed.

## Backend Environment Variables

Set these in the backend's server environment (placeholders shown — substitute
real staging values, keep them out of the repo):

```bash
DATABASE_URL=postgresql+psycopg://<user>:<password>@<host>:5432/<db>?sslmode=require
SUPABASE_URL=https://<project-ref>.supabase.co
SUPABASE_SERVICE_ROLE_KEY=<server-only-service-role-key>
SUPABASE_JWKS_URL=https://<project-ref>.supabase.co/auth/v1/.well-known/jwks.json
```

- `SUPABASE_SERVICE_ROLE_KEY` is **server-only** — never expose it to the
  frontend or commit it.
- See `backend/.env.example` for the complete list of backend settings; this
  runbook only highlights what staging bring-up needs.

## Frontend Environment Variables

The frontend receives **public** values only:

```bash
VITE_SUPABASE_URL=https://<project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=<public-anon-key>
```

- The frontend **must never** receive `SUPABASE_SERVICE_ROLE_KEY` or
  `DATABASE_URL`. Those are server-only.
- See `web/.env.example` for the full frontend variable list.

## Seed Sequence

The seed is `backend/scripts/seed_sample_data.py`. It is idempotent and
single-transaction. Run from `backend/`.

1. **Dry-run** (no `--target` needed — write-free preview, rolls back):

   ```bash
   DATABASE_URL="<staging-db-url>" \
       python -m scripts.seed_sample_data --dry-run
   ```

   Review the printed summary (created/existing counts per entity). Nothing
   is committed.

2. **Apply against staging**:

   ```bash
   DATABASE_URL="<staging-db-url>" \
       python -m scripts.seed_sample_data --target staging
   ```

   (For a local dev DB use `--target local`.) A second run is a safe no-op:
   every entity is found-or-created on a stable identity (store code, user
   email, product name, variant SKU, store+variant inventory,
   store+idempotency-key order, notice content-hash, …).

What the seed creates:

- 1 global admin, 2 demo stores, per-store role users (see
  [Smoke Users](#smoke-users)).
- 20 products (mixed compliance statuses), variants, inventory (incl.
  low-stock / depleted), orders across every status, inventory logs, and one
  `created` order-audit row per order.
- Metadata-only product images for a small product subset.
- A minimal **manual** regulatory fixture: source → notice → product match →
  compliance alert → decision audit. Advisory only — it never holds, bans or
  blocks a product, and mutates no product state.

**Deferred fixtures** (documented, intentionally not seeded in F2.27.2):

- `operational_audit_logs` — deferred; would require driving the operational
  audit service, out of scope for this phase.
- `product_compliance_audit_logs` — deferred; these rows represent real
  compliance *transitions*, and seeding raw rows would misrepresent that
  audit invariant. Generate them through the normal admin compliance flow if
  needed for a deeper smoke.

## Supabase Auth Backfill Sequence

Seeded users are `public.users` app records only (`auth_user_id` is NULL) and
**cannot sign in** until a Supabase identity is provisioned and linked. Use
`backend/scripts/backfill_supabase_auth_users.py`. It is **dry-run by
default**; `--apply` is required for writes.

1. **Dry-run** (no Supabase API call, no writes):

   ```bash
   DATABASE_URL="<staging-db-url>" \
       python -m scripts.backfill_supabase_auth_users
   ```

2. **Apply** (only with server-only service-role env vars set, and an
   operator-provided temporary password):

   ```bash
   DATABASE_URL="<staging-db-url>" \
   SUPABASE_URL="https://<project-ref>.supabase.co" \
   SUPABASE_SERVICE_ROLE_KEY="<server-only-service-role-key>" \
   NUBERUSH_BACKFILL_PASSWORD="<operator-provided-temporary-password>" \
       python -m scripts.backfill_supabase_auth_users --apply
   ```

   To bootstrap just the admin first:

   ```bash
   DATABASE_URL="<staging-db-url>" \
   SUPABASE_URL="https://<project-ref>.supabase.co" \
   SUPABASE_SERVICE_ROLE_KEY="<server-only-service-role-key>" \
       python -m scripts.backfill_supabase_auth_users \
       --apply --email admin@nuberush.dev --password "<operator-provided-temporary-password>"
   ```

3. **Post-apply verification** (dry-run again — should report every user
   already mapped):

   ```bash
   DATABASE_URL="<staging-db-url>" \
       python -m scripts.backfill_supabase_auth_users
   ```

Backfill safety contract:

- **dry-run by default**; `--apply` required for writes;
- `NUBERUSH_BACKFILL_PASSWORD` is **operator-provided** (or `--password`);
- `SUPABASE_SERVICE_ROLE_KEY` is **server-only**;
- **no passwords are committed** anywhere;
- the temporary password and service-role key are **never printed**;
- the **frontend must never receive the service-role key**;
- backfilled users should reset their password via Supabase's reset flow.

## Manual Smoke Checklist

1. Sign in as `admin@nuberush.dev` (after backfill) and confirm the admin
   dashboard renders.
2. Confirm the **Admin Regulatory** surface shows the seeded source, notice,
   product match, compliance alert and the decision-trail row.
3. Sign in as `owner@demo-a.nuberush.dev`; confirm the store dashboard,
   low-stock widget and aging-order alert show real rows.
4. Confirm inventory and orders lists render across statuses for both stores.
5. (Optional) Upload a real object for a seeded product through the normal
   product-image flow and confirm it renders (manual storage validation).
6. Confirm a `staff` / `driver` account's access is scoped as expected.

---

## Rollback / Cleanup

- The seed never deletes data; to start clean on a disposable staging DB,
  `alembic downgrade base && alembic upgrade head`, then re-seed.
- For Supabase Auth identities created by the backfill, remove them via the
  Supabase dashboard / Admin API if you need a clean auth state.
- Because the seed is idempotent, re-running after a partial failure simply
  completes the missing pieces.

---

## No-Secret Rules

- Every value in this runbook is a **placeholder**:
  - `SUPABASE_URL=https://<project-ref>.supabase.co`
  - `SUPABASE_SERVICE_ROLE_KEY=<server-only-service-role-key>`
  - `DATABASE_URL=postgresql+psycopg://<user>:<password>@<host>:5432/<db>?sslmode=require`
  - `VITE_SUPABASE_ANON_KEY=<public-anon-key>`
  - `NUBERUSH_BACKFILL_PASSWORD=<operator-provided-temporary-password>`
- No real credentials, private keys, or passwords are committed.
- The service-role key and the backfill temporary password are server-only
  and are never printed by the tooling.
- The frontend gets only the public URL + anon key.

- [ ] No secrets committed; service-role key never exposed to the frontend.
