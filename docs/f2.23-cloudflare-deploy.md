# F2.23 — Cloudflare Pages Deployment Guide (NubeRush Frontend)

## 1. Objective

This guide documents how to configure **Cloudflare Pages** to host the NubeRush
**Vite/React frontend only**. It covers project settings, environment variables,
SPA routing, headers, backend CORS alignment, deployment steps, verification, and
rollback.

It does **not** change any runtime behavior, and it does **not** introduce any
secrets. It assumes F2.23.D is closed and passed, so the repo already contains the
static hosting artifacts:

- `web/public/_redirects`
- `web/public/_headers`

## 2. Architecture boundary

This deployment preserves the F2.22 hybrid boundary. Each component keeps a single
clear responsibility:

- **Cloudflare Pages** = static frontend hosting **only** (HTML/CSS/JS assets).
- **FastAPI** = business authority, separately hosted. All business reads/writes go
  through it.
- **Supabase** = Auth, Postgres, Storage, Realtime, and Row-Level Security (RLS),
  separately hosted.
- **Frontend** = thin client. It renders UI and orchestrates calls; it holds no
  business authority.

Data-flow rules:

- Business reads/writes go through `apiRequest()` → FastAPI.
- `supabase-js` in the frontend is used **only** for:
  - auth/session,
  - storage signed upload,
  - realtime invalidation signals.

Explicit prohibitions for this boundary:

- **No Cloudflare Workers business logic.**
- **No Supabase RPC business logic.**
- **No Edge Functions.**

Cloudflare Pages serves static files. It does not host, replace, or proxy FastAPI,
and it does not run server-side business code.

## 3. Cloudflare Pages project settings

Configure the Pages project with exactly these settings:

| Setting | Value |
| --- | --- |
| Root directory | `web` |
| Build command | `npm run build` |
| Build output directory | `dist` |
| Framework preset | `Vite` (or `None` with the explicit build command above) |
| Production branch | `main` |
| Node version | `20` (20 LTS) |
| Preview deployments | Enabled **only** with an explicit env strategy (see §5) |

Notes:

- The build output directory `dist` is relative to the root directory `web`, i.e.
  `web/dist`.
- Node `20` should be pinned in the Pages project (and, if desired, via a
  `NODE_VERSION` build variable) to keep builds reproducible.

## 4. Production environment variables

Set these in the Cloudflare Pages **Production** environment. Use **placeholders
only** — never commit real values to the repo or paste real values into this doc.

```
VITE_API_BASE_URL=https://<fastapi-production-origin>
VITE_SUPABASE_URL=https://<project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=<public-anon-key>
```

These three are the only frontend-facing variables. `VITE_SUPABASE_ANON_KEY` is the
**public anon key** and is safe for browser exposure; it is not a secret and is not
the service-role key.

`VITE_API_BASE_URL` is **required** for production builds. The frontend fails fast
(F2.23.F1): if it is missing or blank when the production bundle loads, the app
throws `VITE_API_BASE_URL is required for production builds.` instead of silently
calling `http://localhost:8000`. The localhost fallback is **development-only**.
Cloudflare Pages production env must therefore set `VITE_API_BASE_URL`.

## 5. Preview environment variables

Cloudflare Pages preview deployments must use a deliberate strategy. Two acceptable
options:

- **Strategy A — Previews point to staging backend/Supabase.**
  Configure the Preview environment with `VITE_API_BASE_URL`, `VITE_SUPABASE_URL`,
  and `VITE_SUPABASE_ANON_KEY` that target **staging** infrastructure. Requires a
  staging FastAPI origin and matching staging CORS configuration.

- **Strategy B — Previews are public-only.**
  Previews render only public, unauthenticated surfaces until a staging backend and
  CORS are configured. Authenticated flows are expected to be non-functional in
  previews under this strategy.

Important: previews must **not** accidentally use the production environment unless
that is intentionally approved. Production env values should never be the implicit
default for preview deployments.

## 6. Forbidden Cloudflare Pages environment variables

The following must **never** be added to Cloudflare Pages (any environment). They
are backend/server-only or private and have no place in a browser-delivered bundle:

- `SUPABASE_SERVICE_ROLE_KEY`
- `DATABASE_URL`
- `SUPABASE_JWKS_URL`
- `SUPABASE_JWT_AUDIENCE` (if backend-only)
- Supabase pooler credentials
- Backend secrets
- Cloudflare API tokens
- Any private credentials

Anything that grants privileged data access or bypasses RLS belongs only on the
separately hosted FastAPI/Supabase backend, never in the frontend host.

## 7. SPA routing

The frontend router uses `createBrowserRouter` with HTML5 history mode. Static
hosts return 404 for unknown deep paths unless a SPA fallback is configured.

- `web/public/_redirects` is **required**.
- Required rule:

```
/* /index.html 200
```

This serves `index.html` with a `200` status for any path so the client router can
resolve it. It preserves direct refresh and deep links for routes including:

- `/login`
- `/for-stores`
- `/legal/*`
- `/app/admin/*`
- `/app/store/*`
- dynamic/parameterized routes

## 8. Headers

`web/public/_headers` provides a baseline of safe response headers. The current
baseline includes:

- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `X-Frame-Options: DENY`
- A conservative `Permissions-Policy`
  (`camera=(), microphone=(), geolocation=(), payment=(), usb=(), interest-cohort=()`)
- Immutable long-lived caching for `/assets/*`
  (`Cache-Control: public, max-age=31536000, immutable`)

**CSP is intentionally deferred** until production domains (frontend origin,
FastAPI origin, Supabase origin) are finalized. A Content-Security-Policy will be
added in a later hardening phase once those origins are known.

## 9. Backend CORS / API alignment

Cloudflare Pages does not host or replace FastAPI. For the authenticated app to be
production-ready, the backend and frontend origins must be aligned:

- FastAPI must have a **production HTTPS origin** before the authenticated app can
  be considered production-ready.
- `VITE_API_BASE_URL` must point to that exact FastAPI production origin.
- `BACKEND_CORS_ORIGINS` (on FastAPI) must include the **exact** Cloudflare Pages
  and/or custom frontend origins.
- **No wildcard CORS in production.** Origins must be enumerated explicitly.
- Cloudflare Pages does not host or replace FastAPI; it only serves the static
  client that calls FastAPI.

## 10. Deployment steps

Dashboard-first deployment:

1. Create a Cloudflare Pages project.
2. Connect the GitHub repository.
3. Select production branch `main`.
4. Set root directory `web`.
5. Set build command `npm run build`.
6. Set build output directory `dist`.
7. Set Node version `20`.
8. Add the production environment variables (see §4).
9. Deploy.
10. Verify the `*.pages.dev` URL serves the app.

## 11. Verification checklist

After deployment, confirm:

- [ ] Home (`/`) loads.
- [ ] `/for-stores` loads.
- [ ] `/legal/terms` loads.
- [ ] `/login` loads.
- [ ] Hard refresh on a deep route works (SPA fallback returns the app, not a 404).
- [ ] No `localhost` API calls appear in the browser Network tab.
- [ ] Supabase config resolves without console env errors.
- [ ] API calls go to `VITE_API_BASE_URL`.
- [ ] No CORS errors once the backend is aligned (see §9).
- [ ] No service-role key or `DATABASE_URL` is present in the browser bundle or
      network traffic.

## 12. Rollback

- **Fastest:** use the Cloudflare dashboard to roll back to a previous successful
  deployment.
- **Repo-level:** revert the offending commit if a repo artifact or docs change
  introduced the problem, then redeploy.
- **Do not rotate secrets** unless there is actual exposure. Rollback is a
  deployment action; secret rotation is a separate incident-response action.

## 13. Non-goals

The following are explicitly out of scope for this phase:

- Deploying FastAPI to Cloudflare Workers.
- Moving business logic to Cloudflare Workers.
- Edge Functions.
- Supabase RPC business logic.
- Payments.
- Checkout.
- Driver app.
- Email/webhooks.
- Replacing FastAPI hosting.
- Production CSP hardening before domains are finalized.

## 14. Open decisions

The following remain to be decided:

- FastAPI production HTTPS origin.
- Custom frontend domain.
- Preview env strategy (Strategy A vs Strategy B from §5).
- Whether the deploy URL should live only in the report or also in docs.

A production fail-fast guard for a missing/blank `VITE_API_BASE_URL` is no longer an
open decision — it was implemented in F2.23.F1 (see §4).
