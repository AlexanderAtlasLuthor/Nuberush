# F2.22.3 — DB bypass role contract (`nuberush_app`)

This document locks the database role FastAPI will use as its runtime
identity once F2.22.3.C+ enables a deny-all RLS baseline on every
`public.*` table. The role is named `nuberush_app` and carries
`BYPASSRLS` so the existing pytest suite (1757 tests) and every
FastAPI endpoint continue to behave exactly as they do today.

This subphase (F2.22.3.C) only **documents** the contract. RLS is not
enabled, no policies exist yet, no helper functions exist yet. The
role itself is provisioned outside the migration tree — see
[§3 Provisioning](#3-provisioning).

---

## 1. Role identity

| Property | Value | Rationale |
|---|---|---|
| Name | `nuberush_app` | Distinct from `postgres`, `anon`, `authenticated`, `service_role` so audits and logs can attribute traffic to FastAPI unambiguously. |
| Attribute: `LOGIN` | yes | The role is a runtime login identity used by FastAPI's Postgres driver. |
| Attribute: `BYPASSRLS` | yes | FastAPI is the **primary** authorization layer (RBAC, tenancy, service-layer guards). RLS is **defense-in-depth**. The runtime role must bypass RLS so the deny-all baseline does not break business reads/writes. See [`f2.22-contract-lock.md`](f2.22-contract-lock.md) §7. |
| Attribute: `SUPERUSER` | no | The role only needs schema/table privileges, never cluster-level powers. Keep blast radius minimal. |
| Attribute: `CREATEROLE` / `CREATEDB` / `REPLICATION` | no | Same reasoning — least privilege. |
| Password | managed outside the repo | Stored in the deployment secret manager (Supabase project settings, GitHub Actions secrets, local `.env`). Never committed. |
| Default schema | `public` | All application objects live in `public.*`. |

### 1.1. What `nuberush_app` is NOT

- It is **not** `SUPABASE_SERVICE_ROLE_KEY`. The service-role key is an
  HTTP API key for the Supabase Admin API (used by
  [`backend/app/services/supabase_admin.py`](../backend/app/services/supabase_admin.py)
  to create `auth.users` rows). It never appears in `DATABASE_URL` and
  has no Postgres role attached. Confusing the two is a category
  error.
- It is **not** the Supabase-managed `service_role` Postgres role
  (which also bypasses RLS by default). We do not reuse it because it
  is the same identity Supabase's PostgREST / Realtime stack would use
  if we ever turned those on — keeping FastAPI on a distinct role
  means we can later restrict `service_role` independently without
  breaking FastAPI.
- It is **not** the `postgres` superuser. Superuser bypasses RLS by
  default but also bypasses every other safety check Postgres has.
  Running production FastAPI as `postgres` would leak that blast
  radius into the application identity.

## 2. Required privileges

`nuberush_app` needs the minimum grants required for FastAPI to read
and write every existing `public.*` application table:

```text
GRANT USAGE ON SCHEMA public TO nuberush_app;
GRANT SELECT, INSERT, UPDATE, DELETE
  ON ALL TABLES IN SCHEMA public TO nuberush_app;
GRANT USAGE, SELECT, UPDATE
  ON ALL SEQUENCES IN SCHEMA public TO nuberush_app;
```

To keep future Alembic migrations from silently revoking access on
newly created objects, default privileges must also be granted so any
table or sequence the migration tree creates later automatically
extends to `nuberush_app`:

```text
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO nuberush_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO nuberush_app;
```

`ALTER DEFAULT PRIVILEGES` is scoped to the role that **runs** the
DDL. The grants above must therefore be re-issued for each role that
will later create tables in `public` — in practice that means running
them as the same identity Alembic uses (typically the Supabase
`postgres` superuser). If Alembic ever switches its migration role,
this contract needs an update.

### 2.1. Privileges `nuberush_app` does NOT need

- No `EXECUTE` on functions in `public`. The application reads/writes
  through SQLAlchemy and FastAPI services; no `SELECT routine(...)`
  RPC pattern is in use.
- No `CREATE` on schema `public`. DDL is owned by Alembic running as
  the schema owner, not by FastAPI.
- No privileges on `auth.*`, `storage.*`, or `realtime.*` schemas.
  Those are Supabase-internal and accessed via Supabase APIs, not via
  the FastAPI Postgres connection.

## 3. Provisioning

### 3.1. SQL template (placeholders only)

```sql
-- F2.22.3 — nuberush_app bypass role provisioning.
-- Run ONCE per environment, BEFORE the first Supabase migration
-- that enables RLS lands (F2.22.3.C+).
--
-- Replace <CHANGE_ME> with a strong, generated password stored in
-- the environment's secret manager. The placeholder below is
-- intentionally invalid syntax-as-a-secret (angle brackets) so a
-- copy-paste mistake fails loudly instead of silently creating an
-- account with the literal string.

CREATE ROLE nuberush_app
  WITH LOGIN
       BYPASSRLS
       NOSUPERUSER
       NOCREATEDB
       NOCREATEROLE
       NOREPLICATION
       PASSWORD '<CHANGE_ME>';

GRANT USAGE ON SCHEMA public TO nuberush_app;

GRANT SELECT, INSERT, UPDATE, DELETE
  ON ALL TABLES IN SCHEMA public TO nuberush_app;

GRANT USAGE, SELECT, UPDATE
  ON ALL SEQUENCES IN SCHEMA public TO nuberush_app;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO nuberush_app;

ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT USAGE, SELECT, UPDATE ON SEQUENCES TO nuberush_app;
```

The template contains **no real password, no real Supabase project
URL, no real host, no real port**. It is safe to commit because every
secret-bearing field is a literal placeholder.

### 3.2. Why the SQL is NOT a Supabase migration file

`CREATE ROLE` and `ALTER ROLE ... BYPASSRLS` are cluster-level
operations:

- They require Postgres superuser (or membership in a role with
  `CREATEROLE` plus the ability to set `BYPASSRLS`, which Supabase
  typically restricts to the `postgres` user).
- Supabase migration runners may operate as a non-superuser role —
  if the migration tree contained `CREATE ROLE`, the migration would
  fail mid-apply on a fresh environment with no clean recovery path.
- The role must exist **before** any migration that enables RLS, so
  that FastAPI's `DATABASE_URL` can be repointed at `nuberush_app` in
  the same change window. Coupling role creation with RLS enable
  inside one migration file makes that ordering impossible.

The role is therefore provisioned as a **one-off bootstrap**, not as a
versioned migration. The accepted execution paths are:

1. **Supabase Dashboard → SQL Editor** (recommended for the
   Supabase-hosted project). Paste the template, replace
   `<CHANGE_ME>` with a generated password, run as `postgres`.
2. **`psql` with the project `postgres` connection string**. Same SQL,
   same caveat about replacing the placeholder.
3. **Local dev (Docker / native Postgres 16)**. `psql -U postgres`
   against the local cluster, then update local `backend/.env` to
   match.

After the role exists, F2.22.3.D+ migrations can land without
touching cluster-level state.

### 3.3. Fallback if the role cannot be provisioned

If `nuberush_app` is not configured **before** the F2.22.3.C+ deny-all
RLS baseline migration applies, FastAPI will hit permission denied on
every business query as soon as the deny-all migration runs:
SQLAlchemy queries connect as whatever role `DATABASE_URL` points at,
and that role will not bypass RLS.

Mitigation:

- Do not merge a deny-all RLS migration until the role is provisioned
  on every environment that migration will reach.
- The F2.22.3 phase **must not close** until each target environment
  reports `nuberush_app` exists with the expected attributes.
- If a fallback is genuinely needed mid-rollout, the deny-all
  migration can be reverted (`ALTER TABLE ... DISABLE ROW LEVEL
  SECURITY` per table) without losing schema state. See
  [`f2.22-contract-lock.md`](f2.22-contract-lock.md) §14 (Rollback
  Plan, F2.22.3 entry).

### 3.4. Verifying the role after provisioning

Run these read-only checks as `postgres` against the target database:

```sql
SELECT rolname, rolbypassrls, rolcanlogin, rolsuper
  FROM pg_roles
 WHERE rolname = 'nuberush_app';
-- Expected: rolbypassrls = t, rolcanlogin = t, rolsuper = f.

SELECT has_schema_privilege('nuberush_app', 'public', 'USAGE') AS schema_usage;
-- Expected: t.

SELECT count(*) FILTER (
         WHERE has_table_privilege('nuberush_app', oid, 'SELECT, INSERT, UPDATE, DELETE')
       ) AS granted_tables,
       count(*) AS total_tables
  FROM pg_class
 WHERE relnamespace = 'public'::regnamespace
   AND relkind = 'r';
-- Expected: granted_tables = total_tables (every public table).
```

## 4. FastAPI `DATABASE_URL` requirement

Once RLS is enabled on any environment, FastAPI's connection string
**must** use `nuberush_app`:

```text
postgresql+psycopg://nuberush_app:<password>@<supavisor-session-pooler-host>:<port>/<database>?sslmode=require
```

- **Driver prefix**: `postgresql+psycopg://` — keeps SQLAlchemy on
  psycopg 3 to preserve the prepared-statement and `SELECT ... FOR
  UPDATE` semantics the inventory/orders concurrency tests rely on.
- **Pooler**: Supavisor **session** pooler. The contract bans the
  transaction pooler ([`f2.22-contract-lock.md`](f2.22-contract-lock.md)
  §6) because it breaks SQLAlchemy session state, psycopg prepared
  statements, and `SELECT ... FOR UPDATE` row locks.
- **TLS**: `?sslmode=require` is mandatory in any non-local
  environment.

Forbidden runtime identities for FastAPI's `DATABASE_URL`:

| Role | Why forbidden |
|---|---|
| `anon` | Designed for unauthenticated PostgREST traffic; subject to RLS; would break every business query under deny-all. |
| `authenticated` | Designed for end-user JWT traffic via PostgREST; subject to RLS; same failure mode as `anon`. |
| `service_role` (Supabase) | Bypasses RLS but is the identity Supabase's own PostgREST/Realtime stack uses. Keeping FastAPI on a distinct role lets us restrict `service_role` later without breaking FastAPI. |
| `postgres` (superuser) | Bypasses RLS, but bypasses every other safety check too; oversized blast radius for a long-lived application identity. Acceptable for one-off DDL only. |

Local development before RLS is enabled may continue using
`postgres:postgres` against a local Docker Postgres — the existing
`backend/.env.example` default is unchanged in F2.22.3.C. The
`nuberush_app` switch lands in the same change window as the
F2.22.3.D+ deny-all migration on each environment.

## 5. Lifecycle and ongoing operations

- **Password rotation**: rotated via the platform secret manager.
  Generate a new password, `ALTER ROLE nuberush_app WITH PASSWORD
  '<NEW>'` as `postgres`, update the `DATABASE_URL` secret, restart
  FastAPI. Brief connection churn is acceptable; the inventory and
  orders flows recover by retrying their transactions.
- **New tables / sequences**: covered by the `ALTER DEFAULT
  PRIVILEGES` grants above, provided Alembic runs as the same role
  that issued those grants. If a future migration manually creates a
  table under a different role, that migration must add an explicit
  `GRANT ... TO nuberush_app` line. F2.22.6 boundary checks should
  flag missing grants.
- **Removing the role**: only if F2.22.3 is rolled back per
  [`f2.22-contract-lock.md`](f2.22-contract-lock.md) §14. Drop in the
  reverse order: `ALTER TABLE ... DISABLE ROW LEVEL SECURITY` first,
  repoint `DATABASE_URL` back to the previous identity, then
  `DROP OWNED BY nuberush_app; DROP ROLE nuberush_app;` as
  `postgres`.

### 5.1. Emergency disable (rollback) SQL

If a misconfigured deploy lands deny-all RLS in an environment whose
FastAPI runtime role does not bypass it, the application will hit
`permission denied` on every business query. The fastest safe
mitigation — pending a proper fix to `DATABASE_URL` or to the role's
`BYPASSRLS` attribute — is to disable RLS on the affected tables. Run
the block below as `postgres` (Supabase Dashboard SQL Editor or
`psql -U postgres`). It is identical to the rollback notes embedded in
[`../supabase/migrations/20260526142048_rls_baseline.sql`](../supabase/migrations/20260526142048_rls_baseline.sql):

```sql
ALTER TABLE public.users                          DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.stores                         DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.products                       DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.product_variants               DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.inventory_items                DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.inventory_logs                 DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.orders                         DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.order_items                    DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.order_audit_logs               DISABLE ROW LEVEL SECURITY;
ALTER TABLE public.product_compliance_audit_logs  DISABLE ROW LEVEL SECURITY;
```

The helper functions
(`public.current_app_user_id` / `public.current_app_user_store_id` /
`public.is_admin`) are dormant without RLS — no policy invokes them
in F2.22.3 — so they may be left in place during an emergency. Drop
them only as part of a planned full F2.22.3 rollback:

```sql
DROP FUNCTION IF EXISTS public.current_app_user_id();
DROP FUNCTION IF EXISTS public.current_app_user_store_id();
DROP FUNCTION IF EXISTS public.is_admin();
```

After the emergency disable, re-enabling RLS is a re-apply of
`supabase/migrations/20260526142048_rls_baseline.sql` (the file is
forward-only and the ALTER TABLE statements are idempotent in
practice). Resolve the root cause — usually `DATABASE_URL` pointing
at a non-bypass role — **before** re-enabling.

## 6. Helper-function forward dependency (informational)

The RLS helpers that F2.22.3.D+ will create — `current_app_user_id()`,
`current_app_user_store_id()`, `is_admin()` — will be defined as
`SECURITY DEFINER` owned by `postgres`, **not** by `nuberush_app`.
They are invoked by RLS policies as the `authenticated` role, so they
do not need to be reachable by `nuberush_app` directly. This is noted
here only so the helper-function migration in F2.22.3.D does not have
to revisit the role-ownership question.

No helper functions are created in F2.22.3.C.

## 8. Runtime validation (F2.22.3.G)

[`backend/tests/test_rls_bypass.py`](../backend/tests/test_rls_bypass.py)
is the regression net for this contract. Run it after applying the
Supabase migration tree to confirm the runtime role honors the bypass
strategy:

```bash
cd backend
python -m pytest tests/test_rls_bypass.py -q
```

Interpreting the results in different environments:

| Environment | Expected outcome |
| --- | --- |
| Fresh local checkout — only Alembic applied | `test_db_role_bypasses_rls_when_baseline_is_applied` SKIPs ("baseline not active"); `test_fastapi_request_succeeds_under_actively_enforced_rls` PASSes (it force-enables RLS within its own transaction); `test_rls_helpers_present_when_helpers_migration_is_applied` SKIPs. |
| Local checkout with `supabase/migrations/*.sql` also applied | All three tests PASS — the detection check actively verifies that `current_user` bypasses RLS, the active regression re-confirms via GET `/auth/me`, and the helper-existence check passes. |
| RLS-active environment with `DATABASE_URL` pointing at a role that does NOT have BYPASSRLS | Both bypass-related tests FAIL with explicit messages naming the role and pointing back to this document. That is the broken-deploy condition the regression is designed to catch. |

The test exercises the FastAPI → SQLAlchemy path only; it does not
touch supabase-js, storage, or realtime. RLS remains defense-in-depth
in the F2.22 hybrid contract (§7).
