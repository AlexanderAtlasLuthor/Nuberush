# Inventory F2.6 Readiness — Gaps & Decisions

Status: pre-implementation note. Locks scope before pagination work and the
F2.6 frontend subphase.

---

## 1. Current State

- Inventory endpoints (11 total) are stable and covered by 176/176 targeted
  tests (`test_inventory_schemas.py` 71, `test_inventory_api.py` 74,
  `test_inventory_services.py` 31).
- `InventoryItemRead` is enriched (BIE.1 → BIE.4): every response carries
  a curated `variant.product` summary, so the frontend can render rows
  without follow-up calls.
- Eager loading via `selectinload(InventoryItem.variant).selectinload(
  ProductVariant.product)` is wired into `_lock_inventory_item`,
  `get_inventory_item`, and `list_inventory_for_store`. Anti-N+1 regression
  test in place.
- Auth (`get_current_user` + role gates) and tenancy
  (`require_store_member` on store-scoped paths, `_assert_can_access_store`
  on item-scoped paths) are enforced; admin bypasses tenancy.
- Frontend inventory: only `InventoryPlaceholderPage` exists. No
  `inventory.api.ts`, no hooks, no list/detail/movement components.

## 2. Gap #1 — Admin Store Context

### Problem

- Backend invariant: `admin.store_id` MUST be NULL (global, no bound
  store). Source of truth: `app/db/models.py` `User` docstring +
  `app/api/deps.py::require_store_member`.
- Every store-scoped inventory endpoint resolves a `store_id` from the
  URL and requires the caller to be a member of that store (or admin).
- Admin can target *any* `store_id` per request, but has no
  per-session "current store" anywhere in the system.
- Backend exposes no `/stores` listing endpoint. The frontend has no
  way to enumerate stores admin could target. Verified: `app/main.py`
  registers `auth`, `products`, `variants`, `inventory`, `orders`. No
  `stores` router.

### Impact

- Admin cannot use any store-scoped inventory page in the frontend.
  The store-context indicator already shows `Global admin scope` for
  this reason (F2.4).
- This is a backend surface gap, not a frontend bug. The frontend has
  no honest data source to populate a store picker.

### Options considered

- **Fake selector populated client-side** — rejected. Re-introduces
  the Lovable visual-prototype anti-pattern (frontend invents authority).
- **Manual UUID input field** — rejected. Unusable UX, bypasses
  any future store activity/auth surface, no validation against the
  list of "stores I can access".
- **Proper `/stores` listing endpoint + admin store-picker UX** —
  correct long-term, but out of scope for F2.6. Touches a new router,
  new schema, and a new admin-only auth gate; deserves its own subphase.

### Decision

```
Defer admin support for Inventory v1.
```

### Scope implication

- Inventory v1 ships for non-admin operational roles only:
  - `owner`
  - `manager`
  - `staff`
- `driver` remains intentionally excluded per
  `domain/inventory_rules.py §2`.
- Admin attempting to reach `/app/inventory` will see the existing
  `Global admin scope` topbar copy and an explicit, non-blocking
  message in the page (e.g. `EmptyState`: "Admin store selection is
  not yet supported"). No fake selector, no broken list.
- Admin support lands later as its own subphase (probably named
  `BS.x` for backend stores + `F2.x` for the frontend picker).

## 3. Gap #2 — No Pagination

### Problem

- `GET /stores/{store_id}/inventory` returns the entire dataset for
  the store as a bare JSON array. Source: `services/inventory.py
  ::list_inventory_for_store` — `select(InventoryItem).where(...)`
  without `LIMIT/OFFSET`.
- No `limit`, no `offset`, no cursor, no `total` count.
- No search or filtering beyond the existing `low_stock_only` boolean.

### Impact

- Stores carrying thousands of `(store, variant)` rows will:
  - return large payloads on every list refetch (network + memory),
  - cause slow first-paint and heavy DOM in the frontend table,
  - degrade TanStack Query cache efficiency (one massive cache entry
    that invalidates on any mutation).
- Without pagination the F2.6 frontend would either ship a fragile
  client-side virtualizer over the full dataset (and rewrite when
  pagination lands) or be functionally limited to small stores.

### Decision

```
Implement minimal pagination BEFORE F2.6.
```

## 4. Pagination Scope (Mini Backend Fix)

Smallest safe scope. Locks the wire shape so the frontend builds against
the final contract from day one.

### Request params (query)

| Param | Type | Default | Constraint |
|---|---|---|---|
| `limit` | int | 50 | 1 ≤ limit ≤ 200 |
| `offset` | int | 0 | offset ≥ 0 |
| `low_stock_only` | bool | false | unchanged from current behavior |
| `status` | enum | none | optional; one of `available`/`flagged`/`quarantined` if trivial to add |

`status` is **conditional**: include it only if it ends up being a
single `WHERE` clause addition. If it forces extra service surface
(separate code path, dedicated test class), defer to a later subphase.

### Response shape

Replace the bare array with a paginated envelope:

```jsonc
{
  "items":  [InventoryItemRead, ...],
  "total":  <int>,            // count BEFORE limit/offset, AFTER filters
  "limit":  <int>,            // echoes the resolved limit
  "offset": <int>             // echoes the resolved offset
}
```

The envelope must align with the existing `PaginatedResponse<T>` type
already declared in `nuberushfrontendd/src/api/types.ts`, so the frontend
generic type works unchanged.

### Endpoints affected

| Endpoint | Change |
|---|---|
| `GET /stores/{store_id}/inventory` | bare array → `PaginatedResponse[InventoryItemRead]` |

Logs (`GET /stores/{store_id}/inventory/logs` and
`GET /inventory/{item_id}/logs`) are out of scope. They already cap at
`limit=100` and the frontend can iterate via `before_id` later.

### Constraints

- Do NOT change mutation endpoints (response shape stays
  `InventoryItemRead`).
- Do NOT change business logic, locking, or transactional semantics.
- Do NOT change permissions or tenancy gates.
- Do NOT add filtering beyond `low_stock_only` (and conditional
  `status`).
- Do NOT introduce cursor pagination — offset is sufficient for v1
  given the dataset size and the absence of insert-heavy live feeds
  in this list.
- `total` MUST count rows after filters but before `LIMIT/OFFSET`, so
  the frontend can render correct page counts.

---

## 5. Why Pagination First

- Prevents the performance cliff before any large store hits it.
- Locks the wire shape (envelope) before the frontend builds against
  the bare array, avoiding a rewrite of the entire `useInventoryList`
  hook + table component when pagination eventually lands.
- Aligns with TanStack Query's `keepPreviousData`/`getNextPageParam`
  patterns and the existing `PaginatedResponse<T>` type already
  defined in the F2.2 client foundation.
- Low-risk backend change: additive `limit`/`offset` query params,
  one wrapped response, no new permissions, no new tables, no
  migration. Two service changes (statement + count query) and a
  schema swap.

## 6. Inventory v1 Definition

After this readiness work, Inventory v1 = the surface F2.6 will
implement:

- **Enriched API** — DONE (BIE.1–4).
- **Paginated list** — NEXT (mini backend fix above).
- **Non-admin only** — owner / manager / staff. Admin sees a clear
  "not yet supported" state. Driver excluded by backend.
- **Operation-based mutations** — receive / adjust / damage / sell /
  reserve / release / return / threshold / status. Each modal
  enforces the right schema profile (required reason, signed delta,
  positive quantity).
- **Logs included** — per-store and per-item log views (existing
  endpoints, already capped at `limit=100`).
- **No bulk operations** — no CSV upload, no batch
  receive/adjust/damage. Out of scope for v1.

## 7. Next Steps

Pipeline before F2.6 implementation:

1. **BIP.0 — Backend pagination diagnosis** (read-only)
   - Inspect `list_inventory_for_store`, the total-count query
     pattern (likely `select(func.count(...)).select_from(...)`),
     and any `PaginatedResponse` references already in the codebase
     (orders/products may have a precedent worth reusing).
   - Confirm test-fixture impact (existing API tests assert
     `response.json()` is a list — they will need to assert
     `response.json()["items"]`).
2. **BIP.1 — Backend pagination implementation**
   - Add `limit`/`offset` query params to the inventory list
     endpoint.
   - Update service: `list_inventory_for_store` returns
     `(items, total)` or grow a sibling `count_inventory_for_store`.
   - Replace the route's response_model from
     `list[InventoryItemRead]` to `PaginatedResponse[InventoryItemRead]`.
3. **BIP.2 — Backend pagination tests / validation**
   - Update existing API tests to read `["items"]`.
   - Add new tests: `limit`/`offset` math, `total` correctness with
     filters, pagination over a multi-page dataset.
   - Run targeted suites; full suite optional behind a clear time/cost
     trade-off.
4. **F2.6 — Inventory frontend architecture**
   - Build against the final paginated wire shape from day one.

- Decisions on admin and pagination are explicit and unambiguous.
- Pagination scope is narrow enough that BIP.0/BIP.1/BIP.2 can be
  shipped without scope creep.
- F2.6 has a known target wire shape and a known role boundary
  (non-admin only) before any frontend code is written.
