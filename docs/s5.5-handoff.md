## What landed on main

`backend/app/services/orders.py` — transaction coordinator per
`app/domain/orders_rules.py` §9. Key surface:

### Public functions

- `get_order(db, order_id)` — load with items, 404 if missing.
- `list_orders_for_store(db, store_id, *, status, created_from, created_to)` — store-scoped list.
- `list_order_audit_logs(db, order_id)` — audit rows ordered asc by created_at; 404 if order missing.
- `create_order(db, store_id, payload, actor_user_id, *, customer_user_id=None)` — single-tx create:
  1. Idempotency check on `(store_id, idempotency_key)` → returns existing order if found.
  2. Asserts store exists.
  3. Builds Order(pending, totals=0).
  4. For each line: resolves inventory_item via `(store_id, variant_id)`, calls `_reserve_inventory_locked`, snapshots `unit_price` from `ProductVariant.price`, computes `line_total`.
  5. Recomputes totals (`tax=0`, `total=subtotal`).
  6. Writes `order_created` audit row.
  7. Single `db.commit()`.
- `transition_order_status(db, order_id, payload, actor_user_id)` — forward transitions only; rejects `→ canceled` and `→ returned` (use dedicated funcs); on `→ delivered` consumes reservations.
- `cancel_order(db, order_id, payload, actor_user_id)` — releases reservations, sets status/canceled_at/cancel_reason.
- `return_order(db, order_id, payload, actor_user_id)` — only from `delivered`; replenishes stock via `_return_to_inventory_locked`.

### Private helpers

- `_resolve_inventory_item(db, store_id, variant_id)` — UNIQUE(store_id, variant_id) lookup.
- `_calculate_order_totals(items)` — Decimal subtotal/tax/total.
- `_assert_valid_transition(prev, new)` — uses `_ALLOWED_TRANSITIONS` dict.
- `_write_order_audit_log(db, order, ...)` — append-only.
- `_reserve_order_items(...)` / `_consume_order_reservations(...)` / `_release_order_reservations(...)` / `_return_order_items_to_inventory(...)` — wrap the inventory `_locked` helpers in line-item loops.
- `_load_order(db, order_id)` — selectinload(items).
- `_find_existing_idempotent_order(db, store_id, idempotency_key)` — UNIQUE-pair lookup.
- `_commit_or_translate(db)` — translates `IntegrityError` (UNIQUE on `uq_orders_store_idempotency_key` → 409; everything else → 422).
- `_recheck_order_sellability(db, order)` — currently a no-op: gate lives inside `_consume_reserved_inventory_locked`.

### State machine (frozen here)

```
pending          → {accepted, canceled}
accepted         → {preparing, canceled}
preparing        → {ready, canceled}
ready            → {out_for_delivery, delivered, canceled}
out_for_delivery → {delivered, canceled}
delivered        → {returned}
canceled         → terminal
returned         → terminal
```

### Audit actions

- `order_created`
- `status_changed`
- `order_canceled`
- `order_delivered`
- `order_returned`

`compliance_blocked` deliberately NOT implemented yet (writing it
would require a successful commit before raising, complicating
rollback semantics). Documented as future refinement.

### Inventory side-effects per transition

| Transition | Inventory effect |
|---|---|
| create order | `_reserve_inventory_locked` per line → ↑ qreserved |
| pending → accepted | none (sets `accepted_at`) |
| accepted → preparing | none |
| preparing → ready | none |
| ready → out_for_delivery | none |
| ready/out_for_delivery → delivered | `_consume_reserved_inventory_locked` per line → ↓ qreserved AND ↓ qoh; sets `delivered_at` |
| any pre-delivered → canceled | `_release_reservation_locked` per line → ↓ qreserved; sets `canceled_at`, `cancel_reason` |
| delivered → returned | `_return_to_inventory_locked` per line → ↑ qoh; sets `returned_at` |

## Risks and known issues

1. **Untested rollback semantics.** If `_reserve_order_items` raises
   on item N, the coordinator does `db.rollback()` then re-raises.
   This SHOULD discard items 1..N-1 plus the parent Order row plus
   any inventory_log rows already written. Unverified.
2. **Idempotency race.** Two concurrent `create_order` calls with
   the same `(store_id, idempotency_key)` could both pass the
   pre-check; the second one is caught by `IntegrityError` on flush
   and re-fetches the existing order. Path is implemented but
   never exercised.
3. **Cross-store inventory consumption.** `_consume_reserved_inventory_locked`
   is called with `expected_store_id=order.store_id`, which raises
   422 on mismatch. Tested via inventory's own test suite, not
   yet via the orders coordinator.
4. **Compliance race (banned between reserve and deliver).** The
   gate fires inside `_consume_reserved_inventory_locked` via
   `_assert_item_operable`. The `delivered` transition will return
   422 in that case. The order's status stays at the previous value
   (no audit row written). Behaviour is correct per orders_rules
   §7 but not exercised by a test.
5. **Money handling.** Totals are recomputed from `ProductVariant.price`
   at create time. Frontend cannot tamper with totals or unit_price
   (extra="forbid" on `OrderCreate`). Verified by S5.4 schema tests
   only — no integration test proves the service ignores them.

### Step 2 — Write `backend/tests/test_orders_service.py`

Required test groups (per S5.5 brief §9):

- **A. Create order** — pending status, items created, inventory_item resolved, reservation written, totals from DB, single audit log.
- **B. Idempotency** — replay returns same order; no duplicate items/reservations/audit logs.
- **C. Atomic rollback** — multi-item where line N fails reserve. Assert: no Order, no OrderItem, no InventoryLog, no OrderAuditLog persists. Use a separate engine/session (not the SAVEPOINT db_session) to verify nothing landed on disk — see `tests/test_inventory_transactional.py` for the pattern.
- **D. Cancel** — cancel from pending releases reservation; sets `canceled_at`, `cancel_reason`; writes audit; cancel-from-delivered fails 422.
- **E. Delivered** — `ready → delivered` consumes reservations; qreserved -= q; qoh -= q; sale inventory_log; delivered_at set; order_delivered audit.
- **F. Return** — `delivered → returned` replenishes qoh; returned_at set; audit. Return-from-pending fails 422.
- **G. State machine** — every entry in `_ALLOWED_TRANSITIONS` accepted; every non-entry rejected 422; terminal states (canceled/returned) reject everything.
- **H. Compliance** — banned product blocks create (422); product banned between reserve and deliver blocks delivered (422); cancel still works after ban.
- **I. Cross-store safety** — store A order cannot resolve store B inventory_item; variant without inventory in store fails 422.
- **J. Money** — totals come from DB; client-supplied price is impossible (schema rejects); `tax_amount = 0`; `total_amount = subtotal_amount`.
