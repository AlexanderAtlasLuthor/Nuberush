# F2.19 — Admin Dashboard + Operations Intelligence: Contract Lock

Status: contract / scope freeze. Locks the exact surface of F2.19 before any
backend or frontend implementation begins. No code, models, migrations,
schemas, services, routes, tests, or UI components are introduced by this
document.

This document is the output of subphase **F2.19.0 — Contract Lock / Scope
Freeze**. Implementation subphases (F2.19.1 onward) must match this contract
exactly; any deviation requires updating this document first.

F2.18 (Admin Operations Views) is the immediate predecessor. The
endpoints and UI surfaces it shipped — admin Stores CRUD, admin Audit,
admin Inventory aggregator, admin Orders aggregator — are consumed but
not modified by F2.19.

---

## 1. Purpose

F2.19 converts the two remaining admin placeholders into real,
backend-computed operational surfaces:

- `/app/admin` → real Admin Dashboard backed by `GET /admin/dashboard`.
- `/app/admin/operations` → real Admin Operations Alerts feed backed by
  `GET /admin/operations/alerts`.

The phase is **read-only** and **computed from existing backend data**.
No new persistence is introduced. Both endpoints derive every value on
request from existing tables (Store, User, InventoryItem, Order, Product,
AuditEvent). There is no Alert table, no incident table, no KPI table,
and no cached projection.

The dashboard provides bounded, aggregated counts plus short recent tails
(orders, audit). The operations alerts feed provides a computed,
filterable, paginated stream of operational issues (low stock, aging
orders, compliance blockers, inactive stores, stores with no inventory)
synthesized at query time.

F2.19 is the last admin-operations-only phase before Admin Products,
Admin Compliance, and Admin Settings (reserved for F2.20+).

## 3. Backend Endpoint Contract

Both endpoints are **new**, **admin-only**, **read-only**, and **computed
on request**. Neither writes any data. Neither introduces a migration or
a model. Both reuse existing tables, services, and RBAC primitives.

### 3.1 `GET /admin/dashboard`

Locked properties:

- **Auth:** `require_admin`.
- **Type:** read-only.
- **Top-level pagination:** none. The endpoint returns a single
  aggregate summary object.
- **Recent tails:** bounded (see limits below).
- **Mutations:** none.
- **Fake values:** none. Every field is computed from a real query
  against existing tables.

#### 3.1.1 Response shape

```ts
AdminDashboardSummary = {
  stores: {
    total: number
    active: number
    inactive: number
  }

  users: {
    total: number
    active: number
  }

  inventory: {
    low_stock_count: number
  }

  orders: {
    open_count: number
    by_status: Record<OrderStatus, number>
    recent: OrderRead[]
  }

  compliance: {
    blocked_count: number
  }

  recent_audit: AuditEvent[]
}
```

#### 3.1.2 Computation sources (locked)

- **Stores total / active / inactive** — from `Store.is_active`.
  - `total` = count of all stores.
  - `active` = count where `is_active = true`.
  - `inactive` = count where `is_active = false`.
- **Users total / active** — from `User.is_active`.
  - `total` = count of all users.
  - `active` = count where `is_active = true`.
- **Inventory low stock** — from the existing low-stock predicate:
  `InventoryItem.quantity_on_hand - InventoryItem.quantity_reserved
  <= InventoryItem.reorder_threshold`.
- **Orders by status** — `Order.status` group-by, returned as
  `Record<OrderStatus, number>` covering every `OrderStatus` value
  (zeros included for statuses with no rows).
- **Open orders** — count of `Order` rows where `status` is one of:
  `pending`, `accepted`, `preparing`, `ready`, `out_for_delivery`.
- **Recent orders** — `Order` rows ordered by `created_at DESC`, hard
  limit **5**, serialized via the existing `OrderRead` schema.
- **Compliance blockers** — count of `Product` rows where compliance is
  blocking sale, derived from `Product.compliance_status` and
  `Product.allowed_for_sale` (a product counts as blocked when
  `allowed_for_sale = false` or `compliance_status` is in the blocking
  set: `banned`, `restricted`).
- **Recent audit** — reuses the existing admin / global audit feed
  query logic (same source as `GET /admin/audit`), ordered by
  `created_at DESC`, hard limit **5**, serialized via the existing
  `AuditEvent` schema.

No field on this endpoint may be fabricated, randomized, mocked, or
sourced from anywhere other than the tables listed above.

### 3.2 `GET /admin/operations/alerts`

Locked properties:

- **Auth:** `require_admin`.
- **Type:** read-only.
- **Computation:** computed on request. No persisted alert rows.
- **Pagination:** standard `limit` / `offset` over the computed list.
- **Filterable:** by `category`, `severity`, `store_id`, and
  `aging_minutes`.
- **Persistence:** none.
- **Alert mutations:** none. No acknowledge, dismiss, resolve, snooze,
  or any other state transition.

#### 3.2.1 Query params

- `limit?: number` — default **50**, min **1**, max **200**.
- `offset?: number` — default **0**, min **0**.
- `category?: AlertCategory` — optional filter.
- `severity?: AlertSeverity` — optional filter.
- `store_id?: string` — optional filter. **Filter only.** It does NOT
  enforce tenancy and does NOT restrict admin visibility; an admin
  remains able to query any store.
- `aging_minutes?: number` — default **1440** (24 hours), min **1**.
  Drives the `aging_order` category threshold.

#### 3.2.2 `AlertCategory` values (locked)

- `low_stock`
- `aging_order`
- `compliance_blocker`
- `inactive_store`
- `store_no_inventory`

#### 3.2.3 `AlertSeverity` values (locked)

- `low`
- `medium`
- `high`

#### 3.2.4 Alert item shape

```ts
AdminOperationsAlert = {
  id: string
  category: AlertCategory
  severity: AlertSeverity
  store_id: string | null
  entity_type: "store" | "inventory_item" | "order" | "product"
  entity_id: string
  summary: string
  created_at: string
}
```

#### 3.2.5 Response shape

```ts
AdminOperationsAlertsListResponse = {
  items: AdminOperationsAlert[]
  total: number
  limit: number
  offset: number
}
```

`total` is the count of computed alerts that match the active filters,
before `limit` / `offset` are applied. `items` is the sliced page.

#### 3.2.6 Alert category sources (locked)

- **`low_stock`** — derived from the same `InventoryItem` low-stock
  predicate used by the dashboard:
  `quantity_on_hand - quantity_reserved <= reorder_threshold`.
  `entity_type = "inventory_item"`. `store_id` = the inventory item's
  store.
- **`aging_order`** — `Order.status` is one of the open statuses
  (`pending`, `accepted`, `preparing`, `ready`, `out_for_delivery`) AND
  `now - Order.created_at >= aging_minutes`. `entity_type = "order"`.
  `store_id` = the order's store.
- **`compliance_blocker`** — `Product.compliance_status` is in the
  blocking set OR `Product.allowed_for_sale = false`.
  `entity_type = "product"`. `store_id` = `null` (products are not
  store-scoped in the current model).
- **`inactive_store`** — `Store.is_active = false`.
  `entity_type = "store"`. `store_id` = the store's id.
- **`store_no_inventory`** — `Store` rows whose `LEFT OUTER JOIN
  InventoryItem` produces zero matches (no inventory rows for the
  store). `entity_type = "store"`. `store_id` = the store's id.

#### 3.2.7 Severity rules (locked)

`available_quantity = quantity_on_hand - quantity_reserved`.
`age_minutes = (now - Order.created_at) in minutes`.

- **`low_stock`:**
  - `high` if `available_quantity <= 0`.
  - `medium` if `available_quantity <= reorder_threshold`.
  - `low` should not normally appear.
- **`aging_order`:**
  - `high` if `age_minutes >= aging_minutes * 2`.
  - `medium` if `age_minutes >= aging_minutes`.
  - `low` should not normally appear.
- **`compliance_blocker`:**
  - `high` if `compliance_status == banned` OR
    `allowed_for_sale == false`.
  - `medium` if `compliance_status == restricted`.
  - `low` should not normally appear.
- **`inactive_store`:** always `medium`.
- **`store_no_inventory`:** always `medium`.

#### 3.2.8 Deterministic ordering (locked)

Computed alerts are sorted by, in order:

1. **Severity priority DESC** — `high` > `medium` > `low`.
2. **`created_at` DESC** — newer entities first.
3. **`category` ASC** — alphabetical, stable across categories.
4. **`entity_id` ASC** — stable tiebreaker.

The same request with the same parameters and the same underlying data
must return the same ordering.

#### 3.2.9 Deterministic alert IDs (locked)

Alert `id` values are derived, not random, so the same condition
produces the same id across requests:

- **`low_stock`:** `low_stock:<inventory_item_id>`
- **`aging_order`:** `aging_order:<order_id>:<aging_minutes>`
- **`compliance_blocker`:** `compliance_blocker:<product_id>`
- **`inactive_store`:** `inactive_store:<store_id>`
- **`store_no_inventory`:** `store_no_inventory:<store_id>`

`aging_minutes` is included in the `aging_order` id because the alert
identity depends on the threshold the caller applied.

## 5. Frontend Namespace Contract

Locked future namespaces:

- `src/features/admin-dashboard/` — owns the Admin Dashboard contract,
  hooks, and UI for `/app/admin`.
- `src/features/admin-operations/` — owns the Admin Operations Alerts
  contract, hooks, and UI for `/app/admin/operations`.

Locked constraint:

- The admin dashboard **must not** extend `src/features/dashboard/`.
  That namespace is tied to **store context** (per-store dashboard) and
  is not safe to widen into admin / global scope. The admin dashboard
  is a separate, admin-scoped surface with a different RBAC model, a
  different data source, and a different intent.

## 6. RBAC Matrix

Both new endpoints are admin-only. The exact access matrix is:

| Actor     | `/admin/dashboard` | `/admin/operations/alerts` |
| --------- | ------------------ | -------------------------- |
| Anonymous | 401                | 401                        |
| Admin     | 200                | 200                        |
| Owner     | 403                | 403                        |
| Manager   | 403                | 403                        |
| Staff     | 403                | 403                        |
| Driver    | 403                | 403                        |

Additional RBAC rules:

- **No store tenancy guard** applies to these endpoints. They are
  global admin surfaces. There is no per-store membership check, no
  `require_store_member`, and no per-store role gate.
- `store_id` in the operations alerts query is **only a filter**, not
  tenancy enforcement. An admin remains free to omit it or to query
  any store id.
- The frontend **must not rely on client-side role gating for
  security**. The backend `require_admin` check is the source of
  truth. Client-side UI gates are UX only.

## 8. F2.19 Success Condition

F2.19 is complete **only when all of the following hold**:

- `/app/admin` is a real backend-computed dashboard.
- `/app/admin/operations` is a real computed operations alert feed.
- `GET /admin/dashboard` exists and is admin-only.
- `GET /admin/operations/alerts` exists and is admin-only.
- There is **no fake data** anywhere in either surface.
- There is **no persisted alert system** — no Alert table, no alert
  mutations, no incident model.
- There are **no migrations or model changes** introduced by F2.19.
- Admin **Products**, **Compliance**, and **Settings** remain reserved
  for later phases and remain placeholder routes.

Failing any of the above means F2.19 is not done, regardless of UI
polish.
