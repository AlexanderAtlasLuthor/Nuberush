# F2.18 — Admin Operations Views: Contract Lock

Status: contract / scope freeze. Locks the exact surface of F2.18 before any
backend or frontend implementation begins. No code, models, migrations,
schemas, services, routes, tests, or UI components are introduced by this
document.

This document is the output of subphase **F2.18.0 — Contract Lock / Scope
Freeze**. Implementation subphases (F2.18.1 onward) must match this contract
exactly; any deviation requires updating this document first.

F2.17 (Admin Backend Foundations) is the immediate predecessor. The
endpoints it shipped — admin store CRUD on `/stores` and `GET /admin/audit`
— are consumed but not modified by F2.18.

---

## 1. Purpose

F2.18 turns the Admin Platform Console from a shell into a functional app
for the three operational surfaces that already have backend support
(Stores, Audit) or only require a thin read-only aggregator (Inventory,
Orders).

F2.18 is the first phase that ships **real Admin UI bound to real
contracts**. It is deliberately scoped down: no KPIs, no alerts, no
mutations beyond what F2.17 already exposes, no migrations, no model
changes, and no work outside the four locked surfaces (Stores, Audit,
Inventory, Orders).

The diagnostic preceding this phase confirmed:

- F2.17 ships admin Stores CRUD + lifecycle on `/stores` and `GET
  /admin/audit`. No additional backend is required for Admin Stores or
  Admin Audit UIs.
- Inventory and Orders today are store-scoped only
  (`GET /stores/{store_id}/inventory`, `GET /stores/{store_id}/orders`).
  An admin who needs a cross-store view must either fan out per-store or
  use a new global aggregator.
- The minimum backend addition to unblock Admin Inventory + Admin Orders
  read-only UIs is **two global read-only endpoints** that reuse existing
  services, schemas, and RBAC. No mutation surface is introduced.
- The frontend (Lovable / `nuberushfrontendd`) currently has Admin
  placeholders. F2.18 replaces those placeholders for the four locked
  routes only.

   - Admin Inventory contract (types, list hook, query-param shape).

   - `/app/admin/stores` — list + create + lifecycle actions.

## 3. Backend Endpoints Locked

### 3.1 New in F2.18

```txt
GET /admin/inventory
GET /admin/orders
```

Both are **admin-only**, **read-only**, paginated, filterable, and reuse
existing services / schemas (see §8). They are the only backend additions
in F2.18.

### 3.2 Existing endpoints consumed by F2.18 frontend (unchanged)

```txt
GET    /stores
POST   /stores
GET    /stores/{store_id}
PATCH  /stores/{store_id}
POST   /stores/{store_id}/deactivate
POST   /stores/{store_id}/reactivate
GET    /admin/audit
```

F2.18 must **consume these as-is**. Any modification to request shape,
response shape, query params, status codes, or RBAC of an F2.17 endpoint
is forbidden in F2.18 and requires a separate contract update.

### 3.3 Explicit non-creation

The namespace `/admin/stores` must **not** be introduced. F2.17 already
established that admin store CRUD lives under `/stores`, gated by
`require_admin` at the route layer (precedent: `/auth/users`). Creating
`/admin/stores` in F2.18 would fork the API surface and is forbidden.

The following backend surfaces are out of scope and must **not** be
created in F2.18 (see §7 for full list):

- `GET /admin/dashboard` or any KPI/metrics endpoint.
- `GET /admin/operations`, `GET /admin/alerts`, `GET /admin/incidents`.
- `GET /admin/reports`, any export endpoint.
- `GET /admin/compliance` mutation endpoints
  (`approve` / `reject` / `hold`).
- `GET /admin/settings` or platform configuration.
- Any product, customer, driver, payments, or delivery endpoint.

## 5. Frontend Contracts Required

Four contracts must be defined as part of F2.18.2 before any Admin UI is
built. Each contract is a TypeScript module that owns: response types,
request types, query-param shape, and the hook(s) the UI consumes.

### 5.1 Admin Stores contract

Surfaces the F2.17 admin Stores backend.

```txt
Types
  StoreRead              { id, name, code, is_active, timezone,
                           created_at, updated_at }
  StoreListResponse      { items: StoreRead[], total, limit, offset }
  StoreCreate            { name, code, timezone? }
  StoreUpdate            { name?, timezone? }

Hooks
  useAdminStores(params)            → GET /stores
    params: { limit?, offset?, is_active?, q? }
  useAdminStore(storeId)            → GET /stores/{store_id}
  useCreateAdminStore()             → POST /stores
  useUpdateAdminStore(storeId)      → PATCH /stores/{store_id}
  useDeactivateAdminStore(storeId)  → POST /stores/{store_id}/deactivate
  useReactivateAdminStore(storeId)  → POST /stores/{store_id}/reactivate
```

The hook names are normative; the underlying client (fetch / axios /
react-query etc.) is an implementation detail of F2.18.2.

### 5.2 Admin Audit contract

Surfaces the F2.17 global audit feed.

```txt
Types
  AuditEventRead          { id, source, entity_type, action, store_id,
                            actor_id, actor_email?, target_id?,
                            target_label?, payload?, created_at }
  AuditEventListResponse  { items: AuditEventRead[], total, limit, offset }

Hooks
  useAdminAudit(params)   → GET /admin/audit
    params: { limit?, offset?, store_id?, source?, entity_type?,
              action?, actor_id?, date_from?, date_to? }
```

Field set on `AuditEventRead` must match the F2.16/F2.17 backend envelope
exactly. Any unknown field surfaced by the backend is preserved
verbatim; the contract does not strip fields.

### 5.3 Admin Inventory contract

Surfaces the new `GET /admin/inventory` aggregator (F2.18.1 shipped
surface).

```txt
Types
  InventoryItemRead          (same shape the store-scoped endpoint
                              returns today — store_id, variant_id,
                              quantity_on_hand, quantity_reserved,
                              reorder_threshold, status,
                              last_counted_at, created_at, updated_at,
                              variant: InventoryVariantSummary)
  InventoryItemListResponse  { items: InventoryItemRead[],
                               total, limit, offset }

Hooks
  useAdminInventory(params)  → GET /admin/inventory
    params: {
      limit?,
      offset?,
      store_id?,
      low_stock?,        // bool, default false
      q?,                // ILIKE on variant.sku + product.name
      product_id?,       // UUID
      variant_id?,       // UUID
      status?,           // InventoryStatus enum on the wire
    }
```

The Admin Inventory UI is **read-only**. No mutation hooks
(receive / adjust / sell / reserve / damage / release / return) are
exposed at admin scope in F2.18. Existing store-scoped mutation hooks
are untouched.

### 5.4 Admin Orders contract

Surfaces the new `GET /admin/orders` aggregator.

```txt
Types
  OrderRead          (same shape the store-scoped endpoint returns
                      today — id, store_id, customer_user_id,
                      idempotency_key, status, subtotal_amount,
                      tax_amount, total_amount, age_verified_at,
                      age_verified_by_user_id, accepted_at,
                      canceled_at, delivered_at, returned_at,
                      cancel_reason, notes, created_at, updated_at,
                      items: OrderItemRead[])
  OrderListResponse  { items: OrderRead[], total, limit, offset }

Hooks
  useAdminOrders(params)  → GET /admin/orders
    params: { limit?, offset?, store_id?, status?, date_from?, date_to? }
```

The Admin Orders UI is **read-only**. No mutation hooks (create /
cancel / return / status-transition) are exposed at admin scope in
F2.18. Existing store-scoped mutation hooks are untouched.

## 6. RBAC / Tenancy Matrix

The matrix below is normative for F2.18. Existing rows are repeated from
F2.17 for completeness and **must not change**; new rows are flagged.

| Endpoint | admin | owner | manager | staff | driver | anon |
| --- | --- | --- | --- | --- | --- | --- |
| `GET /stores` (F2.17) | 200 | 403 | 403 | 403 | 403 | 401 |
| `POST /stores` (F2.17) | 201 | 403 | 403 | 403 | 403 | 401 |
| `GET /stores/{id}` (F2.14) | 200 any | 200 own / 403 cross | 200 own / 403 cross | 200 own / 403 cross | 200 own / 403 cross | 401 |
| `PATCH /stores/{id}` (F2.14) | 200 any | 200 own | 403 | 403 | 403 | 401 |
| `POST /stores/{id}/deactivate` (F2.17) | 200 | 403 | 403 | 403 | 403 | 401 |
| `POST /stores/{id}/reactivate` (F2.17) | 200 | 403 | 403 | 403 | 403 | 401 |
| `GET /admin/audit` (F2.17) | 200 | 403 | 403 | 403 | 403 | 401 |
| `GET /admin/inventory` **(F2.18, new)** | 200 | 403 | 403 | 403 | 403 | 401 |
| `GET /admin/orders` **(F2.18, new)** | 200 | 403 | 403 | 403 | 403 | 401 |

Tenancy / anti-probe rules for the new endpoints:

- Non-admin on `GET /admin/inventory` or `GET /admin/orders` → **403**.
- Anonymous → **401** (standard auth gate).
- `store_id` query param on a new endpoint, when the store does not
  exist → **404** (consistent with `list_admin_audit` behavior).
- `store_id` query param identifying an inactive store → returns events
  unchanged (deactivated stores retain history; not a 400).
- The new endpoints must use `require_admin` only. They must not use
  `require_store_member` or any tenancy guard — admins can see every
  store; tenancy concepts do not apply to global feeds.

Frontend RBAC mirroring:

- Admin nav exposes the five locked admin routes only to users with
  `role === "admin"`. Non-admin users must not see admin nav entries or
  reach admin routes by deep link. Existing role-gating already in place
  for F2.17 placeholders is reused — F2.18 does not introduce new role
  guards.

## 8. Backend Implementation Constraints

These constraints govern F2.18.1 (Global Inventory / Orders aggregators).
They are normative: the implementation must satisfy each item or the
contract is broken.

### 8.1 Reuse, do not re-create

- `GET /admin/inventory` **must** reuse `InventoryItemRead` and
  `InventoryItemListResponse` unchanged. No new response schema.
- `GET /admin/orders` **must** reuse `OrderRead` and `OrderListResponse`
  unchanged. No new response schema.
- Eager-loading conventions
  (`InventoryVariantSummary` + `InventoryProductSummary` on inventory;
  `OrderItemRead` + `InventoryVariantSummary` on orders) **must** match
  the existing store-scoped endpoints to avoid N+1.

### 8.2 Filters and pagination

`GET /admin/inventory` (shipped surface, F2.18.1):

```txt
limit            int            ge=1, le=500, default 100
offset           int            ge=0,         default 0
store_id         UUID           optional      scope to one store
low_stock        bool           optional,     default False
                                              (quantity_on_hand - quantity_reserved
                                               <= reorder_threshold)
q                str            optional      ILIKE over variant.sku and
                                              variant.product.name
                                              (case-insensitive, trimmed,
                                               whitespace-only collapses to none)
product_id       UUID           optional      filter to one product
variant_id       UUID           optional      filter to one variant
status           InventoryStatus optional      filter by item status
                                              (wire name `status`,
                                               service kwarg
                                               `inventory_status` to avoid
                                               shadowing `fastapi.status`)
```

F2.18.0 originally enumerated `limit`, `offset`, `store_id`,
`low_stock_only`, `q`. F2.18.1 expanded the read-only filter set with
`product_id`, `variant_id`, and `status`, and renamed `low_stock_only`
→ `low_stock` so the admin namespace can grow consistently. All
additions are read-only filters on existing model relationships; no
new columns, joins beyond `InventoryItem → ProductVariant → Product`,
or schema changes were required.

`GET /admin/orders` (shipped surface, F2.18.1B):

```txt
limit       int          ge=1, le=200, default 50
offset      int          ge=0,         default 0
store_id    UUID         optional      scope to one store
status      OrderStatus  optional      filter by current status
                                       (wire name `status`,
                                        service kwarg `order_status`
                                        to avoid shadowing
                                        `fastapi.status`)
date_from   datetime     optional      inclusive lower bound on
                                       Order.created_at (>=)
date_to     datetime     optional      inclusive upper bound on
                                       Order.created_at (<=)
```

`q` is intentionally **not** implemented for `/admin/orders` in
F2.18.1B. The `Order` model has no clean text-search target
(`idempotency_key` and `notes` are operational metadata, not
user-facing search keys); skipping `q` keeps the admin surface aligned
with the original F2.18.0 §8.2 lock for `/admin/orders`. The brief
listed `q` as a target filter "only if cleanly supported"; the
decision recorded here is that orders do not satisfy that condition.

Both endpoints **must** return `total` pre-pagination, mirroring F2.17
list envelopes (`StoreListResponse`, `UserListResponse`,
`AuditEventListResponse`).

### 8.3 Sort

- `GET /admin/inventory` — **shipped order: `updated_at DESC, id ASC`**
  (F2.18.1). Newer activity first so the admin view surfaces recently
  mutated stock; `id ASC` is the deterministic tie-breaker. This
  supersedes the F2.18.0 sketch (`store_id ASC, variant.sku ASC` /
  `created_at DESC, id ASC`).
- `GET /admin/orders` — **shipped order: `created_at DESC, id ASC`**
  (F2.18.1B). Newest orders first, with `id ASC` as the deterministic
  tie-breaker — matches the rest of the admin namespace
  (`list_admin_audit`, `list_admin_inventory`). The store-scoped
  `GET /stores/{store_id}/orders` keeps its existing
  `created_at DESC, id DESC` sort unchanged.

### 8.4 Services

- A new service function may be added per endpoint
  (e.g. `list_inventory_global`, `list_orders_global`) **only if** the
  existing per-store service cannot be reused with an optional
  `store_id=None` argument.
- The preferred approach is to **extend the existing service** with an
  optional `store_id` parameter (`None` ⇒ global). Decision deferred to
  F2.18.1, but the contract is: no parallel duplicate of business
  logic. One source of truth per entity.
- Services **must** keep the `actor` parameter and enforce admin-only
  at the service layer, exactly as `list_admin_audit` does.

### 8.5 RBAC at the route

```txt
GET /admin/inventory → Depends(require_admin)
GET /admin/orders    → Depends(require_admin)
```

No tenancy dependency. No store-member dependency. No role-alias gate
other than `require_admin`.

### 8.6 What the backend must NOT do

- No new `AuditSource` enum value, no new audit log writes from the new
  endpoints (reads only).
- No model changes. No migrations. No alembic revision.
- No new mutation endpoints anywhere.
- No idempotency layer (these are GETs).
- No background tasks, no caching layer.
- No fake / mock data, no test fixtures beyond what tests in §10 need.
- No edits to F2.17 routes, services, schemas, or tests except as
  necessary to add the optional `store_id` parameter referenced in §8.4
  (and only on the inventory/orders services, never on stores or audit).

## 9. Frontend Implementation Constraints

These constraints govern F2.18.2 (contracts), F2.18.3 (Stores UI),
F2.18.4 (Audit UI), and F2.18.5 (Inventory/Orders read-only UI).

### 9.1 Contracts before UI

F2.18.2 must land before any of F2.18.3 / .4 / .5 begins. The four
contracts in §5 are the only API surface admin pages may touch; no page
may issue an ad-hoc fetch outside its contract.

### 9.2 Real data only

- No mock data in components.
- No hard-coded admin data (no static arrays standing in for API
  responses).
- No `// TODO wire up` comments left in shipped code.
- Loading, empty, and error states must be real and match existing
  store-side conventions.

### 9.3 Read-only enforcement

- Admin Inventory UI must not render mutation controls (no
  receive/adjust/sell/etc. buttons or forms).
- Admin Orders UI must not render mutation controls (no
  create/cancel/return/status-transition buttons or forms).
- Admin Stores UI may render create + lifecycle (deactivate /
  reactivate) controls because those F2.17 endpoints exist.
- Admin Audit UI is inherently read-only; filters only.

### 9.4 Routing and shell

- The five locked routes must be registered as real routes; the
  remaining admin routes (`/app/admin`, `/app/admin/operations`,
  `/app/admin/settings`, and optionally `/app/admin/products`,
  `/app/admin/compliance`) stay as placeholders.
- Admin nav must list the five locked routes for admin users only.
- Non-admin users hitting an admin route by deep link must be redirected
  or shown a 403 consistent with current behavior. F2.18 does not
  redesign this gate.

### 9.5 What the frontend must NOT do

- No model / schema duplication beyond the contract files.
- No new auth flows or token handling.
- No new global state primitives (Redux slices, Zustand stores, context
  providers) beyond what react-query (or equivalent) needs for the four
  contracts.
- No customer / driver / marketplace components.
- No edits to `/app/store/*` operational pages.
- No design-system additions unrelated to the locked admin routes.
- No analytics / telemetry instrumentation beyond what exists today.

## 13. Scope Guard

This section is the contract's enforcement clause. It exists to make
scope creep visible.

### 13.1 Hard prohibitions (no exceptions in F2.18)

```txt
Do NOT modify backend implementation files except as F2.18.1 requires.
Do NOT modify frontend implementation files except as F2.18.2 / .3 /
   .4 / .5 (and optionally .6) require.
Do NOT modify test files except as F2.18.1 / .2 onward require to
   cover the locked surface.
Do NOT touch db models.
Do NOT touch migrations.
Do NOT introduce a new alembic revision.
Do NOT add fake data.
Do NOT add mock admin data.
Do NOT add customer / driver / payments / delivery scope of any kind.
Do NOT create /admin/stores backend namespace.
Do NOT create Admin Dashboard / Operations / Settings backends or
   pages beyond the placeholders.
Do NOT create reporting / export endpoints.
Do NOT create compliance mutation endpoints.
Do NOT create platform settings endpoints.
Do NOT create product media endpoints.
Do NOT introduce new RBAC roles or guards.
Do NOT modify F2.17 endpoint shapes (request / response / RBAC).
Do NOT modify existing store-scoped inventory or order mutation
   endpoints.
```

### 13.2 Soft prohibitions (require this document to be updated first)

```txt
Adding a new query param to a locked endpoint.
Adding a new field to a locked response schema.
Reordering subphases.
Renaming a locked route.
Renaming a locked hook.
Promoting any placeholder route to a real page mid-phase (other than
   F2.18.6 if approved).
```

Any of the soft prohibitions requires (a) updating this document,
(b) re-confirming with the human owner, and (c) recording the change in
the subphase that introduces it.

### 13.3 Drift signals (treat as build-stop)

If any of the following appears during F2.18, halt the subphase and
re-open this document:

- A new file under `backend/app/db/migrations/` or similar alembic
  output.
- A new column in `backend/app/db/models.py`.
- A new enum value in `AuditSource`, `OrderStatus`, `InventoryStatus`,
  or any user-role enum.
- A new route prefix `/admin/stores`, `/admin/dashboard`,
  `/admin/operations`, `/admin/alerts`, `/admin/incidents`,
  `/admin/reports`, `/admin/settings`, or any export path.
- A new mutation endpoint anywhere under `/admin/*`.
- Any frontend file containing a hard-coded admin record literal that
  isn't part of a test fixture.

End of contract.
