# F2.22 Contract Lock — Supabase Hybrid Backend Adoption

Status: contract / scope freeze. This document is the output of subphase
**F2.22.0 — Contract Lock / Scope Freeze**. It locks the target
architecture, scope, boundaries, non-goals, phase plan, security model,
testing model, rollback plan, and validation gates for adopting Supabase
**without** replacing FastAPI.

No code, migrations, policies, buckets, or tests are introduced by this
document. Implementation subphases (F2.22.1 onward) must match this
contract exactly; any deviation requires updating this document first.

This contract supersedes the earlier "Supabase-native backend
replacement" diagnostic, which proposed a full FastAPI retirement. A
backend audit (60 Python modules, 72 HTTP endpoints, 1365 pytest tests,
4 Alembic migrations, the React/Vite/Capacitor frontend) found the
business core to be sound and pinned by tests. F2.22 therefore adopts a
**hybrid** architecture instead.

---

## 1. Purpose

F2.22 exists to adopt Supabase for the infrastructure concerns where it
removes real risk or real work, **without** rewriting the business logic
the audit found to be correct and well-tested.

The problems F2.22 solves:

- **Session is a known stub.** The frontend access token lives only in
  memory (`web/src/api/session-token.ts`) and does not survive a reload;
  the file itself says "F2.3 will replace this." There is no native
  session persistence for the Capacitor iOS/Android builds.
- **Self-managed Postgres** is an operational burden with no managed
  backups, pooling, or observability story.
- **No file storage** exists — product images, store assets, and
  compliance attachments have nowhere to live.
- **No realtime** — operational surfaces (orders, inventory) only update
  on manual refetch.

What F2.22 deliberately does **not** try to solve: the orders state
machine, inventory transactionality, RBAC, tenancy, compliance, audit,
earnings, and dashboards. The audit confirmed these are correct and
covered by 1365 tests. Rewriting ~2000 lines of transactional Python
into PL/pgSQL would orphan that suite for value the hybrid model already
captures.

---

## 2. Core Decision

**F2.22 adopts Supabase as the infrastructure layer but keeps FastAPI as
the business-logic authority.**

NubeRush moves to a hybrid Supabase + FastAPI architecture:

- **Supabase = infrastructure layer** — Auth, managed Postgres hosting,
  Storage, Realtime, RLS as defense-in-depth.
- **FastAPI = business authority** — RBAC, tenancy, orders, inventory,
  compliance, product approval, earnings, dashboards, audit, admin and
  store operations, user management.
- **Frontend = thin client** — keeps calling FastAPI through
  `apiRequest()`; uses `supabase-js` only for auth, storage, and
  realtime invalidation signals.
- **RLS = defense-in-depth** — never the primary authorization gate.
- **Realtime = invalidation signal only** — never a data source.
- **Alembic = schema authority** — owns every `public.*` app table.

F2.22 does **not** replace FastAPI. See §15 for the full non-goals list.

## 4. Responsibility Split

### 4.1 Supabase responsibilities

- Hosting managed Postgres (backups, pooling, observability).
- Identity provider: login, refresh, invite, password reset, the
  `auth.users` table.
- Object storage: the four buckets in §8.
- Realtime change streams on `orders` and `inventory_items`.
- Enforcing RLS as a secondary, defense-in-depth layer.

### 4.2 FastAPI responsibilities

- Verifying the Supabase JWT and resolving the application user.
- RBAC: the 5 roles and 4 authorization matrices (`permissions.py`).
- Tenancy / `store_id` enforcement (`deps.py`).
- The orders state machine and its single-transaction coordinator.
- Inventory transactions, including `SELECT ... FOR UPDATE` row locking.
- Compliance rules and the product approval workflow.
- Earnings and dashboard aggregation.
- The audit feed (read-time merge of the three log tables).
- Admin operations, store operations, and user management.
- Issuing signed Supabase Storage upload URLs after authorizing them.
- Writing all storage metadata rows.

FastAPI remains the **only** component that reads or writes business
tables. This responsibility set is unchanged by F2.22 except for JWT
verification and user creation (§5).

### 4.3 Frontend responsibilities

- Authenticating through `supabase-js` and holding the Supabase session.
- Sending the Supabase access token to FastAPI as a Bearer token.
- Uploading/downloading files through `supabase-js` storage helpers,
  using signed URLs issued by FastAPI.
- Subscribing to Supabase Realtime and converting events into TanStack
  Query cache invalidations.
- Performing all business reads and writes through `apiRequest()` and
  the existing feature `api.ts` modules.

The frontend must **not** use `supabase-js` for direct business table
`SELECT`, `INSERT`, `UPDATE`, or `DELETE`.

### 4.4 Alembic responsibilities

- Remains the single schema authority for every `public.*` application
  table: columns, indexes, CHECK constraints, enums, foreign keys.
- Owns the planned `public.users.auth_user_id` column addition (§5) and
  any storage-metadata tables.
- F2.22 does not migrate the schema into Supabase's migration system.

### 4.5 RLS responsibilities

- Provides a secondary containment layer so the Realtime channel and any
  accidental direct access cannot leak cross-tenant data.
- Is **not** the primary authorization gate — FastAPI is.
- Starts deny-all on every `public.*` table.
- Carries positive `SELECT` policies only for the realtime-exposed
  tables (`orders`, `inventory_items`) in F2.22.
- Carries **no** `INSERT` / `UPDATE` / `DELETE` policies for the
  `authenticated` role.

## 5. Auth Architecture

Decision input: the project is **pre-launch with no production users**
(seed/dev data only). This removes the need for an invite ceremony, a
password-reset campaign, or legacy-login coexistence.

Locked:

- **Supabase Auth becomes the only identity provider.** FastAPI stops
  minting its own JWTs.
- The **frontend logs in with Supabase** (`supabase-js`
  `signInWithPassword`).
- The frontend **sends the Supabase access token to FastAPI** as the
  `Authorization: Bearer` header.
- **FastAPI verifies the Supabase JWT via JWKS** (the project JWKS
  endpoint; the project JWT secret is a fallback only). Expected
  audience is `authenticated`.
- **FastAPI loads `public.users` by `auth_user_id`** (= the JWT `sub`).
- **Role, `store_id`, and `is_active` come from `public.users`**, never
  from JWT claims. The JWT establishes identity only; authority data
  stays in the application table.
- `POST /auth/login`, `create_access_token`, `verify_password`,
  `hash_password`, `get_dummy_password_hash` are **removed** in F2.22.2.
- `POST /auth/users` is **kept**: it still enforces the full RBAC
  creation matrix, but creates the `auth.users` row via the Supabase
  Admin API and the matching `public.users` row in the same request
  (both or neither).
- `GET /auth/me` is **kept** (reads `public.users`).
- **No `auth.users` insert trigger** auto-creates `public.users`: a new
  identity has no role or `store_id`, which are decided by the RBAC
  matrix at `POST /auth/users` time. User creation stays admin-driven.
- **Admin bootstrap** is a one-off seeded step.

Planned DB change (owned by Alembic, executed in F2.22.2):

```text
public.users.auth_user_id  uuid  unique  nullable
```

`nullable` only during the F2.22.2 window; `password_hash` becomes
nullable in F2.22.2 and is dropped in a later cleanup migration.

## 6. Database Architecture

Locked:

- **Supabase hosts Postgres.**
- **Alembic remains the schema authority** for every `public.*`
  application table.
- **Supabase migration files** (`supabase/migrations/*.sql`) own only:
  - RLS policies,
  - the Realtime publication,
  - Storage buckets,
  - Storage policies,
  - Supabase-specific auth wiring (the `auth_user_id` FK to
    `auth.users`, RLS helper functions).
- Apply order on any environment: **Alembic first** (tables), **Supabase
  migrations second** (policies / publication / storage).
- **FastAPI must use the Supavisor session pooler.**
- **Transaction-mode pooling is forbidden** — it breaks SQLAlchemy
  prepared statements and the `SELECT ... FOR UPDATE` row-lock semantics
  the inventory and orders flows depend on. The SQLAlchemy pool size
  stays small.

---

## 7. RLS Architecture

Locked:

- **RLS is defense-in-depth only.** FastAPI authorization remains the
  primary gate.
- **Every `public.*` table starts with RLS enabled and a deny-all
  baseline.**
- **FastAPI connects with an RLS-bypassing role** (the service role, or
  a dedicated `BYPASSRLS` application role), so FastAPI behavior and the
  1365 tests are unaffected by RLS.
- **Positive `SELECT` policies exist only for the realtime-exposed
  tables** in F2.22: `orders` and `inventory_items`. A row is visible to
  an `authenticated` user when the user is an admin, or when the row's
  `store_id` equals the user's `store_id`. These policies use helper SQL
  functions (e.g. `current_app_user_store_id()`, `is_admin()`) that read
  `public.users` keyed by `auth.uid()`.
- **No direct frontend write policies.**
- **No `INSERT` / `UPDATE` / `DELETE` policies for `authenticated`
  users.** All writes go through FastAPI. This is the explicit hybrid
  boundary.

## 8. Storage Architecture

Storage is greenfield. F2.22 provisions all four buckets and their
policies; only `product-images` is wired end-to-end in this phase.

| Bucket | Access | Wired end-to-end in F2.22 |
| --- | --- | --- |
| `product-images` | Public read; writes via FastAPI-issued signed URLs | Yes |
| `store-assets` | Signed URLs | Bucket + policy only |
| `compliance-attachments` | Private | Bucket + policy only |
| `exports` | Private | Bucket + policy only |

Locked `product-images` upload flow:

```text
1. Frontend requests a signed upload URL from FastAPI.
2. FastAPI checks RBAC + tenancy for the action.
3. FastAPI issues a short-lived signed Supabase Storage upload URL.
4. Frontend uploads the file directly to Supabase Storage.
5. FastAPI writes the image metadata row (public.* table, Alembic-owned).
6. Frontend refetches product / image data through FastAPI and renders
   the FastAPI response.
```

- Every stored object has a corresponding `public.*` metadata row
  (owning entity, uploader, created_at). FastAPI owns metadata writes.
- The frontend never holds the service-role key; it only ever receives a
  short-lived signed URL.

---

## 9. Realtime Architecture

Locked:

- **Realtime is only an invalidation signal.** The frontend never
  renders realtime payloads directly.
- **Realtime tables in F2.22:** `orders` and `inventory_items`.
- Event lifecycle:

```text
Realtime event received
  → invalidate the matching TanStack Query key
  → refetch from FastAPI
  → render the FastAPI response
```

- Realtime change visibility is bounded by the §7 `SELECT` policies.
- Out of scope for F2.22: realtime on dashboards, alerts, compliance, or
  audit — those remain poll/refetch.

## 10. Security Model

- **Primary security layer:** FastAPI — RBAC, tenancy, and the business
  service logic. Every business read and write is authorized here.
- **Secondary security layer:** Supabase RLS — deny-all baseline plus
  the narrow realtime `SELECT` policies, containing accidental or
  realtime-channel data exposure.

Frontend **allowed** environment variables (public, build-time):

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`
- `VITE_API_BASE_URL`

Frontend **forbidden** secrets (must never reach the client bundle):

- `SUPABASE_SERVICE_ROLE_KEY`
- `DATABASE_URL`
- the JWT secret
- pooler credentials

Backend **allowed** environment variables:

- `DATABASE_URL` (Supavisor session pooler connection string)
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `SUPABASE_JWKS_URL`
- `SUPABASE_JWT_AUDIENCE`

The service-role key and database credentials live only on the backend.

## 12. Subphase Scope Summaries

### F2.22.0 — Contract Lock / Scope Freeze

- **Objective:** lock the hybrid architecture, scope, and boundaries.
- **Allowed work:** create/replace `docs/f2.22-contract-lock.md`.
- **Forbidden work:** any code, migration, policy, bucket, or test.
- **Exit gate:** the contract exists with all 17 sections; architecture,
  phase plan, non-goals, rollback, and testing model are explicit; no
  implementation file changed.

### F2.22.1 — Supabase Postgres Host Cutover

- **Objective:** move Postgres hosting to Supabase with no behavior
  change.
- **Allowed work:** provision the Supabase project; run the existing
  Alembic migrations and the seed against Supabase Postgres; repoint
  `DATABASE_URL` to the Supavisor session pooler; adjust test-fixture
  connection strings.
- **Forbidden work:** auth changes, RLS, storage, realtime, schema
  changes, business-logic changes.
- **Exit gate:** the full backend suite is green against Supabase
  Postgres; the inventory-concurrency and orders-state-machine suites
  pass under the session pooler.

### F2.22.2 — Supabase Auth Adoption

- **Objective:** make Supabase Auth the only identity provider.
- **Allowed work:** add `public.users.auth_user_id` (Alembic); switch
  `get_current_user` to Supabase JWT/JWKS verification; remove
  `POST /auth/login` and the password helpers; rework `POST /auth/users`
  onto the Supabase Admin API; replace the frontend in-memory session
  stub with the `supabase-js` session; seed the bootstrap admin; rewrite
  the auth tests.
- **Forbidden work:** RLS, storage, realtime, business-logic changes.
- **Exit gate:** all authentication flows through Supabase; FastAPI
  mints no tokens; the frontend session persists across reload and on
  mobile; auth tests rewritten and green; the rest of the suite green.

### F2.22.3 — RLS Defense-in-Depth Baseline

- **Objective:** enable RLS as a secondary layer without affecting
  FastAPI.
- **Allowed work:** enable RLS deny-all on all `public.*` tables; create
  the RLS helper functions; confirm FastAPI runs on an RLS-bypassing
  role; introduce the RLS test layer.
- **Forbidden work:** frontend-facing write policies; storage; realtime;
  business-logic changes.
- **Exit gate:** RLS enabled deny-all everywhere; FastAPI unaffected and
  its suite green; the RLS test layer green; no write policy exists for
  `authenticated`.

### F2.22.4 — Supabase Storage

- **Objective:** provision storage and wire `product-images`.
- **Allowed work:** create the four buckets and their policies; add the
  storage-metadata table (Alembic); implement the FastAPI signed-URL
  endpoint and metadata writes; add the frontend storage helper and the
  `product-images` upload flow.
- **Forbidden work:** wiring `store-assets` / `compliance-attachments` /
  `exports` end-to-end; realtime; business-logic changes beyond the
  signed-URL/metadata endpoint.
- **Exit gate:** `product-images` upload flow works end-to-end; storage
  upload tests green; the rest of the suite green.

### F2.22.5 — Supabase Realtime

- **Objective:** deliver realtime invalidation for orders and inventory.
- **Allowed work:** add the Realtime publication on `orders` and
  `inventory_items`; add the §7 `SELECT` policies; add frontend
  subscription hooks that invalidate TanStack Query keys.
- **Forbidden work:** realtime on any other table; rendering realtime
  payloads directly; business-logic changes.
- **Exit gate:** the frontend receives change signals for orders and
  inventory and refetches through FastAPI; realtime invalidation smoke
  tests green; the rest of the suite green.

### F2.22.6 — Global Regression / Final Closure

- **Objective:** confirm the whole hybrid system end-to-end and close
  the phase.
- **Allowed work:** run the full validation matrix (§13); run the
  boundary enforcement checks (§16); fix only defects surfaced by
  regression; update this document if reality diverged.
- **Forbidden work:** new features; new scope; business-logic rewrites.
- **Exit gate:** every §13 suite green; every §16 boundary check passes;
  the §17 definition of done is satisfied.

- **Backend pytest suite** — the existing 1365 tests, run against

- orders state machine
- RBAC

## 14. Rollback Plan

- **F2.22.1 (DB host cutover):** the previous Postgres instance is kept
  running and untouched until F2.22.1 is verified. Rollback = repoint
  `DATABASE_URL` to the previous instance.
- **F2.22.2 (Auth migration):** `password_hash` is kept (nullable) until
  F2.22.2 is verified, so the FastAPI login path can be restored from
  git. With no production users, the realistic rollback is "recreate
  seed users." `password_hash` is dropped only in a later cleanup.
- **F2.22.3 (RLS):** RLS is additive and FastAPI bypasses it. Rollback =
  `ALTER TABLE ... DISABLE ROW LEVEL SECURITY` per table and drop the
  helper functions.
- **F2.22.4 (Storage):** additive and isolated. Rollback = remove the
  buckets and their policies, the metadata table, and the frontend
  storage helper / FastAPI signed-URL endpoint.
- **F2.22.5 (Realtime):** additive and isolated. Rollback = drop the
  Realtime publication and the `SELECT` policies, and remove the
  frontend subscription hooks.

## 15. Explicit Non-Goals

F2.22 explicitly does **not** include:

- Retiring, freezing, or removing FastAPI.
- Moving business logic to Supabase RPC.
- Moving business logic to database triggers.
- Moving business logic to Edge Functions.
- Making RLS the primary authorization system.
- Letting the frontend write directly to `public.*` business tables.
- Customer checkout, cart, or storefront.
- A driver app or driver surface.
- Payments, billing, commission collection, or invoicing.
- Email or webhook integrations.
- Rewriting all frontend API modules.
- Replacing Alembic with Supabase migrations.

Work that drifts into any of these must update this document first.

## 16. Boundary Enforcement Checks

These checks are run during F2.22.6 (and may be run after any subphase)
to confirm the hybrid boundary held. They are reference commands, not a
substitute for review.

**No direct supabase-js business table reads/writes.** The supabase-js
client `.from(...)` (table query) and `.rpc(...)` must not appear in
feature business code:

```bash
grep -rnE "\.(from|rpc)\(" web/src/features
# expected: no business-table query/rpc hits; storage uses .storage.from()
grep -rn "supabase" web/src/features --include=api.ts
# expected: no matches — feature api.ts modules stay on apiRequest()
```

**No Edge Functions added:**

```bash
test ! -d supabase/functions || find supabase/functions -type f
# expected: directory absent, or empty
```

**No RPC / business-logic migration into the database:**

```bash
grep -rniE "create (or replace )?function" supabase/migrations
# expected: only RLS helper functions (is_admin, current_app_user_*),
# never order/inventory/compliance mutation logic
```

**FastAPI business endpoints preserved:**

```bash
grep -rhcE "@(router|variants_router)\.(get|post|patch|put|delete)\(" \
  backend/app/api/routes/*.py | paste -sd+ - | bc
# expected: >= 72 (minus only POST /auth/login removed in F2.22.2 → 71)
```

**Alembic still owns the public schema** (Supabase migrations create no
app tables):

```bash
grep -rniE "create table" supabase/migrations
# expected: no public.* application tables
```

**No checkout / driver / payment scope creep:**

```bash
grep -rniE "checkout|stripe|\bpayment\b|driver-app" \
  web/src backend/app supabase
# expected: no new occurrences introduced by F2.22
```

- `docs/f2.22-contract-lock.md` exists.
- Architecture boundaries are explicit (§§2–10).
- The phase plan is locked (§§11–12).
- Non-goals are explicit (§15).

- No implementation files are changed.
- Markdown validation passes if the repo supports it.

Supabase Postgres via the session pooler; all auth flows through
Supabase Auth; RLS is deny-all on every `public.*` table with FastAPI
